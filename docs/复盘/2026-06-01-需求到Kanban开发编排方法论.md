# 需求文档 → 设计方案 → Kanban 执行：通用开发编排方法论

> 适用场景：从已写好的 PRD / 需求说明 / 技术设计文档出发，把它们整理成可直接执行的开发计划、Kanban 任务链路、worktree 并行开发方案，以及最终验收闭环。

---

## 1. 这份方法论解决什么问题

很多项目失败，不是因为代码写不出来，而是因为：

- 需求没先冻结就开始拆任务
- 设计文档和实现任务之间没有明确映射
- 任务图过度串行，导致开发很慢
- review / QA / PM sign-off 没有独立门禁
- 实现卡没有测试约束，最后只能靠人工口头验收
- 主分支直接开发，导致多个任务互相覆盖
- 阻塞状态、review-required、真正 blocker 混在一起
- manifest、看板、文档三者不一致，最后没人知道“真相”在哪

这套方法论的目标是：

1. 从需求和技术设计中提取完整覆盖矩阵
2. 把覆盖矩阵转成可执行的 Story / Task DAG
3. 在 Kanban 上组织并行、review、QA、closeout 的门禁链路
4. 用 worktree 隔离并行实现
5. 用 TDD + 单元测试 + 验证证据作为进入下游的硬条件
6. 通过 manifest + live board + watcher 做持续监督
7. 最终把产品完整实现并验收通过

---

## 2. 统一的层级顺序

推荐把整个流程看成下面这条链：

```text
PRD / 需求文档
    -> 技术设计 / 共享契约
        -> 覆盖矩阵
            -> Story 级 Kanban 计划
                -> 并行实现 lanes
                    -> reviewer
                        -> QA
                            -> PM sign-off
                                -> program closeout
```

### 核心判断

- **需求文档**回答：要做什么、边界是什么、哪些不做
- **技术设计**回答：怎么做、契约怎么定、边界怎么冻结
- **Kanban 计划**回答：谁先做、谁并行、谁 review、谁 QA、谁收口
- **worktree**回答：并行实现时怎么避免互相污染
- **测试和验证**回答：怎么证明做对了

---

## 3. 先定哪些角色，后定任务图

不是所有任务都该直接进入实现。

### 先判断是否需要 PM

当以下任一情况成立时，先让 PM 收口：

- 需求目标不清
- 验收标准不清
- 产品范围不清
- 有用户可见表达、对外材料、发布说明
- 需要明确哪些是 deferred / accepted / rejected

### 再判断是否需要 Architect

当以下任一情况成立时，先让 Architect 收口：

- contract / schema / 状态机 / API 边界变化
- 跨服务或跨模块协作
- 新模块边界不稳定
- 迁移、回填、兼容性、回滚风险
- 共享事件、共享状态、共享身份体系需要冻结

### 什么时候不需要强拉 PM / Architect

- 小 bug
- 范围很小且边界清晰
- 单点修补
- 只是局部实现修正

---

## 4. 从需求和设计文档到 Coverage Matrix

这是最关键的一步。

### 输入材料

优先读取：

- PRD / 需求文档
- 技术概要设计
- 详细设计文档
- 共享契约文档
- 现有 repo 约束（AGENTS.md、架构说明、运行手册）

### Coverage Matrix 要回答的事

对每一个页面 / 能力 / 契约条目，必须明确属于哪类：

- 已完成
- 计划中已有 Story
- 明确延期 / deferred by scope
- 定义不清，需 PM 补充确认

### 原则

- 不要把“主路径覆盖”误当成“全量覆盖”
- 不要因为已经有 MVP 就跳过 PRD 条目级映射
- 不要把不清晰的条目自动吞掉
- 不要让“可选项”伪装成“已覆盖”

### 这一步产物

应该产出一个明确的映射表：

- PRD 条目 -> Story ID
- PRD 条目 -> deferred
- PRD 条目 -> need clarification

如果映射不完整，就不要宣称计划已达成全覆盖。

---

## 5. 先冻结共享契约，再并行拆分

这是避免返工的关键。

### 必须先冻结的共享前置

- API contract
- 状态枚举
- auth / session 规则
- 共享事件口径
- 审计 / 权限 / identity 规则
- 需要多个实现卡共享的 DTO / schema / 协议

### 为什么

如果共享前置还没冻结，就让多个任务并行，很容易出现：

- 前后端各写一套语义
- 数据路径和服务路径互相打架
- reviewer 发现的不是实现 bug，而是 contract 冲突
- QA 变成补契约

### 结论

> 共享契约未冻结时，不要急着把实现卡并行化。

---

## 6. Story 级拆解原则：按执行 lane 拆，不按 checklist 拆

### 正确的拆法

把每个 Story 拆成若干可以独立完成、独立验证、独立 review 的实现卡。

适合并行的典型 lane：

- backend-eng
- frontend-eng
- data-rd
- 必要时 architect 前置卡

### 错误的拆法

- 把一个实现卡再拆成很多 checklist 子步骤
- 把原本可以并行的任务串成 A -> B -> C
- 把 review / QA / closeout 也混进实现卡里

### 判断标准

一个任务只有在满足下面三点时，才值得成为独立实现卡：

1. 责任边界清楚
2. 可以独立验证
3. 不会和别的并行卡互相踩文件

---

## 7. Kanban 的标准结构

推荐使用这种结构：

```text
root / scope freeze
    -> shared contract freeze
        -> parallel implementation lanes
            -> reviewer
                -> QA
                    -> batch closeout
                        -> next batch / final chain
```

### 批次建议

- Batch 0：覆盖矩阵 / 共享契约 / scope freeze
- Batch 1：底层基础能力
- Batch 2：产品能力绑定
- Batch 3：体验链路 / 协作 / Loop
- Batch 4：商业化 / 系统后台 / 安全与治理
- Final：全量回归 -> PM sign-off -> program closeout

### 每个 batch 都应该有

- implementation fan-out
- batch review
- batch QA
- batch closeout

不要跳过 batch closeout。

## 8. worktree 策略

### 什么时候必须用 worktree

只要存在以下情况之一，就建议用 worktree：

- 多个实现卡并行
- 多个 profile 同时开发
- 需要避免主分支污染
- 需要让 review / QA 证据可追溯
- 需要降低回滚和冲突成本

### worktree 的粒度

**按并行执行卡拆，不按 story checklist 拆。**

推荐原则：

- 一个实现卡 = 一个 worktree
- 一个卡如果要并行开发，就不能和别的卡共享同一个目录
- reviewer / QA / closeout 通常不需要独立 worktree，保持共享集成工作区即可

### 推荐目录规则

- `/.worktrees/<task_id>/`
- 每张实现卡一个独立 branch
- 例如：`wt/<task_id前8位>`

### 重要提醒

worktree 只是隔离手段，不是任务切分的目标。

不要为了“看起来并行”把图拆得太碎。

---

## 9. TDD / 单元测试 / 验证门禁

所有实现卡在进入 reviewer 前，都应该满足下面的硬门禁。

### 必须满足

- 先写失败测试，再写实现
- 每个新增或修改行为都要有测试
- 至少要有单元测试
- 涉及跨层 / 流程 / contract 的行为，再补 integration / contract test
- 完成时必须给出 fresh verification evidence

### 推荐验证证据

- failing test 命令和预期失败原因
- targeted passing test 命令和结果
- regression test 命令和结果
- exit code
- 未覆盖区域 / known issues

### DDD 驱动要求

实现卡要明确写出它属于哪个边界：

- bounded context
- aggregate
- domain service
- application service
- repository
- frontend feature boundary

不要把 UI、Team Panel、Gateway、Runtime 语义混在一个层里。

---

## 10. reviewer / QA / PM / closeout 的职责边界

### reviewer

职责是做技术质量门，不是重写实现。

应该检查：

- contract 是否冻结
- 实现是否遵守边界
- 是否有测试
- 是否有 fresh verification
- 是否有 worktree / branch / artifacts
- 是否存在 architecture drift

### QA

职责是验证用户路径和回归证据。

应该检查：

- 自动化测试是否通过
- 手工验证路径是否清楚
- 空态 / 错态 / 权限态是否覆盖
- unverified areas 是否明确列出

### PM sign-off

职责是产品验收，不是技术 review。

应该输出：

- accepted
- partially accepted
- deferred
- rejected

### closeout

职责是把中间证据、风险、剩余阻塞和 sign-off 状态收束起来。

> 最终完成必须依赖显式 closeout task ID，而不是“看起来都完了”。

---

## 11. blocked / review-required / done / heartbeat 的处理规则

这是看板监督里最容易出错的地方。

### review-required 不是自动 blocker

如果实现者已经完成 scope，并且只是要求 review，而图里已经有显式 review gate：

- 不要把它误标成 blocked
- 应该把 review concern  carried 到 review card
- 原实现卡可按 operator correction 完成或推进

### `done` 但文本 verdict 是 `BLOCKED`

这是**真 blocker**。

处理方式：

- 创建 rework 卡
- 创建 re-review 卡
- 把 QA 依赖重新接到 re-review
- reclaim 可能已经过早启动的 QA

### `heartbeat`

表示：

- 任务还在运行
- 当前窗口内没有异常
- 这不是完成，也不是失败

### `alert`

必须先分类：

- blocked / scheduled
- failed
- stale-running

然后再决定：

- unblock
- rework
- escalation
- wait

### 规则总结

> 不要把 review_required、blocked、failed、stale、heartbeat 混为一谈。

---

## 12. manifest / board / plan 的关系

三者不是一回事。

### `.plans`

是需求拆解和执行计划。

### `kanban.db`

是**真实执行中的任务图**。

### `manifest.json`

是这条 workflow 的**全局链路快照**。

### 重要原则

- 计划文件不是执行真相
- manifest 不是代码实现
- board 才是活的执行状态
- 需要以 explicit task ID 为准，不要靠标题猜

### 必须保持一致

- root task id
- graph review task id
- PM sign-off task id
- program closeout task id
- task_count
- stages
- batch review / QA / closeout gates

---

## 13. 推荐的监督方式

不是所有监控都要靠人工问答。

### 最佳实践

- 用明确 task IDs
- 用 watcher / stats / show / tail
- 用 bounded foreground loop
- 用 manifest 作为结构镜像
- 用 live board DB 作为执行镜像

### 不推荐

- 只靠标题搜索
- 只靠会话记忆
- 只靠“我感觉差不多了”
- 不看 review verdict 就推进 QA

## 14. aiteam 这次 Kanban 复盘提炼出的经验

这次 aiteam 的复盘里，几个关键教训很稳定，可以直接沉淀成方法论。

### 经验 1：共享契约必须先冻结

之前的问题不是单纯实现慢，而是共享 contract 不够硬，导致前端、后端、数据路径、运行态口径都容易偏。

### 经验 2：不要把实现卡串成长链

串行 DAG 会导致速度慢。

正确做法是：

- 前置共享契约冻结
- 之后按执行 lane 并行
- review / QA / closeout 再 fan-in

### 经验 3：worktree 是并行执行的基础设施

如果还在主分支上多卡并行，最终会出现：

- 文件互相覆盖
- review 难回溯
- QA 证据混淆
- 回滚成本高

所以实现卡应该使用 worktree 隔离。

### 经验 4：reviewer 的语义要严格区分

- `review-required` 不是 blocker
- 真 blocker 才应该 rework
- `done` 但 verdict blocked 才是真正要回流的情况

### 经验 5：最终收口必须显式

要有：

- final review
- final QA
- PM sign-off
- program closeout

不能只靠“全图都 done 了”来宣布结束。

### 经验 6：manifest 不能漏节点

这次就碰到过 manifest 漏掉 batch 子卡的情况。

因此必须验证：

- manifest stage 数量
- 实际可达任务数
- root 到 closeout 的路径是否闭合

## 15. 一份可复用的标准操作流程

### Phase A：读文档

1. 读 PRD / 需求文档
2. 读技术设计 / 共享契约
3. 读 AGENTS / 项目约束
4. 读已有复盘 / runbook / 历史 plan

### Phase B：定范围

1. 提取 coverage matrix
2. 标注已完成 / 计划中 / deferred / unclear
3. 找出必须先冻结的共享前置
4. 明确 PM / Architect 是否需要前置

### Phase C：写计划

1. 先写 stage
2. 再写 construction axis
3. 再写 validation axis
4. 把 story 拆成可执行实现卡
5. 给每张卡写验收和测试要求
6. 明确 review / QA / PM / closeout gate

### Phase D：拆 Kanban

1. root
2. scope freeze / contract freeze
3. batch fan-out
4. batch review
5. batch QA
6. batch closeout
7. final review / QA / PM / closeout

### Phase E：配置 worktree

1. 并行实现卡使用独立 worktree
2. reviewer / QA / closeout 保持共享 workspace
3. 每张实现卡记录 branch / path
4. 开发不要回到 main/master

### Phase F：执行与验证

1. 先写失败测试
2. 再实现最小代码
3. 再跑验证
4. 证据写进 handoff metadata
5. reviewer 只看证据，不看感觉

### Phase G：监督与返工

1. 只用显式 task ID 追踪
2. 区分 review-required / blocked / failed / stale / heartbeat
3. 真 blocker 走 rework/re-review
4. QA 不要替实现者补代码
5. closeout 必须显式完成

## 16. 一页版检查清单

### 计划阶段

- [ ] 读了需求 / 设计 / 复盘材料
- [ ] 做了 coverage matrix
- [ ] 标注了 deferred / unclear 项
- [ ] 冻结了共享契约
- [ ] 选定了 PM / Architect 是否前置

### 拆卡阶段

- [ ] story 按执行 lane 拆，而不是 checklist
- [ ] 每张实现卡边界清楚
- [ ] review / QA / closeout 独立
- [ ] 并行关系明确
- [ ] 最终 closeout 显式存在

### worktree 阶段

- [ ] 每张实现卡有独立 worktree
- [ ] 不在 main/master 上直接开发
- [ ] branch / path 已记录
- [ ] review / QA / closeout 不混在实现 worktree 中

### 开发阶段

- [ ] 先写测试
- [ ] 先跑失败
- [ ] 再最小实现
- [ ] 再跑通过
- [ ] 再跑回归
- [ ] handoff 有 artifacts / verification / known_issues / next_steps

### 收口阶段

- [ ] review pass 或 pass-with-nits 已记录
- [ ] QA confirmed pass 已记录
- [ ] PM sign-off 明确
- [ ] program closeout 完成
- [ ] manifest 与 board 一致

---

## 17. 最后一句话

> 这套方法论的本质，是把“需求 → 设计 → 计划 → 并行执行 → 验证 → 收口”变成一条可审计、可回流、可复用的链路，而不是靠个人记忆和临场感觉来推进项目。
