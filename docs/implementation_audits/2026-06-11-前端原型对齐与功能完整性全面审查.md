## 0. 总体结论

**当前实现是一个「API 联调控制台」与「产品」的混合体，距离"和原型总体一致、所有功能可用"的交付目标有明显差距。**

> ⚠️ 2026-06-11 场景级验收（§0.6）修正：下述「主要差距集中在前端」的判断已被推翻——Gateway/Runtime 执行层存在 5 条断裂链路（P0-12~P0-17），**五个演示场景没有一个能以用户视角完整走通**。请以 §0.6 的判定总表为准。

- 后端北向 API 覆盖面较好（runs / conversations / group / org / knowledge / skills / connectors / solutions / memories / billing / office / workbench / uploads 均已落地），**主要差距集中在前端**。
- 人才市场、工作台、办公室画布、企业后台多数页面已有可用骨架，部分页面（人才市场）已接近原型。
- 但三类问题普遍存在：
  1. **联调态 UI 直接交付给用户**：sender_id 手填、cursor/display_state/runtime_handle/contract 等内部口径全部暴露在界面上，群聊页最严重；
  2. **死控件 / 假实现**：私聊附件上传是写死的假数据、私聊搜索框无事件绑定、办公室缩放与全屏按钮在 canvas 模式下全部失效、知识库建库入口在非空态下消失；
  3. **核心演示场景受阻**：场景 2（上传 PDF 知识问答）因前端只支持纯文本上传而无法演示；场景 3（Loop）只有员工抽屉里一个裸 cron 表达式输入框，无启停/状态/执行记录可视化。

问题按优先级分级：
- **P0**：阻断核心演示场景或基本不可用，必须修复；
- **P1**:功能缺失/明显偏离原型/暴露内部口径，交付前应修复；
- **P2**：体验或一致性问题，可排期。

### 运行时新发现（代码审查未覆盖）

**P0-10 系统后台运行时整体不可用（5 页全部 403）**
- 现象：/system/accounts、/system/templates、/system/solutions、/system/finance 页面只显示「您没有权限访问此内容」（截图 31~34）；console 全是 `/api/system-admin/* 403`。
- 根因：后端按 query/body 里的 `role`/`actor_role` 解析角色，缺省回落 `owner`（`router_team.py:844-850`），owner 无 `system_read`；而前端 api-client 调系统接口从不传 role（`api-client.js:342-352`），`role-state.js` 的 activeRole 默认空字符串且无任何赋值入口。
- 验证：`GET /api/system-admin/enterprises` → 403；`GET /api/system-admin/enterprises?role=system_admin` → 200。
- 修复方向：角色应来自登录会话（Auth 收口口径），由后端从 session 解析；过渡期前端至少把 getActiveRole() 透传为 query 参数，并提供角色切换入口。

**P0-11 群聊发消息链路真实环境完全走不通（比 P0-1 更深一层）**
- 即使填了 sender_id 也被拒：`POST /group-conversations/{id}/messages` 返回 `INVALID_REQUEST: sender_id usr_test is not an active conversation member`。
- 根因：`POST /group-conversations` 创建群时只写入 member_employee_ids，不会把创建者（人类用户）写成 human member；而消息接口要求 sender 必须是 active member。前端也没有任何途径把人类用户加进群（「新增员工」入口只接受 employee_id）。
- 结论：当前不存在任何一条用户能在群聊里发出消息的操作路径。修复：建群时自动把当前登录用户加入为 human member，发消息 sender 取登录态。

**P1-22 余额不足（402）前端无产品化提示**
- 后端余额拦截完整：`POST /runs` 在余额为 0 时返回 402 `INSUFFICIENT_BALANCE`，含 `recharge_required: true`、`balance_cents`、`low_balance_warning` 等字段。
- 前端单聊只显示「发送失败，可重试发送」，未消费 402 语义（应提示余额不足并引导 /admin/billing/recharge）。BRD 7.16 明确要求「限流 / 余额不足提示」。

**环境/数据阻塞项（非代码缺陷，但阻断交付演示）**
1. **LightRAG 依赖缺失**：上传文档后入库失败，文档状态 `error`，失败原因 `No module named 'lightrag'`（截图 36）。即使修复 P0-5 的 PDF 上传，场景 2 在当前环境也跑不通。需在 `app/.venv` 安装 lightrag 或接通外部 LightRAG 服务。
2. **演示数据贫乏**：人才市场只有 1 个模板（Marketing Analyst）、行业方案/技能目录数据少，无法支撑「行业方案一键应用」「多角色招募」演示，需要 seed 脚本。
3. **企业余额为 0**：发任何消息都被 402 拦截，演示前需 mock_pay 充值或 seed 余额。
4. **运行环境多实例混乱**：8787（Jun01 旧代码）、8788（~/.hermes 旧实例）、18080（其他工作区 sqlite 实例）均为过期服务；`.env` 的 DATABASE_URL 指向 5433，实际 Postgres 容器在 55433，靠一个临时 Python 转发进程续命。建议固化 docker-compose（Postgres@5433）+ 单一服务实例。

### 运行时修正的代码审查结论

### 场景级验收新增问题清单

**P0-12 员工 profile 供给从未成功：`hermes profile create --home` 参数不存在**
- `profile_provisioner.py:41-48` 拼出 `hermes profile create <name> --home <path>`，实测该 CLI 无 `--home` 选项 → usage error → 每次 RuntimeError。
- 连锁后果：① SOUL.md 员工人格永不生效（回复自称「Hermes Agent」）；② cron_create 报 Profile not exist（击穿场景 3）；③ 编排子任务 CLI 执行同样报 usage error（击穿场景 1）。
- 修复方向：用 `HERMES_HOME=<home>` 环境变量调用 CLI，并在创建后把主 home 的 provider 凭证带入 profile。

**P0-13 私聊稳定态 100% 失败：半供给 profile 目录使 agent 初始化丢失 API key**
- webui chat/start 会自动建出 `profiles/emp-test/`（含 sessions/state.db，但无 config.yaml/.env）；此后该员工所有 run 在 `agent_init.py:824` 抛 `no API key was found`，executor 等满 300s 写「stream timeout (> 300s)」。
- 首次 run（目录不存在时）反而成功——演示会出现「第一句话能回，后面全挂」的最坏形态。实测向 profile 目录补 .env 后仍失败（profile env 路由未把 provider 凭证带进 agent init），需按根因修 profile 凭证继承。

**P0-14 会话 session 持久化每次失败：runtime_binding CHECK 不含 'conversation'**
- `runtime_executor.py:288` 写 `owner_type="conversation"`，但 `001_initial_schema.sql:383` CHECK 仅允许 `employee/team_run/team_task/scheduled_job` → 每 run 报 CheckViolation。
- 后果：会话与 runtime session 的绑定永远建不起来，**每条消息新开一个 Hermes session，私聊多轮上下文完全断裂**（实测同一会话 3 条消息产生 3 个 session）。修复：迁移扩枚举或改用合法 owner_type。

**P0-15 费用看板被脏数据永久击穿（500）**
- `billing_view_service.py:89` 物化 usage_ledger 时把无 entry_employee_id 的 run 记为 `employee_id='unknown'` → FK violation → 整个 `/billing/usage/overview` 与企业后台费用页 500，且每次刷新都重试重炸。
- 修复：无员工 run 跳过或归集到企业级行；`POST /runs` 同时应回落 conversation.entry_employee_id（见 P1-23）。

**P0-16 Loop 调度链路断头：tick 入口零调用方**
- `create_scheduled_job_run`（`scheduled_job_service.py:103`）是唯一把 cron 触发翻译成 TeamRun 的入口，但无任何北向路由、内部调度器或 Hermes 回调调用它。叠加 P0-12（cron 注册失败）与注册参数缺失（不传 schedule_expr/goal），场景 3「每 10 分钟巡检」在当前代码下不可能发生。

**P0-17 连接器测试同义反复 → 授权死锁**
- `router_team.py:3977`：非 MCP 连接器 `ok = connector.status == "online"`；新建即 offline、测试永远 failed、PATCH 不能改 status、grants 要求 online。api_key/oauth 连接器全链路死锁，场景 5 前半段不可能完成。
- 修复：实现真实凭证探活（或 MVP 期按 credential_state=configured 视为通过），并让 grants 的前置校验与之自洽。

**P1-23 `POST /runs` 不回落会话入口员工**
- 仅带 conversation_id 时 `entry_employee_id=None`，导致无人格/无模型/无知识注入 + 账单脏数据（P0-15 的源头）。前端虽传 employee_id，但契约层应在 create_run 内回落 `conversation.entry_employee_id`，消除这个特殊情况。

**P1-24 编排子任务拆解是复读机**
- 3 个子任务均为「处理: <原文>」，无角色化拆解（调研/起草/润色），与 BRD 7.5「任务拆解与责任分配」差距大；主持汇总因子任务全败未观测到。

**P1-25 模板 default_model_ref 不传导**
- 人才市场招募与方案应用创建的员工 model_provider/model_name 均为空，模板里的 `default_model_ref` 被丢弃 → 新员工开箱即坏。`_handle_recruitments_post` / `_handle_solution_apply_post` 应落模板默认模型。

**P1-26 连接器创建无入参校验**
- 非法 connector_type 直接撞 DB CHECK 返回 500 裸 Internal server error；应 400 + 枚举提示（grants 错误形态静默 200 同类，见场景 5 第 3 步）。

**P1-27 lightrag/fastembed 依赖未声明**
- `lightrag_service.py` import lightrag/fastembed/numpy，但 `app/requirements.txt` 完全未声明 → 任何新环境必然复现「No module named 'lightrag'」。原 §0.5 把它归为环境项，实为依赖管理缺陷。

**P2 追加**：`POST /uploads` 忽略 file_name 落「file.bin」；`scheduled_job_action` 未带 `scheduled_job.scheduled_job_id` 时被静默忽略仍返回 200；executor 与 gateway 的 profile 命名不一致（`emp-test` vs `emp_test`，后者用于 cron --profile，必然找不到）。

### 对 §0 总体结论的修正

§0 曾判断「主要差距集中在前端」。场景级验收推翻该判断：**Gateway/Runtime 执行层与前端同等糟糕**——profile 供给（P0-12/13）、会话持久化（P0-14）、Loop 调度（P0-16）、连接器探活（P0-17）、账单物化（P0-15）五条执行链全部断裂。当前没有任何一个 §9 演示场景能以用户视角完整走通；最接近的是场景 4（方案应用），也带着「创建出的员工不可用」的尾巴。
