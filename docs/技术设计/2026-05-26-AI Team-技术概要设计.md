# AI Team 技术概要设计

## 1. 需求背景

### 1.1 背景

AI Team 的目标不是提供单点 AI 对话能力，而是面向个人、团队与企业构建一套可配置、可协作、可持续运行、可治理的数字员工平台。平台需要同时承接企业空间、数字员工、知识与技能绑定、单任务会话、多角色协作、周期任务、治理与经营分析等完整能力面。

从业务侧看，AI Team 要解决的不是“再做一个聊天框”，而是把分散的模型能力、工具能力、知识能力与协作能力，组织成一套接近企业真实岗位分工的数字员工体系。

### 1.2 现状与问题

当前需要解决的核心问题主要有四类：

1. **AI 使用割裂**
- 多个 AI 工具、知识工具、办公工具相互分散。
- 用户需要反复切换系统与重复搬运上下文。
- 单一对话难以承接完整任务闭环。

2. **缺少数字员工级抽象**
- 现有产品大多停留在聊天机器人层。
- 很难直接映射企业岗位、职责、协作关系与组织语义。

3. **缺少持续执行与能力沉淀机制**
- 普通会话式 AI 难以承接周期任务、长期任务和持续巡检任务。
- 历史经验、知识与技能沉淀无法稳定复用。

4. **缺少统一治理与经营闭环**
- 模型、知识、技能、连接器、费用与日志分散在不同系统。
- 难以形成统一的权限、审计、成本和运营管理视图。

### 1.3 分析总结

因此，AI Team 的技术设计不能继续沿用“前端 + 单一聊天后端”的常规方式，而需要明确回答以下问题：

- 系统建设目标是什么，能力边界是什么。
- 平台功能应如何分层。
- 整体系统应如何划分核心部分与外部依赖。
- 构建 Agent Runtime、知识引擎、技能生态和连接器能力时，系统内部边界如何保持稳定。

## 2. 需求分析

### 2.1 要解决什么问题

AI Team 需要解决的是：如何把数字员工、知识、技能、模型、连接器与协作能力组织成一个统一的平台，使用户能够在企业空间中完成数字员工招募、能力装配、任务发起、协作执行、结果回流以及治理运营。

### 2.2 解决方式是什么

采用“**前端交互页面群组 + Team Panel + Agent Gateway + Agent Runtime + External Capability**”的总体结构：

- 前端交互页面群组负责承接产品交互。
- Team Panel 负责业务对象、能力装配、任务翻译、运行态包装与治理经营。
- Agent Gateway 负责运行时接入、事件回流、审批/中断/调度触发等统一接入能力。
- Agent Runtime 负责统一执行真相层。
- External Capability 负责知识、技能、连接器与外部系统接入。


**在业务主链路上**，前端优先调用 Team Panel，由 Team Panel 把业务任务翻译为运行时请求，再交给 Agent Gateway 接入 Agent Runtime。
**在实现形态上**，Agent Gateway 采用基于 Agent Python SDK 模块的进程内运行时适配层。

### 2.2.1 对业务解决方案与 BRD/PRD 需求的承接

设计方案不是脱离业务文档单独成立的一套技术想象，而是对业务解决方案以及 PRD/BRD 中已明确需求的技术承接。主要承接关系如下：

说明：
- 所有功能都由前端页面承接交互入口与页面表达。
- 对于 AI Team 的主产品流，前端默认不直接面向 Runtime，也不直接绑定 runtime-native contract，而是优先通过 Team Panel 进入业务主链。

- **人才市场 / 数字员工招募 / 行业方案应用**
- Team Panel 承接模板、员工实例、方案绑定与启停管理。

- **私聊对话 / 群聊协作 / @提及协作**
- Agent Gateway 承接多入口消息接入、流式事件输出与审批/中断能力。
- Agent Runtime 承接单员工执行与多 Agent 协作执行。

- **知识库 / RAG 检索 / 企业知识绑定**
- Team Panel 承接知识库对象、权限与员工绑定关系。
- External Capability 承接知识引擎接入。
- Agent Runtime 在执行时完成检索增强注入。

- **长期记忆 / 初始记忆 / 记忆管理**
- Team Panel 承接员工级绑定、查看与治理语义。
- Agent Gateway 承接运行时读取、事件透传与结果回流接口。
- Agent Runtime 承接长期记忆检索、注入、沉淀与执行调用。

- **Loop / 周期任务 / Goal 调度**
- Team Panel 承接业务配置、启停规则与任务入口。
- Agent Gateway 承接调度触发、运行提交与状态查询接口。
- Agent Runtime 承接 Cron Scheduling 与持续运行能力。

- **企业后台 / 系统后台 / 工资与成本治理 / 运营统计**
- Team Panel 承接治理与经营对象、统计口径和结果包装。
- Agent Gateway / Agent Runtime 提供 usage / cost metadata、运行事件与执行真相。

- **Skill 生态 / MCP / 企业连接器 / 外部系统集成**
- External Capability 承接外部能力域。
- Agent Runtime 承接工具、MCP、Plugins 与调用执行。
- Team Panel 承接员工级授权、绑定和可视化管理。

因此，当前概要设计能够覆盖业务解决方案中已经明确的主闭环：

**企业空间 → 人才市场 / 行业方案 → 招募员工 → 配置模型 / 技能 / 知识 / 记忆 / 连接器 → 私聊 / 群聊发任务 → 协作 / Loop 执行 → 日志 / 记忆 / 费用沉淀 → 后台治理**

### 2.3 术语定义

本文术语统一如下：

- **数字员工**：平台中的可配置执行主体，具备岗位、模型、知识、技能、记忆和连接器绑定能力，对应一个独立的 Profile。
- **Team Panel**：AI Team 自建业务控制核心部分，负责业务对象管理、能力装配、任务翻译、运行态包装与治理经营。
- **Agent Gateway**：AI Team 的运行时接入核心部分，负责把 Team Panel 形成的业务请求接入 Agent Runtime，并统一承接流式事件、审批、中断、调度触发与运行态句柄管理。
- **Agent Runtime**：平台复用的统一执行底座，承接多入口宿主、主循环、工具、技能、记忆、会话、多模态与协作执行能力。
- **External Capability**：知识引擎、技能生态、AI Relay、企业连接器与第三方业务系统等外部能力集合。
- **多智能体协作编排**：面向业务语义的多角色协作组织能力，用于把任务拆解、分配、执行和汇总。

## 3. 设计目标

### 3.1 系统建设目标

AI Team 的建设目标是构建一套面向企业与团队的数字员工平台。该目标直接对应业务解决方案文档与 PRD/BRD 已明确的三端诉求：企业前台可用、企业后台可管、系统后台可运营，并要求数字员工、知识、记忆、技能、协作和成本治理形成统一闭环。

具体目标如下：

1. 用户能够在企业空间中快速招募并配置数字员工。
2. 数字员工能够绑定模型、知识、技能、连接器和记忆能力。
3. 用户能够通过单任务会话、群聊协作和周期任务等方式发起工作。
4. 平台能够支持单员工执行、多员工协作和持续运行任务。
5. 执行结果、进度、日志、成本和治理数据能够统一回流。
6. 企业后台与系统后台能够基于统一业务对象进行运营与治理。

### 3.2 设计原则

概要设计阶段采用以下原则：

- **低耦合**：业务控制、统一入口、执行真相层与外部能力接入分层隔离。
- **无侵入复用**：尽量复用成熟执行底座与外部能力，不重复建设已有能力。
- **隔离性**：企业空间、员工实例、会话与能力绑定应具备清晰隔离边界。
- **灵活性**：支持单任务、多角色协作、周期任务与多模态任务。
- **易用性**：产品交互层与业务控制层必须面向企业用户表达，而不是直接暴露底层执行复杂度。

### 3.3 功能分层

AI Team 功能架构按五层划分：

1. 接入与工作台层
2. Team Panel 业务控制层
3. 协作与执行控制层
4. 底层全局能力与能力装配层
5. 治理与经营闭环层

### 3.4 功能架构说明

#### 1）接入与工作台层

负责承接用户可见交互面，包括身份与空间入口、单任务工作台、多角色协作工作台、自主任务入口以及治理控制台。

#### 2）Team Panel 业务控制层

负责身份与空间管理、数字员工管理、模板与行业方案管理、能力绑定管理以及治理与经营能力。它是 AI Team 自建业务控制面的核心承载层。

#### 3）协作与执行控制层

负责任务识别与路由、多智能体协作编排、任务树与进度回流、定时与 Loop 管理，是 AI Team 核心业务闭环的主要能力层。

#### 4）底层全局能力与能力装配层

负责承接和装配 Runtime 提供的多入口宿主、Provider 路由、工具系统、MCP、Plugins、记忆、技能、上下文、会话存储、知识库绑定、多模态能力及研发办公技能池。

#### 5）治理与经营闭环层

负责执行结果、审计、成本归集、模板治理和平台经营分析，确保平台具备可治理、可经营和可追踪的运行闭环。

### 4.2 系统架构说明

#### 1）前端交互页面群组

负责承接企业前台工作台、私聊页、群聊协作页、员工配置/方案应用页、企业后台和系统后台等产品交互界面。该层只承接交互与展示，不直接面向 Runtime。

#### 2）Team Panel 核心部分

负责身份与空间管理、数字员工与方案管理、能力绑定管理、任务入口与请求翻译、运行态映射与结果包装、治理与经营等业务控制能力。

其边界是：
- 负责业务对象真相层与产品化表达。
- 负责把员工、模板、团队、业务任务、权限和治理语义翻译成运行时请求。
- 不负责底层执行主循环、依赖求解、会话恢复和执行调度。
- 不需要把 Agent Gateway 的全部运行时接口逐个重包，只需要承接 AI Team 自己的业务 contract。

#### 3）Agent Gateway 核心部分

负责运行时接入接口、消息平台接入、事件输出、审批/中断与调度触发等能力，是 Team Panel 与 Agent Runtime 之间的统一运行时适配层。

其边界是：
- 在业务主链路上位于 Team Panel 下游，承接 Team Panel 生成的运行时请求。
- 负责把运行请求接入 Runtime，并把流式事件、审批状态、停止信号和调度状态统一返回给上游。
- 不负责员工、模板、权限、企业空间、经营治理等业务对象定义。

#### 4）Agent Runtime 核心部分

负责统一主循环、多入口宿主、Provider Runtime、Context/Compression/Cache、Tools/MCP/Plugins、Session Storage/Profiles、Single-Agent、Agent Group、Kanban Workflow、Cron Scheduling、Memory + Skills 以及 Multimodal I/O 等执行能力。

其继承能力重点体现在：
- **统一主循环**：由同一 Agent 主循环承接推理、tool call、retries、fallback、memory flush、persistence 等执行真相能力。
- **多入口宿主**：CLI、Gateway、Cron、ACP、Batch 共享同一运行时内核，只是交互入口不同。
- **Provider 与推理治理**：支持 OpenAI-compatible、Anthropic、Gemini、custom 等 provider 路由，并承接 api_mode、fallback、auxiliary 等运行时治理能力。
- **上下文治理**：承接 prompt assembly、context compression、prompt caching 等能力，避免前端或 Team Panel 直接承担上下文工程复杂度。
- **扩展能力面**：承接 Tools、MCP、Plugins、Memory、Skills 等可持续扩展机制；对 AI Team 而言，可直接服务于 GitHub、Docs、Slides、Web Research、Notion 等工作型能力装配。
- **持久化与隔离**：承接 Session Storage、FTS、lineage、profile-scoped isolation 等能力，用于多实例、多角色、多环境并行运行。
- **执行与多模态能力**：承接 Single-Agent、Agent Group、Kanban Workflow、Cron Scheduling，以及图片理解 / 图片生成 / 语音理解 / 语音生成等多模态 I/O 能力。

其边界是：
- 持有任务执行真相层。
- 承接真正的运行态状态与恢复能力。
- 不负责企业后台、业务对象和治理展示。

#### 5）External Capability 核心部分

负责统一接入知识引擎、技能生态、AI Relay、企业连接器与外部系统。

其边界是：
- 作为平台外部依赖能力域存在。
- 不负责内部协作编排语义。

### 4.3 主协作关系

系统主协作关系固定为：

- **产品主链**：前端交互页面群组 → Team Panel → Agent Gateway → Agent Runtime → External Capability
- **结果回流主链**：Agent Runtime / External Capability → Agent Gateway → Team Panel → 前端交互页面群组
- **专家/调试链（可选）**：内部调试台或运行时专家工作台可在受控前提下直接接入 Agent Gateway，用于调试 run、事件流和审批链路，但这不是 AI Team 主产品流的默认路径。

该主链用于约束后续接口设计和详细设计，不在当前阶段下沉到时序级细节。

## 5. 复杂度分析

架构设计的核心目的是控制系统复杂性。AI Team 当前需要重点处理的复杂度如下：

### 5.1 扩展性复杂度

平台需要同时支持单任务会话、多角色协作、周期任务、多模态任务和企业治理场景，因此要求业务控制面与执行真相层分离，避免所有能力堆在一个单体后端中。

### 5.2 一致性复杂度

业务对象状态、员工配置状态、运行态状态、执行结果与运营数据不是同一种真相源。概要设计必须提前明确：

- Team Panel 持有业务对象与配置真相。
- Runtime 持有执行态真相。
- 结果回流后再由 Team Panel 组织产品化展示对象。

### 5.3 可靠性复杂度

多员工协作、周期任务、消息入口和外部连接器都存在失败、超时和恢复问题，因此 Runtime 必须作为统一执行真相层存在，不能把失败恢复逻辑散落在 Team Panel 或前端。

### 5.4 安全与隔离复杂度

企业空间、员工实例、知识库、连接器权限与运营数据需要隔离，因此必须保证：

- 前端不直接调用 Runtime。
- Team Panel 统一承接业务语义与授权。
- External Capability 通过受控方式接入。

## 6. 风险与约束

### 6.1 技术风险

1. **Team Panel 漂移成第二套 Runtime**
- 如果业务控制层不断吸收执行逻辑，会造成边界失真。

2. **Gateway 漂移成业务后端**
- 如果运行时接入层继续吸收员工、模板、权限、经营治理等业务语义，会破坏 Gateway 作为运行时适配层的稳定性。

3. **前端绕过 Team Panel 直接面向 Runtime**
- 会导致权限、审计、运营与结果包装链路被绕开。

4. **外部能力耦合进入内部核心部分**
- 如果知识引擎、连接器和 Relay 的细节渗入 Team Panel 或 Runtime 核心设计，会降低替换能力。

### 6.2 工程约束

- 当前方案以复用既有 Agent Runtime 为前提，不重写执行底座。
- Agent Gateway 作为架构层保留，但 V1 实现优先采用基于 Hermes Python 模块的进程内适配层，而不是默认要求独立部署 Hermes 原生 API Server。
- 概要设计阶段只定稿系统级边界、能力分层与主协作关系，不展开模块内部详细实现。
- 后续详细设计必须遵守本概要设计的主边界，不应重新把业务控制、运行时接入和执行真相层混合。

## 8. MVP 范围定义

## 8.1 MVP 原则

**模块尽量全，深度适中，优先可演示。**
### 8.2 演示级必须具备的模块

以下模块必须进入 MVP，并且至少有一条可演示闭环：

| 模块                     | 核心场景覆盖       | 演示场景关联  |
| ---------------------- | ------------ | ------- |
| 登录 / 企业空间              | 用户身份与空间隔离    | 场景1-5基础 |
| 人才市场 / 专家详情 / 招募       | 数字员工获取入口     | 场景1、4   |
| 数字员工配置                 | 员工能力定义       | 场景1-5   |
| 私聊                     | 单员工一对一交互     | 场景2     |
| 群聊                     | 多员工协作交互      | 场景1     |
| 多智能体委托协作                 | 多 Agent 任务编排 | 场景1核心   |
| Loop / 定时任务            | 自主执行能力       | 场景3核心   |
| 企业知识库                  | RAG 知识问答     | 场景2核心   |
| Skill 安装与启用            | 工具扩展能力       | 场景5     |
| 连接器配置                  | 外部系统接入       | 场景5     |
| 行业方案一键应用               | 方案快速落地       | 场景4核心   |
| 记忆查看/编辑                | 员工能力沉淀       | 场景3     |
| 工资管理 / 消耗统计            | 商业化治理        | 场景5     |
| 系统后台专家管理 / 方案管理 / 财务总览 | 平台运营能力       | 场景4、5   |

### 8.3 演示级可浅实现的部分

以下能力进入 MVP，但不追求深做：

- 复杂组织树拖拽体验
- 记忆高级淘汰算法
- 全量模型与全量连接器覆盖
- 企业级复杂权限矩阵
- 大规模多 Agent 并发治理
- 完整市场运营机制
- 深度财务结算与利润分摊

### 8.4 本次 MVP 的非目标

以下不是本轮 MVP 的主要目标：

- 生产级稳定性和高并发能力
- 全行业方案覆盖
- 面向外部开发者的开放生态平台
- 完整零代码工作流编辑器
- 企业级合规体系的深做

## 9. 示例演示场景

为确保方案不是"功能列表堆砌"，MVP 至少应围绕以下场景准备演示。每个场景都明确标注了**核心验证能力**和**涉及的主要模块**。

### 场景 1：内容生产团队协作（多智能体协作编排协议核心验证）

**场景描述：**
用户从人才市场招募调研员、写作者、编辑三个角色，在群聊中输入"帮我写一篇新能源汽车行业深度报告"，通过协作编排协议完成资料收集、起草、润色与汇总。

**核心验证能力：**
- 人才市场浏览与多角色招募
- 群聊中 `@` 激活员工
- 多智能体协作编排指令解析与执行
- 子任务依赖排序与并行执行
- 协同气泡展示与任务树可视化
- 主持 Agent 结果汇总与输出

**涉及模块：** 人才市场、群聊、多智能体协作编排协议、数字员工配置、Hermes 执行基座适配

---

### 场景 2：企业知识问答 + 文档处理（RAG 核心验证）

**场景描述：**
企业上传产品手册 PDF 到知识库，绑定到"产品顾问"员工，用户在私聊中询问产品规格问题，员工基于知识库回答并展示引用来源。

**核心验证能力：**
- 知识库创建与文档上传
- 文档切块、向量化与语义检索
- 员工与知识库绑定
- 私聊对话与 RAG 问答
- 引用片段展示与来源标注
- 推理链可视化

**涉及模块：** 企业知识库、私聊、数字员工配置、记忆系统

---

### 场景 3：周期巡检 / Loop 任务（自主执行核心验证）

**场景描述：**
创建"系统监控员"员工，配置每 10 分钟执行一次服务器巡检，展示 PDAR 循环状态、最近记忆积累与告警结果。

**核心验证能力：**
- 员工级 Loop 任务创建与 Goal 配置
- PDAR（Plan-Do-Act-Reflect）循环执行
- Hermes cron 调度与状态展示
- 记忆自动提取与沉淀
- 暂停/恢复/停止控制
- 执行日志与记忆关联查看

**涉及模块：** Loop / 定时任务、数字员工配置、记忆系统、Hermes scheduler

---

### 场景 4：行业方案一键应用（方案模板核心验证）

**场景描述：**
从行业方案市场选择"电商客服团队"方案，查看包含的员工模板、技能包、知识库结构，一键应用到企业空间，自动创建对应员工并配置好基础能力。

**核心验证能力：**
- 行业方案卡片浏览与详情展示
- 方案内员工模板、技能、知识库预览
- 一键应用与批量创建
- 应用结果确认与跳转
- 系统后台方案管理与统计

**涉及模块：** 行业 AI 解决方案、人才市场、数字员工配置、Skills 中心、企业知识库

---

### 场景 5：连接器与费用治理（商业化闭环验证）

**场景描述：**
配置 GitHub 连接器并授权，设置仅"开发助手"员工可见，触发代码审查技能执行，完成后回到企业后台查看该员工 token 消耗与工资排行。

**核心验证能力：**
- 连接器配置与 OAuth/API Key 授权
- 连接器状态检测
- 员工可见性控制
- Skill 触发与执行
- Token 消耗实时统计
- 企业后台费用看板与工资排行

**涉及模块：** 连接器中心、Skills 中心、数字员工配置、商业化与统计、企业后台

---
