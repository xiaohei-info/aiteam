# AI Team 共享运行口径定稿版

## 1. 文档定位

本文不是替代五份详细设计子文档，而是作为它们共同依赖的**权威口径文档**，专门定稿以下 5 类开发前必须统一的共享契约：

1. 时间线事件协议
2. 事件游标协议
3. 控制面主状态机
4. 北向 API 最小固定契约
5. 权限角色模型

从本版开始，文档体系统一使用“**口径**”而不是“真相”来描述跨模块共享契约；若旧文档仍保留“业务真相 / 运行真相 / 执行真相”等历史表述，均应理解为“该层的权威口径”。

---

## 2. 适用范围与裁决规则

### 2.1 适用范围

本文对以下对象生效：

- `Team Panel领域模型与数据架构详细设计`
- `Agent Gateway运行时适配与事件流详细设计`
- `会话群聊编排Loop核心流程详细设计`
- `前端页面与接口契约详细设计`
- `Team Panel内部服务与聚合视图详细设计`
- 以及后续实现中的 API schema、SSE handler、前端类型定义、数据库 migration、测试用例与联调 mock

### 2.2 裁决规则

若其他详细设计文档与本文冲突，统一按本文为准；其他文档负责解释各自模块内部细节，不得再各自发明第二套共享协议。

## 3. 分层权威口径

### 3.1 Team Panel 口径

Team Panel 是业务控制面的权威口径，负责：

- 企业、成员、员工、模板、连接器授权、知识/记忆绑定
- `Conversation` / `TeamRun` / `TeamTask` / `ScheduledJob` 等产品对象
- 业务允许性、权限可见性、审计与经营统计

### 3.2 Agent Gateway 口径

Agent Gateway 是运行接入与展示结果转换口径，负责：

- 业务对象到 Runtime 请求的翻译
- `runtime_handle` 维护
- 原始 runtime 事件到 `RunTimelineEvent` 的转换
- 断流补拉、回放、对账

### 3.3 Hermes Runtime 口径

Hermes Runtime 是执行事实口径，负责：

- profile / session / task / cron 等真实执行对象
- 模型调用、tool 执行、memory flush、kanban dispatch、cron 触发
- 原始 stdout / token / tool / task / scheduler 事件

### 3.4 边界判定

- 前端不直接依赖 Runtime 原始对象。
- 北向 API 不直接暴露 Runtime 原始事件名。
- Team Panel 不把 UI 展示态直接写成持久化主状态。
- Gateway 不定义高于 Team Panel 的业务角色与业务对象。

## 4. 时间线事件统一口径

## 4.1 正式北向事件模型

前端消费的最小稳定事件单位统一为 `RunTimelineEvent`。

SSE 帧统一为：

```text
event: timeline
data: {RunTimelineEvent JSON}
```

禁止再把 `event: run_created`、`event: single_agent_started`、`event: task_stream_delta` 之类 Runtime/Gateway 内部事件名直接作为正式北向 SSE 事件名输出。

### 4.2 RunTimelineEvent 固定结构

```json
{
  "event_id": "evt_run_001_129",
  "event_cursor": 129,
  "run_id": "run_001",
  "event_type": "message_delta",
  "source_type": "session",
  "source_id": "sess_abc",
  "employee_id": "emp_marketing_001",
  "event_ts": "2026-05-28T08:00:00Z",
  "preview": "根据今天晨会内容，先给出三点摘要……",
  "payload": {
    "delta": "根据今天晨会内容，",
    "message_id": "msg_221"
  }
}
```

字段语义：

- `event_id`：事件唯一标识，用于去重与排障
- `event_cursor`：同一 `run_id` 下单调递增的数字游标
- `run_id`：业务执行实例 ID
- `event_type`：前端可稳定依赖的固定枚举
- `source_type/source_id`：该事件来源于哪个 runtime 对象
- `employee_id`：若事件与某员工相关则必带
- `event_ts`：事件发生时刻
- `preview`：给 UI 列表/时间线做轻展示的摘要字段
- `payload`：按事件类型扩展的细节载荷

### 4.3 最低固定事件枚举

V1 正式固定以下事件类型：

- `run_created`
- `routing_decided`
- `run_started`
- `message_delta`
- `tool_call`
- `task_created`
- `task_started`
- `task_completed`
- `task_failed`
- `run_waiting_human`
- `result_merged`
- `memory_written`
- `usage_recorded`
- `run_succeeded`
- `run_failed`
- `run_cancelled`
- `heartbeat`
- `error`

### 4.4 映射规则

Gateway 可在内部继续接收原始事件，如：

- `single_agent_started`
- `task_stream_delta`
- `run_completed`
- `job_tick`

但必须先映射为上面的固定枚举后再对外输出，例如：

- `single_agent_started` -> `run_started`
- `task_stream_delta` -> `message_delta`
- `run_completed` + success -> `run_succeeded`
- `run_completed` + failure -> `run_failed`

## 5. 事件游标统一口径

## 5.1 对外游标

V1 对外统一采用 **numeric cursor**：

- 类型：`bigint`
- 字段名：`event_cursor`
- 语义：同一 `run_id` 下单调递增

前端、BFF、北向 API、测试 mock、DB 查询统一按数字游标理解。

### 5.2 对内游标

Gateway 内部若需要保留 Runtime 原始游标（如 `{timestamp}-{sequence}`）或 source offset，可作为内部字段保存，例如：

- `runtime_source_cursor`
- `runtime_checkpoint`

但这些字段不直接暴露给前端，也不作为北向 API 游标。

### 5.3 API 约定

- 建立流：`GET /api/team/runs/{run_id}/stream?cursor=129`
- 断流补拉：`GET /api/team/runs/{run_id}/events?cursor=129&limit=100`
- 响应中的 `next_cursor` 仍使用 numeric cursor

### 5.4 存储约定

- `run_event.cursor_no`：正式持久化游标字段
- `runtime_binding.event_cursor`：记录最近一次已同步的 numeric cursor
- `runtime_binding.runtime_source_cursor`：可选，仅供 Gateway 内部恢复使用

## 6. 主状态机统一口径

## 6.1 设计原则

必须区分两类状态：

1. **持久化主状态**：数据库长期保存、决定对象可写性与治理语义
2. **运行展示态**：供 UI 展示当前轮次体验，不作为主表长期状态枚举

禁止把 `waiting_reply / streaming / resolved / reconnecting` 这类展示态直接混进 `conversation.status` 主字段。

## 6.2 Conversation 持久化主状态

统一口径：

- `draft`：已创建但尚未有有效首条消息
- `active`：可继续接收/发起消息
- `paused`：因权限、员工状态或人工冻结，不允许发起新执行
- `muted`：仅群聊可用；可继续存在但默认不推办公室动态
- `archived`：历史只读，不可再新增消息/成员变更

说明：

- 私聊通常使用 `draft / active / paused / archived`
- 群聊可额外使用 `muted`
- `routing / busy / waiting_reply / streaming / resolved` 一律作为展示态，不作为 `conversation.status` 主枚举

## 6.3 Conversation 展示态

允许的展示态：

- `idle`
- `routing`
- `waiting_reply`
- `streaming`
- `busy`
- `resolved`
- `reconnecting`

这些状态来源于 `conversation.status + latest_run.status + latest timeline events` 的聚合计算。

## 6.4 TeamRun 持久化主状态

统一固定为：

- `queued`
- `routing`
- `submitting`
- `running`
- `waiting_human`
- `succeeded`
- `failed`
- `cancelled`

说明：

- `waiting_children` 不再作为 `team_run.status` 主枚举，改为编排类 run 的展示态/聚合标识
- 编排子任务尚未收敛时，`team_run.status` 仍保持 `running`
- 是否“等待子任务”由 `execution_mode=kanban_orchestration` + `open_task_count > 0` 推导

## 6.5 TeamTask 持久化主状态

统一固定为：

- `planned`
- `queued`
- `running`
- `waiting_deps`
- `succeeded`
- `failed`
- `cancelled`

## 6.6 ScheduledJob 持久化主状态

统一固定为：

- `draft`
- `enabled`
- `paused`
- `error`
- `archived`

## 7. 北向 API 统一口径

## 7.1 创建 run

```http
POST /api/team/runs
Content-Type: application/json
```

幂等键统一放在 **body**，V1 不再并行定义 `X-Idempotency-Key` 为正式产品契约。

请求示例：

```json
{
  "employee_id": "emp_marketing_001",
  "conversation_id": "conv_private_001",
  "message": {
    "text": "请总结今天销售晨会的重点",
    "attachments": []
  },
  "workspace_id": "ws_default",
  "idempotency_key": "run-20260528-001"
}
```

同步响应示例：

```json
{
  "run_id": "run_001",
  "status": "queued",
  "conversation_id": "conv_private_001",
  "stream_url": "/api/team/runs/run_001/stream?cursor=0",
  "events_url": "/api/team/runs/run_001/events?cursor=0",
  "runtime_handle": {
    "kind": "session",
    "profile_name": "ent001-marketing-001",
    "session_id": "sess_abc"
  }
}
```

补充规则：
- `runtime_handle` 只承载运行句柄最小集合，不承载 `event_cursor`。
- `event_cursor` 只属于 stream/events 协议、`run_event.cursor_no` 与 `runtime_binding` 对账字段。

## 7.2 群聊消息提交

```http
POST /api/team/group-conversations/{conversation_id}/messages
Content-Type: application/json
```

请求示例：

```json
{
  "message": {
    "text": "@营销分析师 和 @财务顾问 请一起给我一个新品上市预算建议",
    "attachments": []
  },
  "route_hint": "auto",
  "idempotency_key": "groupmsg-20260528-001",
  "sender_id": "usr_001"
}
```

响应中的 `route_decision` 与 `runtime_handle` 固定为：

```json
{
  "message_id": "msg_801",
  "run_id": "run_group_001",
  "route_decision": {
    "route_mode": "orchestration",
    "target_employee_ids": ["emp_marketing_001", "emp_finance_001"],
    "planner_employee_id": "emp_orchestrator_001"
  },
  "stream_url": "/api/team/runs/run_group_001/stream?cursor=0",
  "events_url": "/api/team/runs/run_group_001/events?cursor=0",
  "runtime_handle": {
    "kind": "kanban_task",
    "session_id": null,
    "task_id": "task_root_001"
  }
}
```

固定规则：

- 在北向鉴权注入正式上线前，调用方必须显式提供 `sender_id`；`actor_id` 仅作为兼容别名临时接受。若两者都缺失，接口返回 `400 MISSING_SENDER_ID`；禁止再回落到任何默认占位值。
- `route_mode` 只允许：`single_agent | orchestration`
- 不再同时保留 `route_type` 与 `require_orchestration` 两套表达
- `group_conversation` 是业务会话容器；`orchestration` 是一次 `TeamRun` 的执行策略。两者不得在 sibling 文档中混写为同义概念。
- 当 `route_mode=single_agent` 时，群消息仍属于群聊场景，只是本轮 run 进入 session 执行路径，而不是 root-task 路径。
- `runtime_handle` 已纳入群聊消息提交的正式北向响应；它是浏览器侧断流补拉与运行对账使用的最小运行句柄，不承载 `event_cursor`。
- 群聊消息响应中的 `runtime_handle.kind` 北向只允许 `session | kanban_task`：
  - `session`：表示本轮被路由到单员工执行；此时 `session_id != null` 且 `task_id = null`
  - `kanban_task`：表示本轮被路由到协作编排；此时 `task_id != null` 且 `session_id = null`
- 群聊消息响应故意不向浏览器暴露 `profile_name`；前端只消费 `kind + session_id/task_id` 这一最小句柄集合。
- 若目标群聊存在但当前不是 `active` 状态，接口返回 `409 CONVERSATION_NOT_ACTIVE`；该错误属于资源状态冲突，而不是资源不存在。


## 7.3 开发入口第一批接口定稿清单

为支持前后端并行推进，以下 12 个接口纳入第一批必须定稿的开发入口口径：

1. `GET /api/team/workbench`
2. `GET /api/team/talent-market/templates`
3. `GET /api/team/talent-market/templates/{id}`
4. `POST /api/team/recruitments`
5. `GET /api/team/conversations/{id}`
6. `POST /api/team/runs`
7. `GET /api/team/runs/{run_id}/stream`
8. `GET /api/team/runs/{run_id}/events`
9. `POST /api/team/uploads`
10. `GET /api/team/employees`
11. `GET /api/team/employees/{id}`
12. `PATCH /api/team/employees/{id}`

原则：
- 第一批接口需要先把路径、请求体、响应体、幂等键、错误码与权限边界定稿。
- 第二批及之后的接口可以延后，但不能反向修改第一批的基础口径。

## 7.4 事件流与补拉

建立流：

```http
GET /api/team/runs/{run_id}/stream?cursor=0
Accept: text/event-stream
```

补拉：

```http
GET /api/team/runs/{run_id}/events?cursor=129&limit=100
```

响应示例：

```json
{
  "items": [
    {
      "event_id": "evt_run_001_130",
      "event_cursor": 130,
      "event_type": "task_completed",
      "preview": "财务校验完成"
    }
  ],
  "next_cursor": 130,
  "has_more": false,
  "run_status": "running"
}
```

## 8. 权限角色统一口径

## 8.1 企业侧角色

V1 统一固定为：

- `owner`
- `enterprise_admin`
- `finance_admin`
- `member`

说明：

- `owner`：企业拥有者，拥有最高企业内治理权限
- `enterprise_admin`：企业后台主要管理者，可管理员工、绑定、知识、连接器、任务配置
- `finance_admin`：企业账本、费用、导出相关权限
- `member`：普通成员，主要在前台发起对话与消费结果

## 8.2 平台侧角色

V1 统一固定为：

- `system_admin`
- `system_operator`

## 8.3 数据层落表要求

`membership.role` 与前端页面权限矩阵统一采用上面的角色集合。

不再把以下历史枚举作为正式主角色：

- `admin`
- `manager`
- `viewer`

如需细粒度差异，统一通过 `permissions_json` 表达，而不是再扩一套平级主角色。

## 9. 对现有详细设计文档的落实要求

## 9.1 数据架构文档

必须落实：

- `membership.role` 改为新角色集合
- `conversation.status` 明确只保存主状态
- `runtime_binding.event_cursor` 改为 numeric cursor
- `run_event.event_type` 改为本文固定事件枚举

## 9.2 Gateway 文档

必须落实：

- 北向 SSE 统一为 `event: timeline`
- 原始 Runtime 事件名降为内部映射输入
- `Cursor 格式: {timestamp}-{sequence}` 改为“内部 source cursor”，不再作为对外契约
- Gateway 文档中的群聊相关章节只能描述 **Group Conversation Ingress / Runtime 接入**，不得把浏览器原生群聊主流程改写成 Kanban 执行流程

## 9.3 流程文档

必须落实：

- `waiting_reply / streaming / resolved / busy` 调整为展示态
- `starting / waiting_children / reconnecting / expired` 等历史状态不再作为 `team_run.status` 主枚举
- 所有流程示例改按 `queued -> routing/submitting -> running -> terminal` 主链表达

## 9.4 前端接口文档

必须落实：

- `POST /api/team/runs` body 幂等键固定
- `route_decision` 只保留 `route_mode`
- 时间线事件枚举改为本文定义
- 页面权限矩阵与数据层角色一致
- 前端接口文档需显式声明：浏览器北向 API 的所有权在 Team Panel，Gateway 只暴露服务内适配接口

## 9.5 Team Panel 内部服务文档

必须落实：

- Team Panel 统一采用 router / command service / query service / repository / integration 分层
- 浏览器可见的页面结果统一通过聚合视图或读模型组装，不让前端自行拼规范化对象
- 群聊是业务容器、orchestration 是执行策略，这一分层必须在 Team Panel 内部服务设计中明确保留
- Team Panel 与 Gateway 的内部调用边界要与北向 API 边界分开表述

## 10. 开发启动判定

只要以下 5 项全部满足，即可解除本轮文档阻塞并进入开发：

- [ ] 五份详细设计子文档均已引用并落实本文口径
- [ ] 事件协议仅剩一套 `RunTimelineEvent` 北向模型
- [ ] 游标协议仅剩 numeric cursor 一套对外定义
- [ ] 持久化主状态与展示态已明确分层
- [ ] 权限角色模型在前端文档、数据文档、接口示例中完全一致

本文完成的作用不是增加一层新抽象，而是消除“各模块各写一套共享契约”的风险，让前端、后端、数据、联调、测试都基于同一套口径实现。
