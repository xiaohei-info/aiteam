# AI Team B05 连接器凭据安全与员工授权契约定稿版

## 1. 文档目的

本文用于在 B05 第二波实现前，收口连接器中心的共享契约，避免 backend/frontend 对以下问题各自猜测：

1. `Connector / EnterpriseConnector` 最小交付字段边界。
2. `credential_ref` 与脱敏展示的安全契约。
3. `list/detail/create/update/test/status/grants` 的北向 API 形状、错误态、权限态、状态流转。
4. 员工授权（grant/revoke）的持久化、可见性与审计边界。
5. backend-eng / frontend-eng 可并行实现的边界，以及 integration/review/QA 必须汇合的证据。

本文是 B05 lane 的唯一裁决文档；若旧文档、现有代码、页面脚本与本文冲突，以本文为准。

---

## 2. 方案调研与 adopt/adapt/borrow 判断

本次不是从零发明连接器/密钥/授权模型，而是按成熟产品共性裁决最小必要做法。

### 2.1 调研对象

1. n8n External Secrets
   - 结论：secret 不直接存业务表，按 vault/project scope 使用；使用权限与管理权限分离；project 级共享受 RBAC 控制。
2. Airbyte Secrets
   - 结论：业务配置中只保留 obfuscated secret coordinate / secret reference；运行前再 hydrate secret；支持 version 化 secret coordinate；secret 字段通过 schema 标记处理。
3. Backstage Scaffolder Permissions
   - 结论：权限不只控制“能不能执行”，也控制“能不能读取 secret-tagged 参数/步骤/任务”；对象读取与对象执行应显式分开。

### 2.2 对 AI Team 的采用判断

1. 直接采用：不采用完整外部产品/框架
   - 理由：AI Team 当前是 Team Panel + Gateway 的既有控制面，不需要引入新的 connector platform 或 credential subsystem 产品。

2. 选择性复用：采用成熟模式，不搬完整实现
   - 复用 n8n/Airbyte 的 `secret reference only` 思路：Team Panel 只存 `credential_ref`，不存明文。
   - 复用 Airbyte 的版本化思路：保留 `rotation_version` 作为凭据轮换版本口径。
   - 复用 n8n 的“使用权限”和“管理权限”分离思路：B05 至少区分 connector 管理、员工可用性、运行时注入三层。

3. 接口层借鉴：借鉴 Backstage/n8n 的细粒度访问控制
   - 列表/detail 中允许返回脱敏展示态、凭据状态、最近测试信息。
   - 原始 secret、可逆密文、secret source 细节、运行时注入值，不属于浏览器/API 响应。

4. 明确不采用
   - 不在本波引入完整 Vault UI、secret policy editor、多项目 vault share UI。
   - 不在本波把 Gateway 变成第二套凭据后台。
   - 不让前端自行拼装/缓存“伪凭据状态”；以 Team Panel 返回为唯一展示口径。

结论：本波推荐“Team Panel 持有 connector truth + opaque credential_ref；Gateway 只在运行前解析；前端只消费脱敏结果”的最小必要架构。

## 3. 边界与非目标

### 3.1 范围内

1. 企业后台 B05 连接器列表、详情、创建、编辑、测试、状态展示、员工授权。
2. `connector_definition`、`enterprise_connector`、`employee_connector_binding` 三层对象的北向暴露口径。
3. `credential_ref` / `credential_mask` / `credential_state` / `last_test_result` 的返回形状。
4. `grant/revoke` 的持久化、权限校验、审计留痕。
5. Team Panel -> Gateway 的凭据解析边界。

### 3.2 非目标

1. 不在本波定义具体 Vault 产品选型。
2. 不在本波开放浏览器侧编辑 raw secret。
3. 不在本波改动会话/群聊/编排领域模型。
4. 不要求 B05 lane 改动正在进行的会话/群聊域代码；只允许通过 Team Panel / Gateway 的 connector seam 对接。

## 4. 核心对象与字段口径

## 4.1 ConnectorDefinition（系统预设目录）

用途：系统级预设连接器定义，驱动表单与掩码/认证分支，不承载企业实例或员工授权。

最小字段：

- `definition_id`
- `provider_code`
- `connector_type`
- `display_name`
- `auth_scheme`：`oauth2 | api_key | mcp | webhook | opaque_ref`
- `config_schema_json`
- `status`：`active | deprecated | hidden`

北向语义：

- `active`：可新建新实例。
- `deprecated`：历史实例保留可见，但前端默认提示不建议新建。
- `hidden`：不在创建表单中展示，仅允许已有实例继续读取。

## 4.2 EnterpriseConnector（企业连接器实例）

用途：企业控制面中的连接器实例真相；不保存明文 secret，只保存连接器业务属性与 secret 引用。

最小字段：

- `connector_id`
- `enterprise_id`
- `definition_id`：若是自定义连接器可为空；预设连接器应回填
- `name`
- `provider_code`
- `connector_type`
- `status`
- `config`
- `scopes`
- `credential_ref`
- `credential_mask`
- `credential_state`
- `rotation_version`
- `last_test_result`
- `created_at`
- `updated_at`
- `updated_by`

字段执行语义：

1. `status`
   - `draft`：实例已创建但尚未通过测试或尚未完成必要配置；不可 grant 给员工。
   - `online`：最近一次可接受测试通过；可 grant。
   - `offline`：连接器不可达或健康检查失败；已有 grant 不自动删除，但运行时应拒绝使用。
   - `auth_failed`：凭据解析成功但认证失败；必须重新配置或轮换凭据。
   - `archived`：实例已归档；不可再 grant，不应出现在默认列表。

2. `config`
   - 只放非 secret 配置，如 endpoint、region、tenant_hint、scope flags。
   - 若字段名或 schema 判定为 secret，则只能以 `****` 出现在北向返回中。

3. `scopes`
   - 表示连接器允许的作用面，如 `invoke`、`writeback`、`sync`。
   - 前端只显示 Team Panel 返回的 scope 列表，不自行推断。

4. `credential_ref`
   - 唯一长期凭据引用口径。
   - 允许类似 `cred://enterprise/{enterprise_id}/{opaque_id}` 或 `vault://path` 的 opaque 形态。
   - 浏览器不拿它做 secret 解析，只做存在性/最近更新时间展示。

5. `credential_mask`
   - 浏览器展示字段。
   - 默认返回 `已配置` / `未配置` 或 `sk-****abcd` 这类不可逆掩码。
   - 不允许返回可恢复原值的片段。

6. `credential_state`
   - `missing | configured | rotated | invalid | revoked`
   - 与 `status` 不同：`credential_state` 描述凭据引用本身，`status` 描述连接器实例当前健康态。

7. `last_test_result`
   - 结构见 §4.4。
   - 这是 UI 最近测试展示的唯一口径，不直接拼接 `status + log` 自造对象。

## 4.3 EmployeeGrant（员工授权）

用途：表达哪些数字员工可以看到/使用某企业连接器。

最小字段：

- `binding_id`
- `connector_id`
- `employee_id`
- `employee_display_name`
- `enabled`
- `access_mode`
- `created_at`
- `updated_at`
- `updated_by`

字段执行语义：

- `enabled=true`：员工在运行时可请求 Gateway 解析该 connector。
- `enabled=false`：历史绑定存在但当前不可用；前端在详情中可见，在“已授权员工”聚合中不可见。
- `access_mode`
  - `invoke`：只允许调用。
  - `invoke_and_writeback`：允许调用并允许回写/同步结果到外部系统。

## 4.4 LastTestResult（最近测试结果）

最小字段：

- `result`: `passed | failed | skipped | never_tested`
- `checked_at`
- `checked_by`
- `error_code`
- `message`
- `log_ref`

约束：

- `message` 只能返回脱敏后的摘要，不得包含 secret/raw header/raw token。
- `log_ref` 是审计/日志系统中的引用，不是大段日志正文。
- 前端若需要展开日志，只能通过受控 audit/log API 拉取脱敏内容。

## 5. credential_ref 安全契约

## 5.1 强制约束（non-negotiable）

1. Team Panel DB 不落 raw secret。
2. 浏览器/API 响应不返回 raw secret、可逆密文、refresh token、完整 API key。
3. Team Panel 业务表中只持久化：
   - `credential_ref`
   - `rotation_version`
   - `credential_state`
   - `updated_at`
   - `updated_by`
   - `last_test_result` 的脱敏元数据
4. Gateway 只在运行前按 `credential_ref` 解析 secret；解析结果只存在于进程内 env overlay 或 0600 权限 bundle。
5. 任意日志、异常、审计 payload 都不得落明文 secret。

## 5.2 允许返回给前端的信息

- `credential_ref`（opaque ref，可显示但不能用于取回 secret）
- `credential_mask`
- `credential_state`
- `rotation_version`
- `updated_at`
- `updated_by`
- `last_test_result.checked_at`
- `last_test_result.result`
- `last_test_result.error_code`
- `last_test_result.message`（脱敏摘要）

## 5.3 不允许返回给前端/API 的信息

- `api_key`
- `client_secret`
- `refresh_token`
- `access_token`
- 明文 webhook signing secret
- 未脱敏 config secret 字段
- 运行时 env overlay / file bundle 内容

## 5.4 轮换与兼容语义

- 每次凭据轮换必须使 `rotation_version +1`。
- 新 run 使用最新版本；已启动的旧 job 可继续使用旧版本直到自然结束。
- `credential_state=rotated` 可作为 UI 中“已更新待复测”的中间态。

统一前缀：`/api/team/connectors`

## 6.2 GET /api/team/connectors/{connector_id}

用途：详情/编辑页拉取单实例完整脱敏信息。

响应与列表单项同构，但可以额外返回：

- `definition`：预设定义对象
- `audit_summary`
- `available_employees`：供授权弹窗选择的员工清单（只返回 id/display_name/status）

## 6.3 POST /api/team/connectors

用途：创建实例。

请求：

```json
{
  "definition_id": "def_slack_webhook",
  "name": "公司 Slack",
  "provider_code": "slack",
  "connector_type": "webhook_target",
  "config": {"channel": "#sales"},
  "credential_input": {
    "mode": "opaque_ref",
    "credential_ref": "cred://ent_xxx/abc123"
  },
  "idempotency_key": "b05-create-001"
}
```

约束：

- 若本次是录入新 secret，浏览器提交应走受控 secret 写入路径；Team Panel 只接收结果性的 `credential_ref`，而不是把 raw secret 直接回写到 `enterprise_connector`。
- 最小返回：`connector_id`、`status`、`credential_state`、`updated_at`。

默认状态：

- 创建后默认 `status=draft`
- 若缺少凭据则 `credential_state=missing`
- 若凭据已存在待测试则 `credential_state=configured`

## 6.4 PATCH /api/team/connectors/{connector_id}

用途：更新非 secret 配置、切换 `credential_ref`、归档等。

允许字段：

- `name`
- `config`
- `scopes`
- `credential_input`（结果必须仍然收口为新的 `credential_ref`）
- `archive` / `unarchive`

禁止：

- 直接 PATCH raw secret 字段
- 前端直接 PATCH `status=online`

状态规则：

- 更新配置或轮换凭据后，若影响联通性，必须把 `status` 收回 `draft` 或 `offline`，等待复测。

## 6.5 POST /api/team/connectors/{connector_id}/test

用途：执行连接测试。

请求：

```json
{
  "mode": "manual",
  "dry_run": false
}
```

响应：

```json
{
  "connector_id": "conn_xxx",
  "result": "passed",
  "status": "online",
  "last_test_result": {
    "result": "passed",
    "checked_at": "2026-06-07T10:00:00Z",
    "checked_by": "user_1",
    "error_code": "",
    "message": "最近一次连接测试通过",
    "log_ref": "audit://connector-test/123"
  }
}
```

强约束：

- 不返回 credential。
- 不返回未经脱敏的测试日志。
- `passed -> status=online`
- `failed` 且原因为认证失败 -> `status=auth_failed`
- `failed` 且原因为网络/服务不可达 -> `status=offline`

## 6.6 GET /api/team/connectors/{connector_id}/status

用途：轻量轮询状态，不强制前端每次回拉完整 detail。

响应字段：

- `connector_id`
- `status`
- `credential_state`
- `last_test_result`
- `updated_at`

## 6.7 PATCH /api/team/connectors/{connector_id}/grants

用途：批量授权/撤销员工访问。

请求：

```json
{
  "grant": [
    {
      "employee_ids": ["emp_a", "emp_b"],
      "access_mode": "invoke"
    }
  ],
  "revoke": [
    {
      "binding_id": "cb_old"
    }
  ]
}
```

响应：

```json
{
  "granted": [
    {"binding_id": "cb_new", "employee_id": "emp_a"}
  ],
  "revoked": ["cb_old"],
  "errors": []
}
```

授权规则：

- 只有 `status=online` 的 connector 才允许新 grant。
- `draft/offline/auth_failed/archived` 不允许新 grant。
- revoke 不要求 connector online；历史无效绑定也允许回收。
- `finance_admin`、`member` 无权 grant/revoke。

## 7. 错误态、权限态、状态流转

## 7.1 权限矩阵

- `owner`：可 list/detail/create/update/test/grants/status
- `enterprise_admin`：可 list/detail/create/update/test/grants/status
- `finance_admin`：全部 B05 接口 403
- `member`：全部 B05 接口 403
- `system_admin/system_operator`：不经企业侧入口直接操作 B05；若需要系统巡检，应走系统后台专用接口，不复用企业后台契约

## 7.2 统一错误码

- `MISSING_BODY`
- `CONNECTOR_NOT_FOUND`
- `CONNECTOR_NOT_ONLINE`
- `CONNECTOR_ARCHIVED`
- `CREDENTIAL_REQUIRED`
- `CREDENTIAL_INVALID`
- `INVALID_FIELD`
- `INVALID_STATUS_TRANSITION`
- `EMPLOYEE_NOT_FOUND`
- `PERMISSION_DENIED`
- `AUDIT_WRITE_FAILED`

## 7.3 状态机

```text
create -> draft

 draft --test passed--> online
 draft --test auth failed--> auth_failed
 draft --test infra/network failed--> offline

 online --credential rotate/config change--> draft
 online --test failed auth--> auth_failed
 online --test failed infra--> offline

 offline --test passed--> online
 auth_failed --credential update--> draft

 any non-archived --archive--> archived
 archived --unarchive--> draft
```

失败路径要求：

- grant/revoke 失败只影响本次批量中的对应项，不得伪造整体成功。
- test 失败必须写入 `last_test_result`，并更新 `status`。
- 若 audit 留痕失败，写接口应整体失败，不允许“主写成功、审计缺失”。

## 8. 员工授权语义与可见性

## 8.1 谁能看到 connector

1. 管理页（B05 页面）
   - 仅 `owner/enterprise_admin` 可见全部连接器与全部 grant 细节。
2. 员工配置页（B01 connector tab）
   - 管理员可查看某员工已被授予的 connectors。
3. 员工运行时
   - 员工本人不需要在 UI 直接看到 secret 或完整 connector 管理对象。
   - Runtime/Gateway 只需根据 `employee_connector_binding(enabled=true)` 判定该员工是否可使用。

## 8.2 未授权员工表现

- 在员工配置/详情中：不出现在“已授权连接器”清单。
- 在运行时：请求解析 connector 时返回不可用/未授权错误，不回退为默认共享连接器。
- 在前端：不伪造为“可见但禁用”的共享资源，除非页面明确是管理员视角。

## 8.3 grant/revoke 持久化与审计边界

1. grant
   - 新建或复活 `employee_connector_binding`
   - 写 `updated_by`
   - 记录审计事件：`connector_grant_created` / `connector_grant_reenabled`
2. revoke
   - soft delete 或 `enabled=false`，但北向口径统一视为 revoke
   - 记录审计事件：`connector_grant_revoked`
3. connector 写操作
   - `connector_created`
   - `connector_updated`
   - `connector_tested`
   - `connector_archived`
   - `connector_unarchived`
   - `connector_credential_rotated`

## 12. B05 lane 隔离结论

1. B05 的正式实现工作区应保持在 `wt/b05-connectors` 或 dedicated lane 中完成。
2. 本波不要求改动会话/群聊域主模型、SSE 协议、群聊编排流程。

## 13. 最终推荐设计

推荐设计：

- Team Panel 持有 `connector_definition + enterprise_connector + employee_connector_binding` 的业务真相。
- Team Panel 只存 `credential_ref` 与脱敏元数据，不存明文 secret。
- Gateway 在运行前解析 `credential_ref`，解析结果不经浏览器。
- B05 北向正式接口冻结为：`list/detail/create/update/test/status/grants`。
- 员工授权以 `employee_connector_binding` 为唯一持久化口径，只有 `online` connector 可新 grant。

边界：

- 不引入新的 vault 控制面产品。
- 不重做会话/群聊域。
- 不把 Gateway 升级成第二套管理后台。

风险：

1. 若 backend 继续沿用当前简化接口，会导致前端继续本地伪造 credential/test 状态。
2. 若不补权限与审计，B05 会留下明显安全债务。
3. 若 grant 与运行时注入口径不一致，会出现“UI 已授权但运行时报未授权”的集成故障。

验证路径：

1. API 合约测试：覆盖 list/detail/create/update/test/status/grants 的成功/失败/403/404。
2. 前端集成测试：覆盖创建、测试、授权、脱敏展示、权限拒绝、刷新恢复。
3. 审计验证：写操作后 audit_event 中存在 connector 专项事件且无明文 secret。
4. 运行时验证：被授权员工可用、未授权员工不可用、轮换后新旧版本行为符合 `rotation_version` 语义。
