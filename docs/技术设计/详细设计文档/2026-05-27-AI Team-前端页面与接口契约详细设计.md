# AI Team 前端页面与接口契约详细设计

## 1. 文档目标

本文是《详细设计方案》的前端与北向接口拆分子文档，目标是把以下三件事收敛到可开发状态。跨文档共享契约（事件协议、游标、主状态机、北向 API、权限角色）统一以《2026-05-28-AI Team-共享运行口径定稿版》为准：

1. 把 PRD 中 22 个页面，按**企业前台 / 企业后台 / 系统后台**重新映射到已验证架构。
2. 明确每个页面真正读写的**业务对象**、推荐的**前端路由**、对应的 **BFF / Team Panel API**。
3. 给出前后端并行开发所需的**契约样例、SSE 事件约定、分页筛选排序规范、权限可见性、空载错态、附件上传路径与复用边界**。

本文遵循的核心边界：

- 前端不直接依赖 Hermes Runtime 内部对象。
- 前端稳定依赖对象是 **Team Panel 北向业务 API**。
- PRD 中出现的伪接口只作为页面意图参考，**不能覆盖**已在主详细设计中确认的架构：
  `前端页面 -> Team Panel API -> Agent Gateway 适配层 -> Hermes Runtime`。

## 2. 设计原则与契约总原则

## 2.1 页面层原则

- **页面表达业务对象，不表达 Runtime 对象**。
  - 页面展示“数字员工 / 群聊 / 行业方案 / 记忆条目 / 连接器授权”。
  - 不展示 `profile path`、`session file path`、`kanban internals` 等运行时内部实现。
- **工作台类页面走聚合视图模型**。
  - 如 P02 工作台、P09 办公室动态优先拿聚合接口，避免前端拼十几个对象接口。
- **会话页与配置页分离**。
  - P05/P06 负责执行与消费事件流。
  - B01/B05/B07/B08 负责配置、授权与治理。

## 2.2 接口层原则

- 北向接口统一收敛到 `/api/team/*`、`/api/enterprise-admin/*`、`/api/system-admin/*` 三组，不直接暴露 WebUI 内部 `/api/sessions` 作为正式产品契约。
- 这些北向接口的**所有权在 Team Panel**；Agent Gateway 只承接 Team Panel 服务层转下来的内部适配请求，不直接作为浏览器业务 API 的权威定义层。
- 所有“创建型接口”必须支持 `idempotency_key`。
- 所有列表接口统一支持 `page` / `page_size` / `sort_by` / `sort_order` / `filters`。
- 所有运行态接口统一支持 `run_id`、`event_cursor`、`stream_url`。
- 群聊与编排页面消费的最小稳定事件单位是 `RunTimelineEvent`，不是纯 token 文本流。
- `POST /api/team/group-conversations/{id}/messages` 表达的是“群消息提交”；该接口返回的 `route_decision.route_mode` 决定本轮 `TeamRun` 走单 Agent 还是编排路径，但不把“群聊”本身降格成单纯的 Kanban 任务。

## 2.3 复用判断原则

正式落地载体为 `Agent Service`，其在当前仓库中的代码目录为 `/home/ubuntu/code/aiteam/app`。它是基于 Hermes Web UI 二次开发的项目；前端页面、北向 API 与 Agent Gateway 适配层都在这一代码库内演进，而不是另外再起一套前端工程。

### 可直接复用 Agent Service / Hermes Web UI 基座的部分

- `static/index.html` 三栏工作台骨架
- `static/ui.js` 的 Markdown 渲染、消息卡片、Toast/Dialog、离线检测等 UI 组件
- `static/messages.js` 的 SSE 客户端连接与消息区增量渲染机制
- `static/panels.js` 的 panel / drawer / switchPanel 交互框架
- `api/upload.py` 的 multipart 上传基础能力
- `api/workspace.py` 的 workspace 文件挂载与安全路径解析
- `server.py` + `api/routes.py` 的 HTTP 宿主壳

### 只能借鉴、不能直接复用的部分

- session 列表语义
- profile 切换语义
- auth 体系
- cron 原生对象直出
- 企业连接器与权限模型
- 企业账本与系统后台对象

### 必须新建的部分

- 人才市场与专家详情
- 企业工作台聚合模型
- 群聊 TeamRun 时间线
- 组织架构
- 办公室动态可视化
- 行业方案一键应用
- 企业后台与系统后台全部业务页面

### 2.4 代码落点口径

前端与北向接口推荐按 Agent Service 项目内部分层组织：

```text
app/
├── team-panel/
│   ├── pages/          # 企业前台 / 企业后台 / 系统后台页面与页面级脚本
│   ├── api_team/       # /api/team/*、/api/enterprise-admin/*、/api/system-admin/*
│   └── models/         # 前端聚合视图依赖的业务对象定义
├── static/             # 继承上游的三栏 UI / SSE / Markdown / Panel 框架
├── api/                # 继承上游的 routes / upload / workspace / helper 基座
└── server.py           # 同进程 HTTP 宿主入口
```

口径上应坚持：

- `api/routes.py` **不做减法只做加法**，保留既有宿主能力并扩展 `/api/team/*` 北向接口。
- 页面脚本可继续借力 `static/*.js`，但业务对象与页面模式要收敛到 `team-panel/pages/` 对应语义。
- 前端不会直接依赖 `api/chat/start` 作为正式产品契约，而是通过 Team Panel 北向 API 间接进入 Agent Gateway。

### 2.5 对扩展开放、对修改封闭的前端/BFF实施原则

- Team Panel 新页面和新业务接口默认通过 **新增模块** 扩展，而不是继续把业务分支压进已有巨型基座文件。
- 新页面优先落在 `team-panel/pages/`；新业务接口优先落在 `team-panel/api_team/*`。
- `server.py` / `api/routes.py` 只承担**挂接与 fallback**：
  - 命中新命名空间时，转给 Team Panel 新 router
  - 未命中时，继续回退给现有宿主 `api/routes.py`
- 原有宿主接口仍继续被使用，但主要承担：
  - 静态资源和宿主能力
  - workspace / upload / settings / 兼容层接口
  - 旧会话 / 调试宿主接口
- 任何正式产品页面都不应直接把旧宿主 session/chat API 当成长期稳定产品契约。

开发时的快速判断标准：

1. 新增 `/api/team/*` 接口时，首先考虑新增 `team-panel/api_team/*.py`，不是直接改 `api/routes.py` 主体。
2. 新增群聊/编排/后台页面时，首先考虑新增 `team-panel/pages/*` 及其聚合视图，而不是在 `static/sessions.js` / `static/panels.js` 中硬塞大量业务分支。
3. 若现有基座能力与 API 语义混淆，先适度抽取底层可复用 helper，再由新接口模块调用。

## 3. 推荐前端应用结构与路由分组

## 3.1 应用分区

建议前端按 4 个应用壳组织，而不是把 22 个页面堆在一个无边界 SPA 中：

1. **认证壳**：登录、会话恢复、企业选择
2. **企业前台壳**：工作台、私聊、群聊、组织、知识库、办公室动态
3. **企业后台壳**：员工配置、技能、连接器、行业方案、记忆、费用、设置
4. **系统后台壳**：企业账号、模板发布、方案发布、平台财务

## 3.2 推荐路由

```text
/auth
  /login                               -> P01

/app
  /workbench                           -> P02
  /marketplace                         -> P03
  /marketplace/:templateId             -> P04（抽屉或独立详情）
  /chat/:conversationId                -> P05
  /group/:conversationId               -> P06
  /org                                 -> P07
  /knowledge                           -> P08
  /office                              -> P09

/admin
  /employees                           -> B01
  /skills                              -> B02
  /billing/usage                       -> B04
  /connectors                          -> B05
  /solutions                           -> B06
  /memories                            -> B07
  /settings                            -> B08
  /billing/recharge                    -> B09

/system
  /accounts                            -> S01
  /templates                           -> S02
  /solutions                           -> S03
  /finance                             -> S04
```

## 3.3 路由实现建议

- P04 建议实现为 `/app/marketplace/:templateId` + 抽屉式渲染，兼容直接访问与列表内弹层体验。
- P05 与 P06 分路由而不是同页模式切换，因为它们依赖的流式契约不同。
- B01 中员工详情建议使用嵌套路由：`/admin/employees/:employeeId/:tab?`，承载模型、Prompt、Skill、KB、Memory、Connector、Loop 子页签。
- 系统后台与企业后台必须物理隔离路由前缀，避免角色穿透与布局混淆。

- `/api/team/recruitments/*`
- `/api/team/templates/*`（企业后台视角仅可读市场模板与企业模板映射）

### D. Employee / Capability Binding

- `/api/team/employees/*`
- `/api/team/skills/*`
- `/api/team/knowledge-bases/*`
- `/api/team/memories/*`
- `/api/team/connectors/*`

### E. Org / Office / Analytics

- `/api/team/org/*`
- `/api/team/office/*`
- `/api/team/billing/*`
- `/api/team/audit/*`

### F. Enterprise Admin

- `/api/enterprise-admin/*`

### G. System Admin

- `/api/system-admin/*`

## 4.2 BFF 组织建议

建议在 Team Panel API 层进一步按前端页面组织 BFF 聚合器：

- `workbench_bff`
- `conversation_bff`
- `marketplace_bff`
- `employee_admin_bff`
- `knowledge_bff`
- `office_bff`
- `billing_bff`
- `system_admin_bff`

原则：

- BFF 可以聚合多个领域对象返回一个 ViewModel。
- BFF 不重定义底层领域对象口径，只做页面级裁剪、拼装、排序和权限过滤。

### 企业前台关键说明

#### P02 工作台

- 复用 Hermes Web UI 左中右布局骨架，但左侧主对象改为 `员工 + 会话摘要`，不再是 session 原生列表。
- 空态来自 `WorkbenchView.empty_state`，而不是前端硬编码判断空数组。

#### P05 单聊页

- 右侧员工详情仅展示业务字段：岗位、模型、技能、知识库、近期运行统计。
- “配置员工”按钮跳转 `/admin/employees/:employeeId`，不在前台直接编辑运行配置。

#### P06 群聊页

- 顶部群信息区展示：群名、成员、默认协作策略、最近运行状态。
- 中间消息区展示用户文本与员工最终回复。
- 右侧或底部时间线展示 `RunTimelineEvent`，解决多员工协作可解释性问题。

#### P08 知识库页

- 读取对象口径在 Team Panel，不直接展示 LightRAG 内部 chunk 或向量细节。
- 文档状态建议最少支持：`uploaded / parsing / indexing / ready / failed`。

#### P09 办公室动态

- 主要读聚合只读视图，不承担配置写操作。
- 点击工位跳转到对应 `conversationId` 或 `employeeId` 的详情页，避免办公室页沦为第二个工作台。

## 7.2 S01 EnterpriseAdminAction

S01 的企业治理动作统一采用一个 northbound 写入口：

```http
POST /api/system-admin/enterprises/{enterprise_id}/actions
Content-Type: application/json
```

正式请求体：

```json
{
  "action": "ban",
  "reason": "多次违规调用",
  "amount": null,
  "message": null,
  "level": null,
  "idempotency_key": "sysacct-ent_001-ban-20260603-01"
}
```

字段与 action 语义：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `action` | enum | 唯一正式枚举：`ban | unban | recharge | notify` |
| `reason` | string\|null | `ban/unban` 建议填写，作为审计理由 |
| `amount` | integer\|null | 仅 `recharge` 使用，单位为最小货币单位整数 |
| `message` | string\|null | 仅 `notify` 使用，表示通知正文 |
| `level` | enum\|null | 仅 `notify` 使用，枚举：`info | warning | critical`，默认 `info` |
| `idempotency_key` | string\|null | `recharge/notify` 建议传入，便于安全重试 |

action 约束：

| action | 必填字段 | 成功语义 |
| --- | --- | --- |
| `ban` | `action` | 企业状态切到封禁态，并留下审计记录 |
| `unban` | `action` | 企业状态解除封禁，并留下审计记录 |
| `recharge` | `action`, `amount` | 记录一次人工充值/余额调整 |
| `notify` | `action`, `message` | 记录一次系统通知投递（Phase 2 允许 mock/outbox） |

最小成功响应：

```json
{
  "enterprise_id": "ent_001",
  "action": "ban",
  "status": "succeeded",
  "message": "enterprise action applied",
  "audit_event_id": "audit_001"
}
```

说明：
- split `/ban|/recharge|/notify` 不再作为正式页面契约
- 若 backend 为兼容保留 legacy alias，也只能视为内部过渡能力，frontend/QA/正式 contract tests 一律只认 `/actions`

## 7.4 群聊消息提交

```http
POST /api/team/group-conversations/conv_group_001/messages
Content-Type: application/json
```

```json
{
  "message": {
    "text": "@营销分析师 和 @财务顾问 请一起给我一个新品上市预算建议",
    "attachments": []
  },
  "route_hint": "auto",
  "idempotency_key": "groupmsg-20260527-001",
  "sender_id": "usr_001"
}
```

说明：

- 在北向鉴权注入正式上线前，调用方必须显式提供 `sender_id`；`actor_id` 仅作为兼容别名临时接受。
- 若 `sender_id` 与 `actor_id` 都缺失，接口返回 `400 MISSING_SENDER_ID`；禁止静默回落到示例账号或测试占位值。

```json
{
  "message_id": "msg_801",
  "run_id": "run_group_001",
  "route_decision": {
    "route_mode": "orchestration",
    "target_employee_ids": ["emp_marketing_001", "emp_finance_001"],
    "planner_employee_id": "emp_orchestrator_001"
  },
  "stream_url": "/api/team/runs/run_group_001/stream?cursor=0",
  "events_url": "/api/team/runs/run_group_001/events?cursor=0",
  "runtime_handle": {
    "kind": "kanban_task",
    "session_id": null,
    "task_id": "task_root_001"
  }
}
```

说明：

- `runtime_handle` 是群聊消息提交的正式北向响应字段，用于浏览器断流补拉与运行对账。
- `runtime_handle.kind` 在该接口只允许 `session | kanban_task`：
  - `session`：本轮被路由到单员工执行；此时 `session_id != null`、`task_id = null`
  - `kanban_task`：本轮被路由到协作编排；此时 `task_id != null`、`session_id = null`
- 该接口故意不暴露 `profile_name`；前端只依赖 `kind + session_id/task_id` 这一最小句柄集合。

#### 错误码

| 状态码 | 错误码 | 说明 |
| --- | --- | --- |
| 400 | MISSING_BODY | 缺少请求体 |
| 400 | MISSING_MESSAGE_TEXT | `message.text` 为空 |
| 400 | MISSING_SENDER_ID | 在鉴权注入前既未提供 `sender_id` 也未提供兼容别名 `actor_id` |
| 400 | INVALID_CONVERSATION_TYPE | 指向的会话不是群聊 |
| 404 | CONVERSATION_NOT_FOUND | 会话不存在 |
| 409 | CONVERSATION_NOT_ACTIVE | 会话存在但当前不是 `active`，不能接受新消息 |

#### Patch语义

#### 响应

```json
{
  "employee_id": "emp_marketing_001",
  "display_name": "高级营销分析师",
  "status": "paused",
  "reprovision_status": null,
  "updated_at": "2026-05-28T11:00:00Z"
}
```

#### 异步场景

当 `status: active` 且触发 profile reprovision 时：

```json
{
  "employee_id": "emp_marketing_001",
  "status": "active",
  "reprovision_status": "pending",
  "reprovision_job_id": "job_reprovision_001",
  "updated_at": "2026-05-28T11:00:00Z"
}
```

前端应轮询或订阅 reprovision 完成事件。

#### 错误码

| 状态码 | 错误码 | 说明 |
|--------|--------|------|
| 404 | EMPLOYEE_NOT_FOUND | 员工不存在 |
| 403 | EMPLOYEE_UPDATE_FORBIDDEN | 无权限更新该员工 |
| 400 | INVALID_STATUS_TRANSITION | 状态迁移不合法（如 archived -> active） |
| 409 | REPROVISION_IN_PROGRESS | 正在进行 reprovision，请稍后重试 |

#### 权限边界

- 企业管理员可更新本企业员工
- 企业成员仅可更新自己创建的员工（部分字段）
- 系统管理员可更新所有企业员工

## 8. SSE / Timeline 消费契约

## 8.1 为什么不能只消费 token 流

对于 AI Team，前端至少要同时处理三类运行信息：

1. **最终消息流**：给用户看的文本/图片/文件结果
2. **协作过程流**：谁在路由、谁接单、谁完成、谁失败
3. **状态恢复流**：断线后如何用 `event_cursor` 补拉

因此，正式契约必须以 `RunTimelineEvent` 为主，token 只是其中一个 `event_type`。

## 8.2 SSE 事件帧格式

正式北向 SSE 统一采用：

```text
event: timeline
data: {RunTimelineEvent JSON}

```

事件体：

```json
{
  "event_id": "evt_run_group_001_129",
  "event_cursor": 129,
  "run_id": "run_group_001",
  "event_type": "task_started",
  "source_type": "kanban_task",
  "source_id": "task_child_004",
  "employee_id": "emp_finance_001",
  "preview": "财务校验开始",
  "event_ts": "2026-05-27T10:32:20Z",
  "payload": {
    "parent_task_id": "task_root_001",
    "phase": "worker",
    "message_anchor": "msg_221"
  }
}
```

## 8.3 最低固定事件枚举

- `run_created`
- `routing_decided`
- `run_started`
- `message_delta`
- `tool_call`
- `task_created`
- `task_started`
- `task_completed`
- `task_failed`
- `run_waiting_human`
- `result_merged`
- `memory_written`
- `usage_recorded`
- `run_succeeded`
- `run_failed`
- `run_cancelled`
- `heartbeat`
- `error`

## 8.4 前端渲染约定

### P05 单聊页

- `message_delta` 合并进当前 assistant message bubble
- `tool_call` 渲染为工具调用卡片
- `usage_recorded` 可延迟渲染到侧栏统计

### P06 群聊页

- `routing_decided` 渲染为“正在分配员工/决定协作方式”
- `task_created / task_started / task_completed` 渲染到时间线或任务树
- 最终合并结果由 `result_merged` 驱动主回复消息

### P09 办公室动态

- 不直接订阅全量 token 流
- 只消费 Presence/TaskDigest 聚合流，避免事件过载

## 8.5 断流恢复约定

### 建立流

```http
GET /api/team/runs/{run_id}/stream?cursor=0
Accept: text/event-stream
```

### 补拉

```http
GET /api/team/runs/{run_id}/events?cursor=129&limit=100
```

```json
{
  "items": [
    {
      "event_id": "evt_000130",
      "event_cursor": 130,
      "event_type": "task_completed",
      "preview": "财务校验完成"
    }
  ],
  "next_cursor": 130,
  "has_more": false,
  "run_status": "running"
}
```

约定：

- SSE idle heartbeat：15s 一次
- 前端断流后 60s 内必须可用 cursor 补拉
- UI 不以“浏览器是否还连着 SSE”判断 run 是否失败
- 以 `run_status + latest_event_cursor` 为准

## 12. 附件 / 上传 / 文件路径设计

## 12.1 设计原则

- 上传先进入 **Team Panel 资产区**，不直接写 Hermes profile 目录。
- 前端只持有 `asset_id / preview_url / filename`，不持有真实磁盘路径。
- 运行时需要用附件时，由 Gateway 把资产临时挂载到运行 workspace。

## 12.2 推荐资产路径

### 对象存储前缀

```text
aiteam/{enterprise_id}/{scope}/{yyyy}/{mm}/{dd}/{asset_id}/{filename}
```

### 本地落盘兼容路径

```text
/var/lib/aiteam/assets/{enterprise_id}/{scope}/{yyyy}/{mm}/{dd}/{asset_id}/{filename}
```

### scope 建议枚举

- `conversation_attachment`
- `knowledge_raw`
- `enterprise_logo`
- `template_asset`
- `solution_asset`
- `connector_temp`

## 12.3 前端上传接口建议

- `POST /api/team/uploads`
- `GET /api/team/uploads/{asset_id}`
- `GET /api/team/uploads/{asset_id}/preview`
- `DELETE /api/team/uploads/{asset_id}`（仅未绑定对象时）

## 12.4 会话附件到 Runtime 的桥接

流程建议：

1. 浏览器上传，获得 `asset_id`
2. 发起 `POST /api/team/runs` 时仅传 `asset_id`
3. Gateway 将 `asset_id` 解析为临时 workspace 文件
4. Runtime 只看运行期安全路径，不知道企业资产存储结构
5. run 结束后临时挂载可清理，资产主副本仍在 Team Panel 资产区

## 16. 对开发团队的直接结论

1. **P05 与 P06 必须分开建模，不能共用一个“聊天接口 + 文本流”思路。**
2. **工作台要以 `WorkbenchView` 聚合返回，不能让前端自己拼员工、会话、办公室摘要。**
3. **员工配置页写的是业务对象，Gateway 再翻译成 `ProfileProvisioningPlan`；前端不得直接编辑 Runtime 文件。**
4. **知识库上传必须先入 Team Panel 资产区，再桥接到运行 workspace。**
5. **B03 与 S02 不能混淆：企业后台消费模板，系统后台发布模板。**
6. **费用页至少拆为“消耗看板”和“余额/充值”两个视图，否则权限与数据粒度会混乱。**
7. **前端真正稳定依赖的会话契约是 `run_id + stream_url + event_cursor`，不是 WebUI 内部 session API。**
8. **前后端并行开发的最小可行定稿面，是工作台、模板市场、招募、单聊 run、SSE、员工详情 6 组接口。**
