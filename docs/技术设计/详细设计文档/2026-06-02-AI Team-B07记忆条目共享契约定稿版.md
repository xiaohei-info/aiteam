# 1. 结论先行

本定稿版用于裁决 Batch2 B07 的唯一共享口径，解决当前 data-rd 与 backend 两条实现线对 `memory_item` 的不兼容定义。

本次裁决结论：
- Batch2 的 canonical 业务对象是 `MemoryItem`，但它不是 Hermes 原始 memory provider 的直接镜像，而是 Team Panel 面向治理/审计/检索注入的“员工记忆条目”对象。
- persistence 层与 northbound API 层共用同一套字段语义，但不要求一字不差同形：数据库保留租户/审计列，北向 API 输出治理视图与派生 trace 视图。
- `importance` 的唯一正式口径固定为 `int(1..5)`；不再接受 `0..1`、`0..10` 或其他并行口径。
- `category` 的唯一正式口径固定为 `preference | habit | decision | event`。
- `source_type` 的唯一正式口径固定为 `manual | extraction | system_policy`。
- `runtime_ref` 的唯一正式口径固定为 JSON 对象，用来表达该条记忆的运行时来源/证据引用；不再同时维护 `extraction_run_id` 与 `runtime_ref_json` 两套并行来源字段。
- 主干唯一权威迁移文件固定为 `app/team_panel/migrations/004_memory_item.sql`；不再接受 `004_memory_item_contract.sql` 作为正式文件名。
- canonical repo/query surface 以企业作用域为主：列表/筛选走 `list_by_enterprise(...)`，分页 envelope/trace 展开留在 query service / router；员工作用域仅作为 create / delete / bulk_delete / policy guard 的从属边界。
- `MemoryReviewDecisionRepo` 的 canonical 命名固定为 `get_latest_by_memory_id(...)` 与 `latest_by_memory_ids(...)`；`get_by_memory_item(...)` 只允许作为过渡兼容别名，不能继续出现在正式测试与文档中。
- `prompt_use_trace` 不是 `memory_item` 主表持久化字段，而是从 `run_event(event_type=memory_written)` 派生出来的读侧 trace 视图。
- `memory_review_decision` 保留为 companion aggregate，用来记录人工审核轨迹；北向 `MemoryItem.review` 是对最新审核状态的聚合投影，不要求把完整审核历史平铺回 `memory_item` 主表。
- 自动提取失败不落半成品 `memory_item`；失败只落 `audit_event(memory.extraction_failed)` 或等价审计事件。只有真正被接受进入记忆池的条目才创建 `memory_item` 行。

如果下游实现、测试、文档与本定稿版冲突，以本文件为准。

# 2. 方案调研与复用判断

本次没有直接采用外部记忆系统 schema，而是做“接口层借鉴 + 语义层借鉴”，理由如下：

## 2.1 Mem0（官方文档）
观察点：
- 把“对话消息输入”与“异步记忆提取完成事件”分开，`add memories` 返回 `event_id`，说明抽取流程和最终条目不是同步强绑定。
- 支持 `user_id / agent_id / run_id / metadata` 这类 provenance 元数据。

可借鉴点：
- 借鉴“提取流程异步、条目结果持久化、保留 run/session 元数据”的模式。
- 不直接采用其平台 API 结构，因为 AI Team 的 northbound 需要的是企业后台可审计 CRUD 列表，而不是纯 provider API。

复用判断：接口层借鉴，不直接采用。

## 2.2 Zep Facts（官方文档）
观察点：
- 把 fact 看成可时间化、可失效、可检索的长期知识单元。
- 明确区分 `created_at / valid_at / invalid_at / expired_at` 等时间语义，并把 provenance 作为第一等公民。

可借鉴点：
- 借鉴“事实条目必须有来源引用与时间语义，旧事实不能靠静默覆盖消失”的治理思想。
- 在 AI Team 中不直接上 temporal graph，也不引入 `valid_at/invalid_at` 全量机制；当前只保留最小必要的 `runtime_ref`、`created_at/updated_at`、`last_used_at`、`review` 与软删除边界。

复用判断：语义层借鉴，不直接采用。

## 2.3 Letta Memory Blocks（官方文档）
观察点：
- Letta 的核心抽象是“始终在上下文中的 memory blocks”，适合 agent prompt memory，而不是企业后台逐条治理的记忆清单。

不采用原因：
- B07 的目标是“员工记忆条目管理 + 搜索 + 审计 + 手工修正 + 注入痕迹”，不是“固定 prompt block 管理”。
- 如果直接借用 memory block 抽象，会把治理对象和运行时 prompt 结构混在一起，破坏 Team Panel / Runtime 边界。

复用判断：明确不采用。

## 2.4 最终采用策略
- 直接采用：无。
- 选择性复用：无外部 schema 直接复用。
- 接口层借鉴：借鉴 Mem0 的异步提取 + provenance 思路。
- 语义层借鉴：借鉴 Zep 对事实 provenance/时间性的重视。
- 明确不采用：Letta memory blocks 作为 B07 的主治理对象。

# 3. 共享对象边界

## 3.1 MemoryItem 是什么
`MemoryItem` 是 Team Panel 业务侧“员工记忆条目”。它用于：
- 后台列表/搜索/筛选/编辑
- 重要度治理
- 自动提取结果审计
- 下次运行时的可解释注入
- 审核与纠错

它不是：
- Hermes provider 原始 memory store 文件的一比一镜像
- prompt 中永远固定存在的 memory block
- `run_event` 的直接替代物

## 3.2 Companion object
- `MemoryReviewDecision`：审核轨迹对象，记录谁在何时对某条记忆做了什么裁决。
- `prompt_use_trace`：读侧派生对象，不单独落主表；由 `run_event(memory_written)` 聚合得到。

## 3.3 上下游边界
上游：
- 手工创建/编辑（B07 管理页）
- 自动提取（会话/任务运行结束后的抽取流程）
- 系统策略写入（少量系统生成记忆）

下游：
- Team Panel 北向 `GET/POST/PATCH /api/team/memories*`
- Gateway/Runtime 检索注入
- 审计与 trace 查询

# 4. Canonical domain contract

## 4.1 MemoryItem 共享字段语义

```json
{
  "memory_id": "mem_001",
  "employee_id": "emp_marketing_001",
  "content": "客户偏好每周一上午收到简明周报",
  "category": "preference",
  "importance": 5,
  "source_type": "extraction",
  "tags": ["vip", "reporting"],
  "visibility_scope": "enterprise",
  "runtime_ref": {
    "source_kind": "run_event",
    "run_id": "run_001",
    "event_id": "evt_101",
    "event_cursor": 101,
    "session_id": "sess_001",
    "source_type": "session",
    "source_id": "sess_001"
  },
  "review": {
    "status": "pending",
    "latest_decision_id": "mrd_001",
    "reviewed_at": null,
    "reviewed_by": null,
    "comment": null,
    "corrected_content": null
  },
  "prompt_use_trace": [
    {
      "run_id": "run_009",
      "event_id": "evt_901",
      "event_cursor": 901,
      "stage": "prompt_injected",
      "used_at": "2026-06-02T09:30:00Z"
    }
  ],
  "last_used_at": "2026-06-02T09:30:00Z",
  "created_at": "2026-06-02T08:15:00Z",
  "updated_at": "2026-06-02T08:40:00Z"
}
```

### 字段执行语义
- `memory_id`
  - 记忆条目主键。
  - northbound / persistence / trace / audit 全链路统一使用该 id。

- `employee_id`
  - 该条记忆归属的员工。
  - 记忆检索、治理权限、注入策略都以此为主归属边界。

- `content`
  - 最终可被检索与注入的自然语言记忆正文。
  - 若审核结果为 `corrected`，运行侧使用修正后的最终正文；不要保留“显示 corrected_content 但实际检索原 content”的双真相。

- `category`
  - 唯一枚举：`preference | habit | decision | event`
  - 语义：
    - `preference`：用户/客户/团队偏好
    - `habit`：稳定行为习惯或重复操作习惯
    - `decision`：已作出的决策、约束、结论
    - `event`：重要事实或已发生事件

- `importance`
  - 唯一范围：`1..5` 的整数。
  - 语义：`1` 最低、`5` 最高。
  - 这是治理/排序/筛选口径，不等于底层检索模型分数。
  - 如果运行侧需要归一化权重，应在检索注入阶段派生，不得把 `0..1` 或 `0..10` 反写为共享 contract。

- `source_type`
  - 唯一枚举：`manual | extraction | system_policy`
  - 语义：
    - `manual`：后台人工新增或人工编辑后保存
    - `extraction`：从运行内容自动提取进入记忆池
    - `system_policy`：系统策略或模板下发形成的长期规则

- `tags`
  - 自由标签数组；用于 B07 筛选与次级治理。
  - 不替代 `category`。

- `visibility_scope`
  - 唯一枚举：`enterprise | admin_only`
  - 语义：
    - `enterprise`：企业后台授权用户可见
    - `admin_only`：仅管理员可见，用于敏感偏好/敏感决策

- `runtime_ref`
  - 统一为 JSON 对象。
  - 作用：表达该记忆来自哪个运行证据，或与哪个运行上下文绑定。
  - 允许为空对象 `{}`（典型于手工创建）。
  - `runtime_ref` 统一吸收此前分裂的 `extraction_run_id` / `runtime_ref_json` 思路。

- `review`
  - 这是 northbound 聚合视图，不要求一整块平铺到 `memory_item` 主表。
  - `status` 语义：
    - `not_required`：该条记忆不需要专门审核（典型于 `manual` / `system_policy`）
    - `pending`：自动提取后等待人工确认或后续治理
    - `confirmed`：审核通过，可继续用于检索注入
    - `corrected`：审核通过但内容被修正；运行侧应使用修正后的最终内容
    - `rejected`：审核否决；该条不得再被检索注入

- `prompt_use_trace`
  - 读侧派生数组，不落 `memory_item` 主表。
  - 用来回答“这条记忆何时被注入过、在哪个 run 被用过”。

- `last_used_at`
  - 最近一次成功进入 prompt 注入链路的时间。
  - 从 `prompt_use_trace` 聚合维护；不要求人工写入。

## 4.2 review 决策对象

`MemoryReviewDecision` 是审核历史，不是第二个 memory item。

建议共享 shape：
```json
{
  "decision_id": "mrd_001",
  "memory_id": "mem_001",
  "decision": "corrected",
  "reviewed_by": "usr_admin_001",
  "comment": "保留周报偏好，但去掉未经确认的时区描述",
  "corrected_content": "客户偏好每周一上午收到简明周报",
  "created_at": "2026-06-02T08:40:00Z"
}
```

唯一枚举：`pending | confirmed | rejected | corrected`

执行语义：
- `pending`：通常由 extraction 产生的初始治理状态
- `confirmed`：内容不改，继续生效
- `rejected`：不再参与检索注入
- `corrected`：内容被人修正后继续生效；最终正文以修正后内容为准

# 5. Persistence 口径

## 5.1 memory_item 主表 canonical columns

`memory_item` 的共享 persistence 口径固定如下：

- `id` PK
- `enterprise_id` FK
- `employee_id` FK
- `content` text not null
- `category` enum(`preference`,`habit`,`decision`,`event`)
- `importance` int check between 1 and 5
- `source_type` enum(`manual`,`extraction`,`system_policy`)
- `tags_json` jsonb default `[]`
- `visibility_scope` enum(`enterprise`,`admin_only`) default `enterprise`
- `runtime_ref_json` jsonb default `{}`
- `last_used_at` timestamptz nullable
- `created_at`
- `updated_at`
- `created_by` nullable
- `updated_by` nullable
- `deleted_at` nullable

说明：
- `retention_expires_at` 不作为 Batch2 canonical 必需列。若实现层为了清理任务方便保留该列，可视为内部派生列，但不得把它重新抬升为北向必需 contract 字段。
- `binding_version` 不属于 `memory_item` 本体共享字段；那是员工记忆策略绑定的版本信息，应继续归属 `employee_memory_binding`，而不是散落到记忆条目上。
- 不再保留单独 `extraction_run_id` 为主来源字段；来源统一并入 `runtime_ref_json`。

## 5.2 memory_review_decision companion table canonical columns

- `id` PK
- `enterprise_id` FK
- `memory_item_id` FK -> `memory_item.id`
- `reviewer_user_id`
- `decision` enum(`pending`,`confirmed`,`rejected`,`corrected`)
- `comment` nullable
- `corrected_content` nullable
- `created_at`
- `updated_at` 可选
- `created_by/updated_by/deleted_at` 按现有审计规范处理

说明：
- 该表保存审核历史；northbound `review` 只投影最新有效决策。
- 不要求把 `review_status` 再额外冗余到 `memory_item`，除非实现层为了查询优化做缓存列；若做缓存列，也必须与本表 latest decision 语义完全一致，不能演化出第二套状态机。

## 5.3 索引建议

最小必要索引：
- `idx_memory_item_enterprise_employee_created(enterprise_id, employee_id, created_at desc)`
- `idx_memory_item_enterprise_employee_importance(enterprise_id, employee_id, importance desc, updated_at desc)`
- `idx_memory_item_tags on gin(tags_json)`
- `idx_memory_review_decision_item(memory_item_id, created_at desc)`

# 6. 提取、搜索、注入 trace 语义

## 6.1 自动提取成功路径

主链路：
1. 运行结束或达到提取边界
2. Hermes / Gateway 触发抽取流程
3. 产出候选记忆
4. Team Panel 创建 `memory_item(source_type=extraction)`
5. 若需要治理，可同时创建最新 `memory_review_decision(decision=pending)`
6. 后续人工可 `confirmed / corrected / rejected`

关键裁决：
- 只有真正接受进入记忆池的条目才创建 `memory_item`
- 自动提取失败只落审计事件，不落半条 `memory_item`

## 6.2 提取失败路径

唯一正式语义：
- 失败记录在 `audit_event(event_type='memory.extraction_failed')` 或等价审计流
- payload 至少包含：`employee_id`、`run_id`、`error_code`、`error_message`
- 不创建缺字段/占位 `memory_item`

## 6.3 prompt_use_trace 语义

`prompt_use_trace` 由 `run_event(event_type='memory_written')` 派生。

最小派生条件：
- `payload_json.memory_id = <memory_id>`
  或
- `payload_json.used_memory_ids[]` 包含 `<memory_id>`

最小 trace item：
- `run_id`
- `event_id`
- `event_cursor`
- `stage`（如 `prompt_injected`）
- `used_at`

关键裁决：
- `prompt_use_trace` 是读模型，不是主表字段
- `last_used_at` 允许作为聚合后的便捷列/响应字段

## 6.4 搜索与可用性语义

默认搜索可见集合：
- `deleted_at is null`
- `review.status != rejected`
- 满足权限与 `visibility_scope`

默认排序：
- `importance desc, updated_at desc`

注意：
- `pending` 不是自动屏蔽条件；它表示“尚未人工定稿”，不是“不可用”
- 真正不可注入的是 `rejected`

## 6.5 Canonical repository / service surface

本轮真正需要收口的不是单个字段，而是 data-rd 与 backend 对 repo/query/service 分层的边界。

### repository 层唯一正式口径

`MemoryItemRepo`：
- `create(item)`
- `get_by_id(memory_id, *, include_deleted=False)`
- `list_by_enterprise(enterprise_id, *, employee_id=None, search_query=None, tag=None, category=None, source_type=None, review_status=None, visibility_scope=None, limit=100, offset=0, sort_by='importance', sort_order='desc')`
- `count_by_enterprise(enterprise_id, *, employee_id=None, search_query=None, tag=None, category=None, source_type=None, review_status=None, visibility_scope=None)`
- `update(item)`
- `delete(memory_id)`
- `bulk_delete(memory_ids, *, enterprise_id, employee_id=None)`

执行语义：
- `delete / bulk_delete` 都是 soft delete，更新 `deleted_at`；不要做 hard delete。
- 仓储层正式查询入口是 `list_by_enterprise`，因为 B07 后台页面与 northbound API 的授权边界首先是 enterprise。
- `employee_id` 是 enterprise 下的从属过滤条件，而不是另一套平行 northbound contract。
- 若实现层为了内部便利保留 `list_by_employee(...)`，只能作为 wrapper/helper，内部委托到 `list_by_enterprise(...)`；正式测试与文档不再把它当 canonical surface。
- 若实现层需要“某员工当前有效记忆条目数”来做策略检查，可额外提供 `count_by_employee(employee_id)` 之类 helper，但它属于 policy helper，不得再绑定共享常量 `_MAX_MEMORIES_PER_EMPLOYEE=5` 作为正式成功条件。

`MemoryReviewDecisionRepo`：
- `create(decision)`
- `get_latest_by_memory_id(memory_id)`
- `latest_by_memory_ids(memory_ids)`

执行语义：
- 该 repo 的共享职责只有“写历史 + 取最新有效裁决”。
- `get_by_memory_item(...)` 命名不再是正式口径，因为它既没有表达 latest 语义，也把业务名 `memory_id` 和表列名 `memory_item_id` 混在了一起。

### query / command service 层唯一正式口径

`MemoryItemQueryService.list_memories(...)` 负责：
- 把 northbound `page/page_size` 翻译成 repo `limit/offset`
- 组装 `items/page/page_size/total/has_more/sort_by/sort_order`
- 聚合 `review`
- 在 `include=prompt_use_trace` 时追加 trace 读模型

`MemoryItemCommandService` 负责：
- `create_manual_memory(...)`
- `patch_memory(...)`
- `delete_memory(...)` / `bulk_delete_memories(...)`（内部命令能力，可供后续 API 或批量治理任务复用）
- 写 `audit_event(memory.create|memory.update|memory.delete|memory.bulk_delete)`
- 如有员工级 retention / 容量策略，只在这里读取 `employee_memory_binding` 或其他策略来源做校验，不把策略硬编码进 repo/domain 常量

明确边界：
- 北向 API 仍然只有 `GET/POST/PATCH /api/team/memories*` 是本轮正式 contract。
- `delete/bulk_delete` 在本轮属于内部命令与 repo 能力，先为后续治理/批量操作收口，不额外冻结新的 public DELETE 路由。

# 7. Northbound API 口径

## 7.1 GET /api/team/memories

用途：列表 + 搜索 + 筛选 + 可选 trace 展开。

建议 query：
- `employee_id`
- `q`
- `category`
- `tag`
- `source_type`
- `review_status`
- `page`
- `page_size`
- `sort_by`
- `sort_order`
- `include=prompt_use_trace`（可选）
- `trace_limit`（可选）

响应 envelope 统一跟随全局分页口径：
```json
{
  "items": [],
  "page": 1,
  "page_size": 20,
  "total": 0,
  "has_more": false,
  "sort_by": "importance",
  "sort_order": "desc"
}
```

单项最小返回字段：
- `memory_id`
- `employee_id`
- `content`
- `category`
- `importance`
- `source_type`
- `tags`
- `visibility_scope`
- `runtime_ref`
- `review`
- `last_used_at`
- `created_at`
- `updated_at`

可选展开字段：
- `prompt_use_trace`

## 7.2 POST /api/team/memories

该接口的 canonical 语义是“人工新增记忆”。

请求 allowlist：
```json
{
  "employee_id": "emp_marketing_001",
  "content": "客户偏好每周一上午收到简明周报",
  "category": "preference",
  "importance": 5,
  "tags": ["vip", "reporting"],
  "visibility_scope": "enterprise"
}
```

执行语义：
- `source_type` 固定写为 `manual`
- `runtime_ref` 默认 `{}`
- `review.status` 默认 `not_required`
- 成功后写 `audit_event(memory.create)`

## 7.3 PATCH /api/team/memories/{memory_id}

该接口同时承担“内容修正”和“审核裁决”，但必须通过 allowlist 收紧。

请求 allowlist：
```json
{
  "content": "客户偏好每周一上午收到简明周报",
  "category": "preference",
  "importance": 4,
  "tags": ["vip", "reporting"],
  "visibility_scope": "admin_only",
  "review": {
    "decision": "corrected",
    "comment": "去掉未经确认的发送频率",
    "corrected_content": "客户偏好收到简明周报"
  }
}
```

执行语义：
- 普通字段变更：更新 `memory_item`
- `review.decision=confirmed/rejected/corrected`：新增一条 `memory_review_decision`
- 当 `decision=corrected` 且 `corrected_content` 非空时：最终 `memory_item.content` 更新为修正后内容
- 成功后写 `audit_event(memory.update)` 或更细分的 `memory.review_*`

## 7.4 DELETE 语义

Batch2 当前北向首批接口没有单独冻结 DELETE 路由，因此本次不新增 northbound DELETE contract。

唯一裁决：
- soft delete 仍然存在，但属于实现/后续扩展能力
- 当前 shared contract 不要求 reviewer 以 `/api/team/memories DELETE` 作为 pass 条件
- 如果后续需要对外开放删除，再单独补 API contract；不要在本轮 backend/data 各自自由发挥

# 9. Migration numbering / merge guidance

当前主干已存在：
- `001_initial_schema.sql`
- `002_conversation_message.sql`
- `003_industry_solution_bindings.sql`

因此 B07 绝不能再保留任何 `002_memory_item*.sql` 进入主干。

唯一 merge 指导：
1. 删除或重命名两条分支上的本地迁移：
   - `002_memory_item.sql`
   - `002_memory_item_schema.sql`
2. 合并为一条新的主干迁移：
   - 唯一正式命名：`004_memory_item.sql`
3. 该迁移只保留一份 canonical DDL：
   - `memory_item`
   - `memory_review_decision`
   - 必要索引
   - 对已存在开发库的兼容性 `ALTER TABLE ... ADD COLUMN IF NOT EXISTS`
4. 不允许两条 memory 迁移并存并都自称正式 002；这会导致：
   - 迁移顺序漂移
   - CI / 本地数据库状态不一致
   - reviewer 无法确认最终 schema

补充裁决：
- migration 编号以主干现状递增，不以子分支“各自第一次实现”编号为准。
- 本轮 merge 后只应存在一个权威 memory migration 文件。
- 旧测试若需要校验 migration 已应用，统一断言 `is_applied(db_conn, '004_memory_item.sql')`；不要再断言 `004_memory_item_contract.sql`，也不要用“两个名字任选其一”的宽松写法掩盖漂移。
- 旧测试若需要从 migrations 目录查找 memory migration，必须只匹配到 `004_memory_item.sql` 这一条；`004_memory_item_contract.sql` 应视为应清理的历史分支产物，而不是兼容选项。

# 12. 风险、边界与验证方法

## 12.1 风险
- 把 `importance` 从 0..10 收到 1..5，会影响已写测试与 seed 数据，需要统一修订。
- 若 backend 继续保留 `auto`，前后端/数据层会再次产生并行 source_type 口径。
- 若 `prompt_use_trace` 被错误持久化进主表，会把运行时事件视图和业务主对象耦死。
- 若 `memory_review_decision` 被去掉，后续 reviewer 将无法追溯“谁把哪条自动提取记忆改掉/拒绝掉”。

## 12.2 边界
- 本定稿版不要求本轮新增单独的 DELETE 北向接口。
- 本定稿版不要求引入更重的 temporal graph / fact invalidation 引擎。
- 本定稿版不要求把 Hermes provider 原始 memory store 完全结构化复制到 Team Panel。

## 12.3 如何验证本裁决有效
满足以下条件即可认为本次 contract freeze 生效：
1. 主干只存在一条权威 memory migration：`app/team_panel/migrations/004_memory_item.sql`
2. data-rd 与 backend 的测试都以 `importance=1..5`、`source_type=manual|extraction|system_policy`、`runtime_ref_json`、`review` 聚合语义为准
3. repo/query 层正式口径收敛到 `list_by_enterprise(...)`、`delete/bulk_delete`、`get_latest_by_memory_id/latest_by_memory_ids`
4. API contract tests 与 repo tests 对同一条 memory item 使用同一组字段/枚举
5. reviewer 复核时，不会再指出“memory persistence 与 API slice 各自定义了不同的 memory_item”

这就是 Batch2 B07 的唯一共享裁决口径。
