# AI Team S01 系统账号管理契约收口

## 1. 文档定位

本文只解决一个问题：Batch4 的 S01 system-accounts 北向契约已经出现真实漂移。正式前端/规格口径采用 `POST /api/system-admin/enterprises/{id}/actions`；而触发本次收口的历史 backend worktree 曾尝试把写接口拆成 `/ban`、`/recharge`、`/notify` 等 split 路径。当前共享仓库主线实现则仍停留在 `router_system_admin.py` 仅保留 `/health`、其余路径统一返回 `501 not_implemented` 的占位阶段。

本收口文档给出唯一裁决口径，目的是让 frontend、backend、reviewer、QA 在同一份文档上收敛，不再靠任务评论或各自测试猜接口。

若现有详细设计、实现分支、测试样例与本文冲突，S01 一律以本文为准。

## 2. 方案调研与复用判断

### 2.1 已核查的成熟模式

1. Microsoft Graph 官方 API
- `POST /users/{id}/sendMail`
- `POST /users/{id}/assignLicense`
- 这类接口不是把“发信”“分配许可”伪装成 CRUD，而是把它们作为用户资源上的明确动作处理。
- 借鉴点：当动作是命令型、异质型、非稳定子资源时，成熟系统允许使用资源下动作端点，而不是强行拆成一批伪资源。

2. Stripe 官方 API
- `POST /v1/payment_intents/{id}/capture`
- `POST /v1/payment_intents/{id}/confirm`
- `POST /v1/payment_intents/{id}/increment_authorization`
- 借鉴点：当对象已经存在，而系统需要触发一组副作用明确、状态转换明确的命令时，成熟系统常用“对象 + 动作”风格；动作本身可以有各自参数，但不会把每个动作再抬升成新的顶层资源。

3. Keycloak Admin REST API
- `PUT /admin/realms/{realm}/users/{user-id}/execute-actions-email`
- 借鉴点：后台治理/运营动作经常天然带有“对某个已存在主体执行一个管理动作”的语义，动作端点是成熟做法，并不要求把每个管理动作都拆成完全独立的 northbound 页面契约。

### 2.2 复用判断

| 参考来源 | 采用方式 | 结论 |
| --- | --- | --- |
| Microsoft Graph 动作接口 | 接口层借鉴 | 采用“资源下命令动作”的 northbound 风格 |
| Stripe 动作接口 | 接口层借鉴 | 采用“已有对象 + 明确命令 + 动作参数”的表达方式 |
| Keycloak Admin 动作接口 | 接口层借鉴 | 采用“后台治理动作集中表达”的思路 |
| 历史 backend worktree 的 split `/ban|/recharge|/notify` 尝试 | 明确不作为正式 northbound 口径 | 若后续某个 backend 分支短期保留这些路径，也只能作为兼容别名，不能再被正式文档采纳 |

### 2.3 最终判断

S01 不是通用 CRUD 页面，而是系统后台对企业主体执行治理命令的页面。`ban / unban / recharge / notify` 都属于“对已存在 Enterprise 执行动作”，不是独立业务资源。因此 northbound 采用统一动作入口比拆成多个页面级写路径更适合当前 Phase 2：

- 对前端更简单：只有一个写入口，不必为每个按钮维护一套 URL 选择逻辑
- 对文档更稳定：动作枚举集中维护，避免新增动作时再复制一条新路径
- 对 backend 可维护性更高：外部契约统一，内部仍可按 action 分发到不同 handler
- 对回滚更安全：只要保住 `/actions` 入口，内部实现可在统一入口后自由调整

结论类型：接口层借鉴，不直接照搬外部 schema。

## 3. 当前漂移点

### 3.1 frontend / spec 口径
当前正式接口文档已经写成：
- `GET /api/system-admin/enterprises`
- `POST /api/system-admin/enterprises/{id}/actions`

这也是本次正式 northbound 收口要继续保持的唯一写入口口径。

### 3.2 触发本次收口的历史 backend worktree 漂移
历史 backend worktree `t_c9ef67dc` 曾把 S01 写接口实现为：
- `POST /api/system-admin/enterprises/{id}/ban`
- `POST /api/system-admin/enterprises/{id}/recharge`
- `POST /api/system-admin/enterprises/{id}/notify`
- 另有 `GET/POST /api/system-admin/enterprises/{id}/quota`

这导致：
- 前后端 contract 不一致
- reviewer 无法把“实现通过测试”直接视为“系统收口”
- QA 无法判断哪一组 URL 才是正式对外口径

### 3.3 当前共享仓库主线实现状态
截至当前仓库状态，`app/team_panel/api_team/router_system_admin.py` 仅实现：
- `GET /api/system-admin/health`

其余 `system-admin` 路径统一返回：
- `501 not_implemented`

因此当前共享仓库主线并不存在正式 `/actions` 实现，也不存在仍在运行的 split `/ban|/recharge|/notify` 实现；本文件的作用是先把权威 northbound 契约收口，再为后续 backend 落地提供唯一实现目标。

## 4. 裁决结论

### 4.1 唯一正式 northbound path shape

S01 企业治理动作的唯一正式 northbound 写入口固定为：

`POST /api/system-admin/enterprises/{enterprise_id}/actions`

这是唯一对前端、接口文档、集成测试、QA 验收生效的正式写路径。

### 4.2 保留的读取/辅助路径

以下路径继续保留，不属于本次漂移裁决争议面：
- `GET /api/system-admin/enterprises`
- `GET /api/system-admin/enterprises/{enterprise_id}`
- `GET /api/system-admin/enterprises/export`
- `GET /api/system-admin/enterprises/{enterprise_id}/quota`
- `POST /api/system-admin/enterprises/{enterprise_id}/quota`

解释：
- `quota` 更接近企业配置/配额读写面，不与本次“命令动作入口”合并
- 本次裁决只统一 ban/recharge/notify/unban 这类命令型动作的写入口

### 4.3 EnterpriseAdminAction 请求体口径

正式请求体如下：

```json
{
  "action": "ban",
  "reason": "多次违规调用",
  "amount": null,
  "message": null,
  "level": null,
  "idempotency_key": "sysacct-ent_001-ban-20260603-01"
}
```

字段执行语义：

- `action`
  - 唯一正式枚举：`ban | unban | recharge | notify`
  - 语义：
    - `ban`：封禁企业，阻断其后续受控访问
    - `unban`：解除封禁，恢复企业正常状态
    - `recharge`：记录一次人工充值/余额调整动作
    - `notify`：向企业管理侧投递一条系统通知（Phase 2 允许 mock/outbox，不要求真实外发）

- `reason`
  - `ban` / `unban` 时建议提供，作为审计理由
  - 其他 action 可为空

- `amount`
  - 仅 `recharge` 使用
  - 单位统一为最小货币单位整数（如分/cents）
  - 非 `recharge` 时必须为空或省略

- `message`
  - 仅 `notify` 使用
  - 表示通知正文

- `level`
  - 仅 `notify` 使用
  - 正式枚举：`info | warning | critical`
  - 省略时默认 `info`

- `idempotency_key`
  - northbound 允许携带
  - `recharge` 与 `notify` 场景建议传入，便于安全重试
  - `ban` / `unban` 可以不强制，但若调用方具备重试器，仍建议传入

### 4.4 action 与字段约束矩阵

| action | 必填字段 | 禁止依赖字段 | 成功语义 |
| --- | --- | --- | --- |
| ban | `action` | `amount`, `message`, `level` | 企业状态切到封禁态，并留下审计记录 |
| unban | `action` | `amount`, `message`, `level` | 企业状态解除封禁，并留下审计记录 |
| recharge | `action`, `amount` | `message`, `level` | 记录一次人工充值/余额调整，并留下审计记录 |
| notify | `action`, `message` | `amount` | 记录一次系统通知投递（mock/outbox 可接受），并留下审计记录 |

### 4.5 成功响应最小口径

为避免 frontend/backend 再次各写各的，成功响应至少应满足下面的最小共享 shape：

```json
{
  "enterprise_id": "ent_001",
  "action": "ban",
  "status": "succeeded",
  "message": "enterprise action applied",
  "audit_event_id": "audit_001"
}
```

约束：
- `enterprise_id`、`action`、`status`、`message` 为所有 action 的共同最小字段
- `status` 在 Phase 2 固定返回 `succeeded` 或等价失败 HTTP 状态，不再为不同 action 发明不同 top-level 状态词
- action-specific 额外字段可以追加在同一响应内，但不能替代这四个共同字段

推荐追加字段：
- `ban` / `unban`：`enterprise_status`
- `recharge`：`amount`, `balance_after`（若当期可得）
- `notify`：`notification_id`, `level`

## 5. 兼容与回滚规则

### 5.1 兼容规则

若后续 backend 分支在实现 `/actions` 的过渡窗口内需要兼容旧调用，可以临时保留以下 legacy alias：
- `POST /api/system-admin/enterprises/{id}/ban`
- `POST /api/system-admin/enterprises/{id}/recharge`
- `POST /api/system-admin/enterprises/{id}/notify`

但这些 alias 只允许作为兼容层存在，必须满足：
1. 形式上不再出现在正式设计文档、前端代码、正式 contract tests、QA 验收口径中
2. 语义上全部转发/归并到同一个 `actions` 处理器
3. reviewer/QA 只以 `/actions` 是否可用作为 S01 收口判断，不以 alias 是否继续暴露作为通过标准

补充说明：当前共享仓库主线尚未实现这些 alias；本节只定义“如果某个过渡 backend 分支暂时保留 alias，应遵守什么边界”，不表示 alias 已是主线现状。

`/unban` 不单独新增 split alias；若需要旧式兼容，也应由 `/actions` 统一承接，而不是再扩一路 `/unban`。

### 5.2 回滚规则

若 Batch4 后续实现发现统一 action 分发器存在缺陷，允许的回滚方式只有：
- backend 在内部回退到旧 handler 逻辑
- `/actions` 入口继续保留并继续作为唯一正式 northbound 契约

不允许的回滚方式：
- 把正式文档重新改回 split `/ban|/recharge|/notify`
- 让 frontend 改去调用 split 路径规避问题
- 在 Phase 2 再并行维护两套都宣称“正式”的写契约

换言之：内部实现可回退，外部 northbound 契约不回退。

## 7. 设计权衡

### 7.1 为什么不选 split `/ban|/recharge|/notify` 作为正式口径

优点：
- URL 自解释
- backend 处理器天然分开

缺点：
- 已经与 frontend/spec 实际口径冲突
- 每新增一个动作都要新增一路 northbound path、测试、文档和权限矩阵项
- 对 S01 这种“后台治理动作页”来说，动作数量会自然增长，split path 容易继续漂移

结论：不采用为正式口径。

### 7.2 为什么选统一 `/actions`

优点：
- 交付速度更快：最少改动即可让 frontend/spec/backend 收敛
- 边界更清楚：northbound 只暴露一个命令入口，内部 handler 如何拆分属于 backend 自由
- 扩展更稳：后续若新增 `freeze_billing`、`reset_quota` 一类动作，不必先扩一路新 URL

风险：
- action body 需要明确枚举和字段约束，否则会退化成“万能 RPC 桶”

控制方式：
- 把 action 限定为最小枚举 `ban | unban | recharge | notify`
- 给每个 action 明确字段矩阵
- 将 `quota` 继续作为独立配置面，不往 `actions` 里继续混入更多形态
