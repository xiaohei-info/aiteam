# AI Team Team Panel 与 Agent Gateway 详细设计方案

## 1. 文档目标

本文是 AI Team 从“业务解决方案 / 技术概要设计”进入“详细设计”阶段后的第一份实现级主文档，目标只有三个：

1. **按功能域列出当前 MVP/正式版方案需要实现的全部功能点**。
2. **把 Team Panel 的业务语义讲清楚**：对象是什么、关系是什么、流程如何流转、核心数据结构长什么样。
3. **把 Team Panel 业务对象映射到 Hermes Web UI / Hermes Agent 的真实落地点**：
   - 前端落到哪个既有页面或需新增页面
   - 后端落到哪个既有接口或需新增接口
   - Runtime/SDK 落到哪个 Python 入口、状态文件、数据库或调度机制

本文不展开 LightRAG、AI Relay、支付、向量库等外部底座的内部实现；本文重点是：

- **Team Panel 业务控制层**
- **Agent Gateway 运行时适配层**
- **AI Team 业务对象如何转成 Hermes 可执行对象**

## 2. 设计边界与复用判断

### 2.1 范围内

- 企业/用户/空间
- 人才市场 / 行业方案 / 员工实例化
- 数字员工配置
- 私聊 / 群聊 / 编排 / Loop
- 知识 / 记忆 / Skills / 连接器绑定
- 办公室动态 / 组织架构
- 成本 / 统计 / 审计 / 后台治理
- Team Panel -> Agent Gateway -> Hermes Runtime 的对象映射

### 2.2 范围外

- LightRAG 内部切块、向量化算法实现
- AI Relay 充值支付链路细节
- 第三方 OAuth provider 逐家接入细节
- 移动端单独详细设计

### 2.3 复用判断结论

#### 直接复用 Hermes Runtime
- `run_agent.AIAgent.run_conversation()`：单次任务执行主入口
- `hermes_cli.kanban_db.Task / create_task() / dispatch_once()`：多 Agent 编排任务主入口
- `cron.jobs / cron.scheduler`：定时任务主入口
- `gateway/platforms/base.py:handle_message()`：真实消息入口统一处理链
- `gateway/session.py:build_session_key()`：私聊/群聊/线程会话键规则
- `gateway/platforms/telegram.py`：Telegram 群聊 / topic / mention 触发参考实现
- `tools/memory_tool.py`、`agent/memory_manager.py`：记忆底层
- `skills/`、`skill_manager_tool.py`：Skill 底层
- `hermes_state.py`、`state.db`、`sessions/`：会话/消息/状态持久化

#### 选择性复用 Agent Service 上游基座（Hermes Web UI）
- `server.py`：HTTP 路由壳、CSP、安全头与请求入口宿主
- `api/routes.py`：现有路由组织方式；V1 以“不做减法只做加法”为原则扩展 `/api/team/*`
- `api/streaming.py`：SSE 流式引擎、Agent 线程调度、取消信号与 env 隔离
- `api/config.py`：Agent 目录发现、Provider 解析、运行态全局注册表
- `api/models.py`：Session / Project JSON 持久化模式与原子写入能力
- `api/profiles.py`：Profile 路径解析与隔离逻辑（去 Cookie 依赖后复用）
- `api/workspace.py` / `api/upload.py` / `api/helpers.py`：文件能力、上传能力与 HTTP 工具函数
- `static/index.html` / `static/ui.js` / `static/messages.js` / `static/sessions.js` / `static/panels.js`：三栏骨架、Markdown 渲染、SSE 客户端、Sidebar / Panel 框架
- `api/auth.py`：仅可借鉴单密码登录壳，不可直接当企业身份体系

#### AI Team 必须自建
- Team / Employee / Template / Solution / TeamRun / TeamTask / CapabilityBinding 等业务对象
- 企业/系统后台治理对象
- 多 Agent 团队语义、任务树、协同气泡、业务状态解释层
- 多租户身份、成员关系、权限与业务审计
- 将业务对象翻译为 Hermes Profile / Session / Task / Cron / Memory / Skill 的适配层

#### 明确不直接采用
- 不把整个 Hermes Web UI 直接等同于 Agent Gateway
- 不让浏览器页面语义直接成为 Runtime contract
- 不让 Team Panel 直接依赖 `GatewayRunner._handle_message()` 之类私有消息入口作为北向 API

## 3. 详细设计总原则

### 3.1 统一口径分层

- **Team Panel** 持有业务控制面口径：企业、员工、模板、绑定、任务、费用、运营语义
- **Agent Gateway** 持有运行接入口径：请求句柄、会话键、执行提交、事件回流
- **Hermes Runtime** 持有执行事实口径：Profile、Session、Message、Task、Run、Cron、Memory、Skill

### 3.2 统一映射方向

所有业务对象都遵循同一条映射链：

`前端动作 -> Team Panel 业务对象 -> Gateway 可执行对象 -> Hermes Runtime 入口 -> 事件/状态回流 -> Team Panel 展示对象`

### 3.3 落点标记约定

- **[W-Existing]**：Hermes Web UI 已有前后端能力，可直接复用
- **[H-Existing]**：Hermes Runtime 已有能力，可直接调用
- **[T-New]**：AI Team 必须新增的业务层/接口/页面/适配逻辑
- **[T-Extend]**：已有基座可复用，但必须二开扩展

### 3.4 运行时结构骨架

```text
浏览器 / 用户
    |
    v
Team Panel Web
(页面、会话入口、SSE 订阅、可视化)
    |
    v
Team Panel API
(企业、员工、群聊、任务、计费、审计)
    |
    v
Agent Gateway Adapter
(业务对象翻译、Profile 供应、运行提交、事件回流、凭据解析)
    |
    +--> Team Panel DB / 审计库 / 账本
    |
    +--> Hermes Web UI 可复用宿主骨架
    |    (单会话 UI、SSE、上传、workspace 壳)
    |
    +--> Hermes Runtime
    |    |- Profile
    |    |- Session / state.db / sessions/*
    |    |- AIAgent.run_conversation()
    |    |- Kanban task graph / dispatcher
    |    |- Cron jobs / scheduler
    |    |- Memory / Skills / Tools / MCP
    |
    +--> External Capability
         |- LightRAG / 知识引擎
         |- OAuth / API Key / MCP 连接器
         |- Telegram / Discord / 邮件等消息平台
```

- Team Panel Web / API 负责产品对象与治理语义，不直接掌握 Hermes 内部状态机。
- Agent Gateway Adapter 是唯一允许同时理解“业务对象”和“Runtime 对象”的边界层。
- Hermes Web UI 在这里是**上游实现来源**；真正落地实施的代码载体是 `Agent Service` 项目。

### 3.5 目标实施形态（Agent Service）

AI Team 的正式落地项目不是继续直接在 upstream `hermes-webui` 仓库内堆业务代码，而是：

- 目标仓库：`/home/ubuntu/code/aiteam/app`
- 技术形态：沿用 **Python stdlib `http.server` + 原生 JavaScript 模块**
- 进程形态：V1 由同一个 `server.py` 进程承载 Team Panel HTTP 宿主壳与 Agent Gateway 进程内适配层
- 演进原则：保留现有 `api/*`、`static/*` 的高复用骨架，在此基础上做 **Team Panel 业务层新增 + Agent Gateway 提取与规格化**

建议的项目级落点口径如下：

```text
app/
├── team-panel/
│   ├── api_team/      # /api/team/*、/api/enterprise-admin/*、/api/system-admin/*
│   ├── models/        # Enterprise / Employee / Conversation / TeamRun 等业务对象
│   └── pages/         # 企业前台 / 企业后台 / 系统后台页面实现
├── agent-gateway/
│   ├── adapters/      # RunRequest -> Runtime 调用翻译
│   └── profiles/      # Employee -> Profile 供应、凭据解析、运行句柄管理
├── api/               # 继承上游的 HTTP / SSE / workspace / upload / helper 基座
├── static/            # 继承上游的三栏 UI / Markdown / SSE 客户端 / panel 框架
└── server.py          # 同进程宿主入口
```

因此，本详细设计中提到的 Team Panel 与 Agent Gateway，是 **Agent Service 项目内部的两个核心模块层**，不是要求 V1 额外拆成两个独立部署服务。

### 3.6 二开实施原则（对扩展开放、对修改封闭）

本项目在落地实施时，默认采用 **Open for Extension, Closed for Modification** 的二开原则。这里的含义不是教条式“完全不改旧代码”，而是：

- **优先新增扩展层，不把 AI Team 新业务继续堆进已有巨型基座文件。**
- **对基座文件只做挂接性、小范围、可验证的修改。**
- **新业务接口走新命名空间，新模块负责；旧接口继续作为宿主能力与 fallback。**

#### 3.6.1 对 Team Panel 业务接口的约束

- `team-panel/api_team/*` 是 Team Panel 北向业务接口的主落点。
- `/api/team/*`、`/api/enterprise-admin/*`、`/api/system-admin/*` 优先在新文件中实现，不继续主要写入 `api/routes.py` 巨型 if/elif 主体。
- `server.py` 只承担**命名空间分流**：
  - 先尝试新业务 router
  - 未命中时再回退到原始 `api/routes.py`
- 原始 `api/routes.py` 中已有接口继续保留和使用，特别是健康检查、静态资源、workspace、上传、原生宿主接口与兼容层接口；它不是被替代，而是被降为**基座路由集合 + fallback 宿主**。

可接受的入口形态示意：

```python
def dispatch_get(handler, parsed):
    if parsed.path.startswith('/api/team/'):
        return team_panel_router.handle_get(handler, parsed)
    if parsed.path.startswith('/api/enterprise-admin/'):
        return enterprise_admin_router.handle_get(handler, parsed)
    if parsed.path.startswith('/api/system-admin/'):
        return system_admin_router.handle_get(handler, parsed)
    return legacy_routes.handle_get(handler, parsed)
```

#### 3.6.2 对 Agent Gateway 执行层的约束

- Agent Gateway 优先通过**轻量包装/适配器层**引用现有通用底层能力，不直接把业务语义写回 `api/streaming.py`、`api/config.py`、`api/profiles.py`。
- 推荐形态是：
  - `start_single_agent_run(run_request)` 包装单 Agent 执行
  - `start_group_run(collaboration_request)` 包装 orchestrator 根任务
  - `create_scheduled_job(job_request)` 包装 cron 提交
  - `sync_run_status(runtime_handle)` / `hydrate_runtime_events(...)` 包装状态回流与补拉
- 如果现有通用能力与 HTTP 路由逻辑混在一起，允许做一次**适度抽取**：先把可复用能力提成内部函数/模块，再由 Team Panel 和 Gateway 通过包装层调用；不要把同一份底层逻辑复制出第二份。

#### 3.6.3 明确禁止的实现方式

- 不复制一整份 `api/routes.py` / `api/streaming.py` 再做平行分叉。
- 不让同一路径既由新业务 router 处理、又在旧 router 中维持另一套不同语义。
- 不把 `/api/chat/start`、`/api/sessions` 这类宿主接口继续提升为正式产品契约。
- 不把 Team Panel 新业务主要沉积到 3000+ 行巨型基座文件中。

#### 3.6.4 开发验证标准

如果实施方向正确，后续开发应满足：

1. 新增一个 Team Panel 业务接口时，通常修改的是 `team-panel/api_team/*`，而不是继续扩写 `api/routes.py` 主体。
2. 旧 router 中 3000+ 行接口仍继续为宿主和兼容层服务，但不再承担主要新业务扩展。
3. Agent Gateway 新能力优先体现在 adapter / wrapper 文件中，而不是把业务分支直接塞回 streaming 基座。
4. 上游 WebUI 更新时，冲突主要集中在少量挂接点，而不是整个业务层大面积冲突。

## 4. 核心领域模型总览

## 4.1 限界上下文

1. **Enterprise Context**：企业、用户、成员、角色、空间
2. **Talent Context**：专家模板、行业方案、招募实例化
3. **Employee Context**：数字员工、Profile 映射、能力装配
4. **Conversation Context**：私聊、群聊、TeamConversation、消息流、事件流
5. **Execution Context**：TeamRun、TeamTask、OrchestrationPlan、ScheduledJob
6. **Knowledge Context**：知识库、文档、绑定关系
7. **Capability Context**：Skill、Connector、Memory 绑定与授权
8. **Governance Context**：日志、审计、费用、统计、后台运营对象

## 4.2 核心对象与 Hermes 映射

### 4.2.1 EmployeeInstance -> Hermes Profile
- AI Team 对象：`EmployeeInstance`
- Hermes 对象：`profile`
- 运行时落点：`~/.hermes/profiles/<profile_name>/`
- 包含：`config.yaml`、`.env`、`memories/`、`skills/`、`sessions/`、`cron/`
- 结论：**一个数字员工 = 一个独立 Hermes Profile**

### 4.2.2 PrivateConversation -> Session
- AI Team 对象：`PrivateConversation`
- Hermes 对象：`Session` / `session_id`
- WebUI 落点：`api/models.py` / `api/streaming.py`
- Runtime 落点：`state.db` + `sessions/*.json|jsonl`
- 执行入口：Gateway 内部 `start_single_agent_run(run_request)` -> `AIAgent.run_conversation()`

### 4.2.3 GroupConversation -> TeamConversation + 多个 EmployeeProfile
- AI Team 对象：`GroupConversation`
- Hermes 原生基础：`MessageEvent + SessionSource + build_session_key()` 处理真实消息平台群聊，`handle_message()` 负责统一接入处理链
- AI Team Web 产品新增：群聊本身是 `TeamConversation`，需要额外的成员管理、路由、回复聚合、协同顺序控制
- 结论：**Hermes 有群聊消息入口能力，但 AI Team 的浏览器群聊协作仍需业务层封装**

### 4.2.4 OrchestrationPlan / TeamTask -> Orchestrator Root Task + Kanban Task Graph
- AI Team 对象：`TeamTask`、`WorkflowStep`、`OrchestrationPlan`
- Hermes 对象：`hermes_cli.kanban_db.Task`
- 提交入口：Gateway 内部 `submit_orchestrator_root_task(orchestration_request)` -> `create_task()` -> `dispatch_once()` -> orchestrator profile 动态拆解 -> worker `AIAgent.run_conversation()`
- 结论：**AI Team 负责协作请求、候选员工与业务解释层；Hermes Kanban + orchestrator 负责真实任务拆解与执行**

### 4.2.5 ScheduledJob -> Cron Job
- AI Team 对象：`ScheduledJob`
- Hermes 对象：`cron.jobs.Job`
- 提交入口：`create_job()` / `run_job()` / `tick()`
- 结论：**员工级周期任务直接映射到 profile-scoped cron job**
### 4.2.6 MemoryItem -> Memory Store / Provider
- AI Team 对象：`MemoryItem`
- Hermes 对象：`MEMORY.md` / `USER.md` / memory provider store
- 结论：AI Team 做可视化、审核、标签与权限包装；底层不重造 memory 内核

### 4.2.7 SkillBinding -> Profile Skills + Allowlist
- AI Team 对象：`SkillBinding`
- Hermes 对象：`skills/` 目录、工具集启停、skill load
- 结论：AI Team 负责“哪些员工能用哪些 skill”，Hermes 负责真正加载和执行

### 4.2.8 ConnectorBinding -> CredentialRef + Runtime Tool Visibility
- AI Team 对象：`ConnectorBinding`
- Hermes 基础：MCP / provider / tool / messaging adapter / env 注入
- 结论：AI Team 要新增“企业连接器对象 + 员工可见性控制 + 凭据引用”，不直接把密钥散落到前端或页面表单

```json
  "last_synced_at": "2026-05-27T10:32:18Z"
}
```
- `RuntimeHandle` 是 Team Panel 与 Runtime 对账的最小句柄集合。对外返回时只承载句柄字段，不承载 `event_cursor`。
- 任何 `TeamRun` / `TeamTask` / `ScheduledJob` 都必须至少能反查到 `profile_name + session_id/task_id/job_id` 之一。

### 5.9 关键状态枚举与执行语义
- `EmployeeInstance.status`
  - `draft`：业务对象已创建，尚未完成 profile 供应。
  - `provisioning`：Gateway 正在同步 profile/config/skills/connectors。
  - `active`：可接单、可聊天、可参与编排。
  - `paused`：保留配置，但不允许新 run。
  - `archived`：从前台隐藏，仅用于历史追溯。
- `TeamConversation.status`
  - `active`：可继续收发消息。
  - `muted`：保留历史，但默认不推送动态。
  - `archived`：只读，不再允许新消息。
- `TeamRun.status`（主状态口径以《2026-05-28-AI Team-共享运行口径定稿版》为准）
  - `queued`：业务请求已落库，尚未提交 Runtime。
  - `routing`：正在做员工路由或协作决策。
  - `submitting`：路由完成，正在向 Runtime 提交 run。
  - `running`：Runtime 已接受并正在执行。
  - `waiting_human`：等待用户输入或人工审批。
  - `succeeded` / `failed` / `cancelled`：最终闭环状态。
  - 注：`waiting_children` / `partial_success` / `reconnecting` 不再作为持久化主状态枚举，仅在编排展示态/聚合视图或 Gateway 内部过程态中使用。
- `ScheduledJob.status`（主状态口径以《2026-05-28-AI Team-共享运行口径定稿版》为准）
  - `draft`：尚未创建到底层 cron。
  - `enabled`：已注册，等待调度。
  - `paused`：保留 job_id，但暂停后续触发。
  - `error`：连续失败达到阈值，需人工处理。
  - `archived`：已停用，仅保留审计记录。
  - 注：单次 tick 执行状态由 `TeamRun + run_event` 表达，`ScheduledJob.status` 不承载 `running`。

### 5.10 CredentialResolutionPlan
```json
{
  "enterprise_id": "ent_001",
  "employee_id": "emp_marketing_001",
  "credential_ref": "cred_001",
  "connector_id": "conn_feishu_sales",
  "injection_mode": "env_overlay",
  "runtime_target": "single_agent_run",
  "alias_env": ["FEISHU_APP_ID", "FEISHU_APP_SECRET"],

```
- Team Panel 只保存 `credential_ref` 与授权关系；不向前端返回原始 secret。
- Gateway 在 run / cron 创建时解析 `CredentialResolutionPlan`，决定是进程内 env overlay 还是安全落地到 profile-scoped runtime bundle。

## 6. 主控制流

### 6.1 招募员工主流程
1. 用户进入人才市场或行业方案页。
2. Team Panel 读取 `AgentTemplate` / `IndustrySolution`。
3. 用户点击招募或一键应用。
4. Team Panel 创建 `EmployeeInstance`。
5. Team Panel 为该员工分配唯一 `profile_name`。
6. Gateway Adapter/Provisioner 创建对应 Hermes Profile 目录。
7. Team Panel 把 Prompt、Skill、Memory、Knowledge、Connector 绑定写入员工配置。
8. 员工进入前台列表、组织架构、办公室动态。

### 6.2 私聊主流程
1. 用户在前台选择某个员工。
2. Team Panel 校验员工状态、权限、绑定资源。
3. Team Panel 查找或创建 `PrivateConversation`。
4. Team Panel 生成 `RunRequest(mode=single_agent)`。
5. Team Panel 通过中间转换层把业务对象翻译为 Gateway 可执行对象，并调用 `start_single_agent_run(run_request)`。
6. Agent Gateway 选择具体宿主实现：可直接调用 `AIAgent.run_conversation()`，也可在 V1 内部复用 WebUI 现有流式会话骨架。
7. Runtime 以员工对应 profile 执行。
8. 流式 token / reasoning / tool 事件回流。
9. Team Panel 包装为消息气泡、工具卡、成本信息、记忆沉淀结果。

### 6.3 群聊主流程
1. 用户在 `GroupConversation` 输入消息，可带 `@employee`。
2. Team Panel 根据成员、@提及、路由规则生成 `TeamRun`。
3. Gateway 先处理群聊入口语义：会话键、成员隔离/共享、thread 归属、回复锚点等规则与 Hermes Gateway 保持一致。
4. Team Panel 做路由决策：若仅需单员工回答，则转 `single_agent`；若需协作，则转 `orchestration`。
5. 单员工路径调用 `start_single_agent_run()`；协作路径调用 `start_group_run()` 并创建 orchestrator 根任务。
6. orchestrator 与下游 worker 执行事件回流到 Team Panel，Team Panel 再按“员工头像 + 阶段 + 回复顺序”重组前端流。
7. 最终汇总写回群聊消息流和运行日志。

### 6.4 编排主流程
1. Team Panel 把业务任务翻译为 `OrchestrationPlan` / `CollaborationRequest`。
2. 请求中携带：任务目标、上下文、候选员工 roster、每个员工的能力摘要，以及可选的行业方案默认协作模板。
3. Gateway 不静态展开整棵任务树，而是先创建一个 assignee=`orchestrator` 的根 Kanban Task。
4. `dispatch_once()` 触发 orchestrator profile 执行，由 orchestrator 动态拆解下游 `TeamTask` 并分配给目标 worker profile。
5. worker 通过 `kanban-worker` 技能完成任务并回写结果；orchestrator 负责汇总、续派与收口。
6. Team Panel 订阅 orchestrator 拆解过程、任务生成事件、worker 执行事件，并实时重组为任务树、协同气泡与进度时间线。

### 6.5 Loop 主流程
1. 用户在员工详情或 Loop 面板创建 `ScheduledJob`。
2. Team Panel 生成 schedule 与 prompt。
3. Gateway 创建 Hermes cron job，并绑定目标 profile。
4. `cron.scheduler.tick()` 触发执行。
5. Runtime 输出结果。
6. Team Panel 把结果回流到日志、记忆、费用、员工状态。

### 6.6 失败、重试与补偿闭环
1. 所有创建型入口（招募、run、scheduled job）都必须带 `idempotency_key`，先落 Team Panel 业务记录，再触发 Runtime。
2. 若 Runtime 尚未接受请求即失败，业务状态停留在 `queued/failed_to_start`，不得生成“幽灵 task/job/session”。
3. 若 Runtime 已接受但前端流断开，Gateway 通过 `runtime_handle + event_cursor` 做增量补拉，恢复 TeamRun 时间线。
4. 编排中某个 worker 失败时，由 orchestrator 决定重试、改派、降级汇总或整体失败；Team Panel 不直接跳过该分支。
5. ScheduledJob 连续失败超过阈值时转 `error/paused`，并触发企业通知策略。
6. 连接器鉴权失败只影响后续 run，不应回写或清空源 `credential_ref`；修复后通过 rotation_version 重新生效。

## 8. Team Panel -> Agent Gateway -> Hermes 映射规则

## 8.1 单员工静态映射
- `employee_id` -> `profile_name`
- `profile_name` -> `~/.hermes/profiles/<name>/`
- `employee prompt config` -> profile 对应 prompt/context 配置
- `employee skill grants` -> profile 可加载的 skills / toolsets
- `employee memory policy` -> `memories/` 或外部 provider 配置
- `employee connector grants` -> profile 可见连接器与 credential_ref 注入

## 8.2 私聊动态映射
- `PrivateConversation` -> WebUI session / Hermes session
- `RunRequest(single_agent)` -> Gateway `start_single_agent_run()` -> `AIAgent.run_conversation()`
- `RunResult` -> 前端消息 + usage + memory/event 回流

## 8.3 群聊动态映射
- `GroupConversation` -> Team Panel 群会话对象
- `MessageRouteDecision(route_mode=single_agent)` -> `TeamRun -> start_single_agent_run()`
- `MessageRouteDecision(route_mode=orchestration)` -> `TeamRun -> orchestrator root task -> Kanban Task[]`
- 若接入真实消息平台，则 session 共享/隔离语义遵从 `build_session_key()`

## 8.4 编排动态映射
- `OrchestrationPlan / CollaborationTemplate` -> orchestrator root task 的输入上下文
- `candidate_employee_ids + roster metadata` -> orchestrator 任务上下文
- orchestrator 拆出的 `WorkflowStep.parents[]` -> `task_links`
- `assignee_employee_id` -> `Task.assignee(profile_name)`
- `workspace_id` -> `workspace_path`
- `TeamRun.status` <- root task / worker heartbeat / task outcome 聚合

## 8.5 Loop 动态映射
- `ScheduledJob` -> Hermes cron job
- `employee_id` -> cron job 的运行 profile
- `goal + policy` -> cron prompt + schedule + notification policy
- `scheduled job status` <- cron recent output / job status

## 8.6 Connector / 凭据 / 认证路径映射
- `EnterpriseConnector + EmployeeConnectorGrant` -> `CredentialResolutionPlan`
- 交互式 run：优先进程内 `env overlay + tool visibility overlay`，不把 raw secret 下发到浏览器
- ScheduledJob / 无人值守任务：允许 Gateway 生成 profile-scoped 安全 runtime bundle，但源口径仍是 `credential_ref`
- `rotation_version` 变化后，新的 run/job 使用新凭据，旧历史记录只保留引用不保留密文

## 8.7 Usage / 审计 / 对账映射
- `TeamRun` / `ScheduledJobRunRecord` -> `UsageLedger + AuditEvent + RunLogDigest`
- `runtime_handle` 是 Team Panel 与 Runtime 之间的 join key
- `event_cursor` 用于断流后的增量补拉，不依赖前端内存态
- `enterprise_id + run_id + profile_name + task_id/job_id` 是最小可检索维度

## 10. 推荐接口分层

### 10.1 Team Panel 北向接口
建议把前端真正依赖的稳定北向接口收敛为以下几组，而不是直接暴露 WebUI 内部 chat 路由：
- `GET /api/team/workbench`：聚合员工、会话、未读、办公室动态摘要
- `POST /api/team/runs`：发起私聊或单员工 run
- `POST /api/team/group-conversations/{conversation_id}/messages`：向群聊提交一条消息并触发路由
- `GET /api/team/runs/{run_id}/events`：获取某次 run 的时间线事件（poll / cursor 模式）
- `GET /api/team/runs/{run_id}/stream`：SSE 订阅某次 run 的流式事件
- `POST /api/team/scheduled-jobs`、`POST /api/team/scheduled-jobs/{id}/pause|resume|run`：Loop 生命周期管理
- 其他对象面：`/api/team/employees/*`、`/api/team/talent-market/*`、`/api/team/knowledge-bases/*`、`/api/team/connectors/*`、`/api/team/billing/*`、`/api/team/admin/*`

推荐的单员工 run 请求体：
```json
{
  "employee_id": "emp_marketing_001",
  "conversation_id": "conv_private_001",
  "message": {
    "text": "请总结今天销售晨会的重点",
    "attachments": []
  },
  "workspace_id": "ws_default",
  "idempotency_key": "run-20260527-001"
}
```

推荐的同步响应体：
```json
{
  "run_id": "run_001",
  "status": "queued",
  "stream_url": "/api/team/runs/run_001/stream?cursor=0",
  "runtime_handle": {
    "kind": "session",
    "profile_name": "ent001-marketing-001",
    "session_id": "sess_abc"
  }
}
```

### 10.2 Timeline / SSE 事件契约
前端不应只消费“文本 token 流”，还必须消费可重放的 `RunTimelineEvent`：
```json
{
  "event_id": "evt_000129",
  "event_cursor": 129,
  "run_id": "run_001",
  "event_type": "task_started",
  "source_type": "kanban_task",
  "source_id": "task_child_004",
  "employee_id": "emp_finance_001",
  "preview": "财务校验开始",
  "event_ts": "2026-05-27T10:32:20Z",
  "payload": {
    "parent_task_id": "task_root_001",
    "phase": "worker",
    "message_anchor": "msg_221"
  }
}
```

最低事件枚举建议固定为：
- `run_created`
- `routing_decided`
- `run_started`
- `message_delta`
- `tool_call`
- `task_created`
- `task_started`
- `task_completed`
- `task_failed`
- `run_waiting_human`
- `result_merged`
- `memory_written`
- `usage_recorded`
- `run_succeeded`
- `run_failed`
- `run_cancelled`
- `heartbeat`
- `error`

### 10.3 Gateway 内部适配接口
这些接口建议服务内可见，不直接暴露给浏览器：
- `ensure_profile(employee_instance) -> ProfileProvisionResult`
- `resolve_connectors(employee_id, runtime_target) -> CredentialResolutionPlan[]`
- `start_single_agent_run(run_request) -> RuntimeHandle`
- `start_group_run(group_request) -> RuntimeHandle`
- `submit_orchestrator_root_task(orchestration_request) -> RuntimeHandle`
- `create_scheduled_job(scheduled_job_request) -> RuntimeHandle`
- `hydrate_runtime_events(runtime_handle, cursor) -> RunTimelineEvent[]`
- `reconcile_runtime_state(runtime_handle) -> ReconcileResult`

### 10.4 Runtime 调用优先级
1. 单员工同步任务：优先 `AIAgent.run_conversation()`；需要标准 API 托管或跨语言隔离时，再切到 API Server `/v1/runs`。
2. 多员工编排：优先创建 assignee=`orchestrator` 的根 Kanban task，再由 `create_task()` + `dispatch_once()` 驱动下游任务图。
3. 周期任务：优先 `cron.jobs.create_job()`。
4. 真实外部消息群聊：复用 Hermes Gateway + `SessionSource/build_session_key()`。

### 10.5 幂等、错误与超时约定
- 所有“创建 run / job / 招募”接口都必须接受 `idempotency_key`。
- 创建成功但流未建立时，以 `run_id + runtime_handle` 为准，不以浏览器 SSE 连接状态为准。
- 推荐错误码：`VALIDATION_FAILED`、`PERMISSION_DENIED`、`PROFILE_NOT_READY`、`CONNECTOR_UNAUTHORIZED`、`RUNTIME_REJECTED`、`RUNTIME_NOT_FOUND`、`USAGE_PENDING`。
- 推荐超时口径：
  - 北向创建接口：15s 内返回 `run_id` 或明确错误
  - SSE idle heartbeat：15s 一次
  - 断流后补拉：60s 内可从 `event_cursor` 恢复

## 11. 非功能、安全与恢复要求

### 11.1 性能与容量目标
- 私聊创建接口应在 15s 内返回 `run_id`，前端可先进入 `queued/running`。
- 健康上游前提下，单员工首个可见流式事件目标为 p95 < 5s。
- 群聊路由决策目标为 p95 < 2s；是否协作必须尽早确定，避免前端长时间无反馈。
- 单次 TeamRun 至少支持 100+ timeline events 的稳定回放，不依赖浏览器内存保序。
- ScheduledJob 执行完成后，状态应在 30s 内同步回 Team Panel。

### 11.2 可观测性与审计字段
每条运行日志、计费记录和审计事件至少带：
- `enterprise_id`
- `employee_id`（如适用）
- `conversation_id` / `run_id`
- `profile_name`
- `session_id` / `task_id` / `job_id`
- `event_id` / `event_cursor`
- `idempotency_key`

日志分层要求：
- Runtime 原始日志：保留在 Hermes 日志与状态系统
- Gateway 适配日志：记录业务对象到 runtime 对象的翻译结果
- Team Panel 审计日志：记录谁在什么企业做了什么动作，不复制全量原始推理内容

### 11.3 鉴权、授权与租户隔离
- Team Panel 用户鉴权和企业角色判断必须先于任何 Runtime 调用。
- 所有业务对象必须带 `enterprise_id`，且 `employee_id/profile_name` 只能在同一企业空间内解析。
- 前端永远不直接拿到 runtime provider key、connector raw secret 或 profile 文件路径。
- 群聊协作中每个 worker 的可见资源由“企业授权 + 员工 grant + runtime target”三者共同决定。

### 11.4 凭据与连接器安全
- 密钥口径保留在 Team Panel secret store（或等价安全存储），业务数据库只引用 `credential_ref`。
- 交互式 run 优先采用进程内注入，减少 secret 落盘。
- 对于 cron 等无人值守任务，如必须落盘，则仅允许写入 profile-scoped 安全 bundle，权限最小化（如 `0600`）并带 rotation_version。
- 任何日志、审计、前端响应都不得回显 raw secret。

### 11.5 备份、恢复与运行对账
- Team Panel 控制面数据库需要独立备份；V1 目标可先按 `RPO <= 15 分钟、RTO <= 4 小时` 设计。
- Hermes 侧至少要覆盖 `state.db`、`sessions/`、profile 目录、cron job 定义。
- 恢复后优先通过 `runtime_handle` 扫描未闭环 run/task/job，并把 Team Panel 状态补齐。
- 任何 UI 显示“已完成”的对象，都必须能回查到 Runtime 终态或明确标记 `reconcile_pending`。

## 12. 关键风险

### 12.1 把 Team Panel 做成第二套 Runtime
风险：在 Team Panel 中重新实现会话、worker、调度、续跑。
控制：单次任务、编排、定时全部优先映射到 Hermes 现成对象。

### 12.2 把 WebUI 原生 Session 直接当业务对象
风险：员工/团队/企业语义丢失。
控制：Session 只作为运行记录对象，不直接替代 Employee/Conversation/Run。

### 12.3 群聊直接套用单会话 SSE
风险：无法表达多员工并发、任务树和接力回复。
控制：新增 TeamRun timeline 事件层。

### 12.4 凭据/连接器直接写进 profile 文件
风险：多租户隔离弱、审计困难。
控制：Team Panel 管理 credential_ref，Gateway 运行时注入。

### 12.5 业务对象与 Runtime 状态无法对账
风险：页面看到“完成”，Runtime 其实失败。
控制：每个业务 run / task / scheduled_job 必须保存 runtime_handle，并能反查到 task_id/job_id/session_id/profile_name。

### 12.6 把行业方案固化成静态 workflow
风险：产品蓝图层与执行策略层混淆，后续方案升级会牵动整套编排。
控制：行业方案只绑定默认协作模板或 roster，不直接等价 workflow 本体。

## 14. 本文可直接指导开发的结论

1. **数字员工必须落成独立 Hermes Profile。**
2. **WebUI 可以作为 V1 宿主壳与前后端骨架，但不能替代 Team Panel 的业务对象层。**
3. **Agent Gateway 的职责是翻译业务对象、供应 profile、提交 run/task/job、回流事件与解析凭据，而不是变成第二套产品后台。**
4. **私聊优先走单 Agent run；群聊优先由 Team Panel 先做路由，再决定单员工还是 Kanban 编排。**
5. **多 Agent 协作优先映射 Hermes Kanban，并由 orchestrator profile 动态拆解下游任务，不自造第二套 worker 协议。**
6. **Loop/定时任务产品对象统一为 `ScheduledJob`，底层优先映射 Hermes Cron。**
7. **所有 run/task/job 都必须带 runtime_handle，支持断流补拉和运行对账。**
8. **连接器与密钥的产品权威口径在 Team Panel，Runtime 只拿到按员工和目标运行态解析后的授权结果。**
9. **跨文档共享契约统一以《2026-05-28-AI Team-共享运行口径定稿版》为准；模块文档不得再各自发明第二套事件、状态、游标或角色协议。**
