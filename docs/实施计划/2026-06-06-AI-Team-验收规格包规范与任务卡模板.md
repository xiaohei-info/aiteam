# AI Team 验收规格包规范与任务卡模板

> 本文件定义验收规格包怎么写，以及开发 / Review / QA / PM 卡怎么引用它。

---

## 1. 验收规格包编写规范

### 1.1 文件粒度
- 一个功能域一份规格文档
- 文档内按功能点拆条目
- 规格文档命名优先按 PRD 域或强耦合域组织

### 1.2 条目粒度
每个 `spec_id` 必须可被单独引用、单独 review、单独 QA、单独 PM 验收。

### 1.2A Canonical 粒度规则
- **功能域验收规格包是最细粒度的长期参考文档。**
- 后续开发任务、Review 任务、QA 任务、PM 验收任务，都应直接从对应规格包中的 `spec_id` 拆出。
- **不再长期并行维护“第一批 / 第二批 / 第三批开发任务清单”这类独立 canonical 文档。**
- 若执行阶段需要临时任务台账，应以 Kanban 卡本身和规格包附录中的开发留档为准，而不是再扩张新的批次文档体系。

### 1.3 每个条目必须包含
- `spec_id`
- `covers_prd`
- `feature_point`
- `user_visible_expectation`
- `design_refs`
- `dev_done_when`
- `review_checkpoints`
- `qa_checks`
- `pm_acceptance`
- `not_done_if`
- `evidence_required`
- `parallel_notes`

## 2. 开发卡模板

```yaml
card_type: backend | frontend | integration
covers_prd: [P05]
spec_refs: [P05-F03]
design_refs:
  - docs/技术设计/详细设计文档/2026-05-27-AI Team-前端页面与接口契约详细设计.md#p05
matrix_ref:
  - docs/实施计划/2026-06-06-AI-Team-PRD行级验收矩阵.md#p05
spec_pack_ref:
  - docs/实施计划/验收规格包/2026-06-06-AI-Team-P05-P06-对话与协作验收规格.md#p05-f03
worktree_scope: dedicated | shared-lane | master-forbidden
worktree_name: wt/p05-private-chat
owner_role: backend-eng | frontend-eng | integration-eng
preconditions:
  - 依赖已完成项
inputs:
  - 需要读取的接口/fixture/前置数据
outputs:
  - 修改文件
  - 产生的接口/页面/测试
not_done_if:
  - 直接摘录规格包中的 not_done_if

evidence_required:
  - 测试结果
  - 接口调用证据
  - 截图/录屏
```

### 开发卡规则
1. 一个开发卡默认只覆盖 1 个 `spec_id`
2. 若覆盖多个 `spec_id`，必须证明它们属于同一功能点子范围
3. 不允许把后端、前端、QA 收口混在一张卡里
4. 任务卡应直接从规格包条目拆出；规格包本身就是最细粒度长期参考文档
5. 若任务已进入 Kanban，执行留档应回填到规格包附录，而不是再扩张新的批次任务文档

## 3. Review 卡模板

```yaml
card_type: review
covers_prd: [P05]
spec_refs: [P05-F03]
worktree_scope: shared-lane
worktree_name: wt/p05-private-chat
review_dimensions:
  - spec_compliance
  - design_compliance
  - regression_risk
required_output:
  - verdict_per_spec
  - blocking_issues
  - followup_cards_if_needed
```

### Review 规则
- review 不是只看代码能不能跑，而是逐条核对 `spec_id`
- verdict 只能是：`pass` / `revise` / `block`
- 若 `block`，必须指出是 PRD 偏离、设计偏离、还是实现不完整

## 4. QA 卡模板

```yaml
card_type: qa
covers_prd: [P05]
spec_refs: [P05-F01, P05-F02, P05-F03]
worktree_scope: shared-lane
worktree_name: wt/p05-private-chat
qa_scope:
  - happy_path
  - empty_state
  - error_state
  - permission_state
  - regression
required_output:
  - checklist_per_spec
  - evidence_links
  - fail_cases
```

### QA 规则
- QA 必须按 `spec_id` 出 checklist
- 不允许只说“页面能打开”或“主流程可用”
- 至少验证主路径 + 空态/错态/权限态 + 关键回归

---

## 5. PM 验收卡模板

```yaml
card_type: pm-acceptance
covers_prd: [P05]
spec_refs: [P05-F01, P05-F02, P05-F03, P05-F04]
matrix_target: P05
required_inputs:
  - dev_evidence
  - review_verdict
  - qa_evidence
  - demo_recording_if_needed
acceptance_output:
  - accepted_verified | stay_integration_partial
  - remaining_gaps
```

### PM 规则
- PM 只能基于证据升级矩阵状态
- 某个功能域还有任何关键 `spec_id` 未通过时，不允许升级矩阵行到 `accepted_verified`

---

## 6. worktree 规则摘要

- 开发 / review / QA 默认不允许直接在 `master` 工作区执行
- 推荐按 lane 建 worktree，而不是按极小任务建 worktree
- 高风险 / 高冲突任务使用 dedicated worktree
- 同域连续小卡可共享 shared-lane worktree
