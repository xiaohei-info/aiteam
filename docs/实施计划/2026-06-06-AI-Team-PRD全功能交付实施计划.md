# AI Team PRD 全功能交付实施计划

> 目标口径：**以需求文档中的 PRD 全功能交付为硬目标**。
>
> 实现约束：**以 `docs/技术设计/` 下已冻结的技术设计方案为准**，不得脱离既有设计边界自行改口径。

---

## 1. 目标与边界

### 1.0 新增强制约束（用户明确追加）

1. **前端实现必须以 PRD HTML 原型为准。**
   - 前端相关功能实现不得自行发挥页面结构、布局、交互链路、信息层级、主按钮位置、主流程跳转。
   - 默认权威原型来源：
     - `docs/需求文档/AI-Team-PRD-v2.html`
     - `docs/需求文档/AI-Team-PRD.html`
     - `docs/需求文档/AI-Team-Demo.html`
     - `docs/需求文档/AI-Team-Office.html`
   - 若现有页面与 PRD HTML 原型冲突，必须以原型为准；如原型之间冲突，再升级给 PM/你裁决。
   - Coding Agent 不允许把“更工程化 / 更简洁 / 更现代”的主观判断当成改原型的理由。
2. **除移动端外，其余功能均以需求文档中的功能原型为准。**
   - 任何历史的 merge / partial / scoped 口径，都不能自动覆盖 PRD 原始目标。
   - 若现有技术设计只提供阶段性实现口径，必须先补设计裁决，再进入最终开发收口。
3. **外部真实服务可暂时保留 mock 形式。**
   - 微信登录、短信登录、支付等与真实外部服务交互的能力，在本轮允许保持 mock/provider-neutral 方式推进。
   - 但前提是：**产品功能路径必须完整可演示、可验证**，不能因为 mock 就省略页面、状态、异常分支、入口与反馈。

### 1.1 最终目标

在不脱离现有技术设计约束的前提下，完成需求文档中 PRD 定义的全功能交付，至少覆盖：

- 企业前台：P01-P10（其中移动端若 PRD 明确要求，则作为单独交付波次；若当前技术设计仍冻结为非目标，则需先补设计裁决后再进入实现）
- 企业后台：B01-B09
- 系统后台：S01-S04
- 横切能力：RBAC、Audit、Export、Billing、Run/Timeline、知识检索、连接器安全、Loop/定时任务结果回流

### 1.2 实现约束

必须遵守：

1. 页面、接口、对象语义优先服从 `docs/技术设计/` 中的正式设计文档。
2. 前端不直接依赖 Hermes Runtime 内部对象，正式产品契约以 Team Panel 北向 API 为准。
3. 不允许再次把「demo 可用 / shell 可展示 / API 有返回」误判为「PRD 完整交付」。
4. 每个功能项必须用 **产品验收路径** 收口，而不是只用代码存在、路由存在或测试存在收口。

## 2. 对旧流程失败的判断

本次不完整交付的主要原因，不是 PRD 完全没被计划到，也不是技术设计完全没覆盖，而是以下三点叠加：

1. **acceptance drift（验收漂移）**
   - story / batch 卡片 done，并不等于对应 PRD 行 accepted_verified
   - scoped partial delivery 被误读成 full-product closure

2. **task grain mismatch（任务粒度失真）**
   - 卡片过大，混合「后端能力骨架」「前端 shell」「产品体验补深」三类性质不同的工作
   - Coding Agent 在大卡里容易只完成最容易的部分（路由、聚合、占位页、基础 API），剩余产品深度被隐藏

3. **缺少稳定的“单一收口真相层”**
   - 需求文档、技术设计、计划、Kanban 卡片、closeout 文档之间没有一个始终被严格执行的 row-level 收口机制

因此，新工作流的首要目标不是直接开干，而是：

> 先建立一个**不会再把 partial 当 full**、同时支持多 Coding Agent 并行推进的执行系统。

## 3. 新工作流：验收规格先行 + 分角色任务执行 + 一条验收主线

### 3.1 计划结构

不再把「设计文档 -> 一个很粗的 plan -> 大量 Kanban 卡」直接串起来，而是拆成下面几层：

#### L0：PRD 行级验收矩阵（唯一收口真相）

这是整个项目的唯一验收基线。每一行必须对应一个明确 PRD 功能项。

每行至少包含：

- `prd_id`：如 P02 / P05 / B05
- `功能名称`
- `PRD 原型 / 需求摘要`
- `技术设计对应文档与章节`
- `后端完成定义`
- `前端完成定义`
- `场景验收定义`
- `状态`：`not_started | backend_partial | frontend_partial | integration_partial | accepted_verified`
- `证据链接`

**规则：**
- 任何 Kanban 卡 done，都不能直接改 PRD 行状态为 accepted_verified。
- 只有通过「后端验证 + 前端验收 + 场景闭环」三证合一，PRD 行才能升级为 accepted_verified。

#### L0.5：功能域验收规格包（强制层）

这一层借鉴 TDD 的思想，但对象不是“先写单元测试”，而是：

> **先为每个功能点写清验收规格，再派发开发 / review / QA / PM 任务。**

推荐组织方式不是“每个很小的功能点单独一篇文档”，也不是“只有很粗的功能域原则”，而是：

- **一个功能域一份规格文档**
- **文档内按功能点拆成独立规格条目**
- **任务卡按功能点或功能点子范围引用条目执行**

每个规格条目至少包含：

- `spec_id`：如 `P05-F03`
- `covers_prd`
- `feature_point`：具体功能点名称
- `user_visible_expectation`：用户能看到/能操作到什么
- `design_refs`
- `preconditions / fixtures`
- `dev_done_when`
- `review_checkpoints`
- `qa_checks`
- `pm_acceptance`
- `not_done_if`
- `evidence_required`
- `parallel_notes`：是否允许并行、是否属于高冲突文件区

这样做的原因：
- 若“一个功能点一篇文档”，文档数量会爆炸，后期很快失控
- 若只保留“功能域级粗标准”，执行卡还是会回到自由发挥
- **因此要把“文件组织按功能域”与“执行粒度按功能点”分开**

#### L1：能力包实施计划（可选增强层）

这一层不再是默认强制前置，而是一个**共享能力依赖复杂时才启用的增强层**。

什么时候需要单独写能力包实施计划：
- 当 PRD 行很多、共享能力依赖复杂、并行团队较多时
- 当同一批任务横跨多个页面、多个后端域、多个前端模块时
- 当如果直接从“矩阵 + 验收规格包”拆 Kanban，容易出现重复造轮子、协议漂移或跨页面共享能力被各自实现时

什么时候可以跳过单独文档，直接拆任务：
- 当 orchestrator 能够严格按 PRD 行矩阵 + 功能域验收规格包 + 技术设计章节直接拆出高质量任务卡
- 当当前波次只覆盖少量功能行，依赖关系清晰
- 当每张卡都能稳定绑定 `covers_prd + spec_refs + design_refs + not_done_if + evidence_required`

换句话说：
- **能力包不是目的，而是降低共享底座漂移的一种组织手段。**
- 当前阶段真正应该强制补上的，是“功能域验收规格包”，不是再写一层泛泛的中间计划。

#### L2：可执行 Kanban 卡（2~8 小时粒度）

这一层才给 Coding Agent。

每张卡必须满足：

- 单一责任
- 单一主目标
- 单一验收标准
- 明确输入输出
- 明确依赖
- 明确 owner

每张卡只允许属于下面三类之一：

1. **后端能力卡**
   - schema / migration
   - API
   - query / command service
   - event / timeline
   - integration adapter

2. **前端实现卡**
   - PRD 页面结构
   - 组件实现
   - 状态机 UI
   - 真接口接线

3. **联调 / 验证卡**
   - scene test
   - E2E
   - regression
   - product acceptance

**禁止：**
- 一张卡同时包含“补 schema + 写 API + 做页面 + 做 QA”
- 一张卡既做后端主能力，又做前端产品表达，又做验收收口

## 4. 颗粒度策略：文件按功能域组织，任务按功能点执行

### 4.1 为什么不能“一个功能点一篇验收文档”

如果每个按钮、每个状态、每个交互点都单独写一篇验收标准，会导致：

- 文档数量爆炸
- 大量重复前置条件与上下文
- reviewer / QA / PM 找不到同域功能的统一口径
- 一旦 PRD 或设计有小调整，维护成本极高

### 4.2 为什么也不能只保留“功能域级粗标准”

如果只有类似“P05 单聊页验收标准”这种很粗的文档，而没有再细到功能点条目，执行任务时仍会出现：

- agent 只实现最容易的路径
- reviewer 只能按整体感觉评估
- QA 无法逐点打勾
- PM 最后仍会被 closeout 叙事带偏

### 4.3 推荐做法：文档按功能域，条目按功能点，任务按条目引用

推荐结构：

- **一个功能域一份规格文档**
  - 如：`P05-单聊验收规格.md`
  - 如：`P06-群聊验收规格.md`
  - 如：`B05-连接器验收规格.md`
- **文档内部按功能点编号**
  - `P05-F01 历史消息加载`
  - `P05-F02 引用消息`
  - `P05-F03 附件上传与展示`
  - `P05-F04 tool_call 可解释渲染`
  - `P05-F05 retry / abort / 错误恢复`
- **任务卡引用条目，不重复发明标准**
  - 开发卡、review 卡、QA 卡、PM 卡都直接引用 `spec_id`

这样做的好处：
- 规格文件数量可控
- 功能点粒度足够细，能直接驱动任务
- 同一功能域的开发 / review / QA / PM 可以共享同一份口径
- 便于后续并行和批量验收

### 4.4 任务粒度建议

#### 规格文档粒度
- 以 **页面 / 后台域 / 核心能力域** 为单位建文件
- 每个文件通常覆盖 5~20 个功能点条目

#### 执行卡粒度
- 单张卡建议对应：
  - **1 个功能点**，或
  - **1 个功能点下的同类子范围**
- 目标仍是 **2~8 小时可完成的完整工作单元**

#### 判断标准
- 如果一张卡不能清楚回答“它实现的是哪个 `spec_id`”，说明太大
- 如果一张卡完成后 reviewer/QA 仍不能独立验收，说明标准还太粗
- 如果一张卡只是在做一个视觉微调或一个字段改名，而脱离完整功能点闭环，说明它太碎

## 5. 并行推进的组织方式

### 5.1 并行原则

为了最快实现 PRD 全功能，采用：

> **后端主轨先行 + 前端按冻结契约并行 + 场景联调持续插入**

不是“后端全做完再开始前端”，也不是“前后端各自乱跑”。

### 5.2 三条并行轨道

#### 轨道 A：后端主轨（最高优先级）
目标：把所有 shell/stub/aggregate-only 的后端能力补成真实能力。

优先顺序：

1. C03 单聊运行与时间线
2. C04 群聊协作与任务树
3. C05 知识库接入与检索
4. C09 连接器与凭据安全
5. C08 计费 / usage 聚合
6. C10 行业方案 apply 原子性
7. C12 Loop / ScheduledJob
8. C13 横切 RBAC / Audit / Export

#### 轨道 B：前端 PRD 对齐轨道
目标：按照需求文档中的 PRD 原型图重做前台与后台产品表达。

做法：
- 先按 PRD 原型图搭页面结构和交互层
- 接口先吃冻结 JSON fixture / contract sample
- 后端 ready 后逐张换成真接口

优先顺序：

1. P02 工作台
2. P03 / P04 人才市场 + 专家详情
3. P05 单聊
4. P06 群聊
5. P08 知识库
6. P09 办公室动态
7. B01 / B04 / B05 / B06 / B08
8. S01-S04

#### 轨道 C：场景联调轨道
目标：防止再次出现“卡 done 了，但产品不能用”。

固定五波验收：

1. 招募闭环：P03 -> P04 -> B01 -> P02
2. 私聊闭环：P02 -> P05
3. 群聊闭环：P06
4. 知识库 / 办公室闭环：P08 -> P05/P06, P09
5. 后台治理闭环：B01/B04/B05/B06/B08 + S01-S04

## 6. 新的 Kanban 工作流约束

### 6.1 卡片类型强制化

每张 Kanban 卡必须带：

- `card_type`
- `covers_prd`
- `spec_refs`
- `design_refs`
- `worktree_scope`

`card_type` 只允许：

- `backend`
- `frontend`
- `integration`
- `review`
- `qa`
- `pm-acceptance`

不同类型的卡有不同 done 规则。

### 6.2 Definition of Done（按卡类型）

#### backend 卡 done 条件
- 代码完成
- 自动化测试通过
- 提供接口示例 / schema evidence
- reviewer 通过
- 但 **不允许** 直接宣称对应 PRD 行关闭

#### frontend 卡 done 条件
- 页面符合 PRD 原型结构
- 状态机完整（空/错/权限/正常）
- 组件与路由行为完成
- reviewer 通过
- 若尚未接真接口，状态只能是 `frontend_partial`

#### integration 卡 done 条件
- 真接口接入完成
- 前后端字段一致
- 场景主链路跑通
- QA 验证通过

#### pm-acceptance 卡 done 条件
- 对应 PRD 行满足：
  - 后端完成定义
  - 前端完成定义
  - 场景验收定义
- 才能把 PRD 行状态升级为 `accepted_verified`

### 6.3 PRD 行状态升级规则

只允许按下面顺序升级：

`not_started`
-> `backend_partial`
-> `frontend_partial`
-> `integration_partial`
-> `accepted_verified`

严禁跳级。

## 7. 防跑偏机制

为了让很多个 Coding Agent 并行推进时仍然不偏目标，必须加 5 个机制。

### 7.1 单一真相包（Source of Truth Pack）

每个 agent 开工前统一只读这 4 类输入：

1. 对应 PRD 行摘要
2. 对应技术设计章节
3. 对应功能域验收规格条目
4. 对应卡片验收标准

禁止 agent 自己重新理解范围、擅自加功能、擅自改契约。

### 7.2 卡片必须显式绑定 PRD 行与规格条目

每张卡必须写：
- `covers_prd`: [`P05`] 或 [`P08`, `P09`] 等
- `spec_refs`: [`P05-F03`, `P05-F05`] 等
- `design_refs`: 对应技术设计章节

没有绑定的卡不允许执行。

### 7.3 reviewer 不只看代码，要看“是否符合 PRD + 技术设计”

reviewer 评审维度固定两段：

1. 是否符合技术设计
2. 是否朝 PRD 行验收定义逼近

而不是只看“代码能跑”。

### 7.4 QA 不按“页面存在”验收，按“场景闭环”验收

例如 P05 不是“聊天页打开就算通过”，而是：

- 历史加载
- 引用
- 附件
- tool_call 显示
- retry
- abort

必须逐项验证。

### 7.5 closeout 文档不再自己定义完成

closeout 只能引用：
- PRD 行矩阵
- reviewer verdict
- QA evidence
- PM acceptance

不能再反过来用一份 closeout 文档宣布 full complete。

## 8. 第一阶段实施建议（正式进入实施前的最小控制面）

### 8.1 先补 3 份控制文档

在真正大规模派发 Coding Agent 之前，先固定这 3 份文档：

1. `PRD 行级验收矩阵`
2. `功能域验收规格包`
3. 本《PRD 全功能交付实施计划》

其中：
- `PRD 行级验收矩阵` 负责回答“要交付哪些功能行”
- `功能域验收规格包` 负责回答“每个功能点怎么算做完”
- 本实施计划负责回答“任务怎么拆、角色怎么协作、门禁怎么执行”

### 8.2 再判断是否需要补能力包层

推荐默认策略：
- **先不额外写独立能力包实施计划**
- 直接基于“实施计划 + 矩阵 + 验收规格包”拆第一批 Kanban 任务

只有在以下情况，才补能力包层：
- 多个功能域反复依赖同一共享后端底座
- 多个并行队列持续在 run/timeline、connector、billing、knowledge 等共享模块冲突
- orchestrator 发现直接拆任务无法稳定控制依赖顺序与复用边界

### 8.3 规格包优先落地顺序

建议先写第一批最关键的功能域验收规格：

1. `P02-工作台验收规格`
2. `P03-P04-人才市场与专家详情验收规格`
3. `P05-单聊验收规格`
4. `P06-群聊验收规格`
5. `P08-知识库验收规格`
6. `B01-数字员工管理验收规格`
7. `B05-连接器验收规格`
8. `B04-B09-计费与充值验收规格`

写完这批，就可以开始真正派发第一波开发 / review / QA / PM 卡。

## 9. 具体分工建议（多 Coding Agent 并行）

推荐角色：

- `orchestrator`：只做拆解、路由、收口，不写代码
- `architect`：接口、边界、迁移、冲突裁决
- `backend-eng-*`：后端能力卡
- `frontend-eng-*`：前端页面卡
- `data-rd`：知识库/计费/连接器/聚合相关数据能力
- `reviewer`：技术与契约审查
- `qa`：场景验证
- `pm`：PRD 行最终签收

### 并行原则

- 同一功能域下，后端卡与前端骨架卡可并行
- 真接口接线卡必须依赖对应后端卡
- QA 卡必须依赖 integration 卡
- PM acceptance 卡必须依赖 reviewer + QA

### 9.1 并行开发的代码冲突控制

你提到的这个点是对的：

> **一旦真的进入多 Agent 并行开发，不做工作区隔离，代码冲突和脏状态污染几乎必然出现。**

结合当前仓库现状（`master` 工作区已有未提交改动，且历史上确实存在多个 worktree 并行留下脏状态的情况），推荐口径是：

- **默认使用 git worktree 做并行隔离**
- **不是每张卡都新建一个 worktree**
- **而是按“任务簇 / lane”建 worktree**

### 9.2 为什么需要 worktree

不隔离的主要风险：

- 多个 agent 修改同一工作区，互相覆盖
- 一个卡的临时调试改动污染另一个卡
- reviewer / QA 无法复现单卡结果
- orchestrator 很难判断某段 diff 属于哪张卡
- 当前仓库本来就不是完全干净基线，继续共享工作区只会放大混乱

### 9.3 worktree 使用规则

#### 必须新建 worktree 的情况

- 不同 agent 并行做不同功能域
- 同一波次中既有后端卡又有前端卡
- 任务会改动高冲突文件区（如共享路由、共享契约、聚合视图、前端主页面骨架）
- 任务预计持续超过一个小回合，且需要 reviewer/QA 独立复现

#### 可以共用同一 worktree 的情况

- 同一 lane 下串行执行的同域小卡
- 明确都服务于同一个功能域规格包，且依赖顺序固定
- owner 不变，且在进入下一卡前会完成最小提交/验证

#### 不推荐的做法

- 所有 agent 共用 `master` 工作区
- 每张极小卡都单独开 worktree，导致管理爆炸
- 在 worktree 内长期堆未提交脏改动，不做阶段性收口

### 9.4 推荐的 worktree 粒度

推荐按 **lane / 功能域任务簇** 建 worktree，而不是按单卡建：

- `wt/p02-workbench`
- `wt/p03-p04-marketplace`
- `wt/p05-private-chat`
- `wt/p06-group-chat`
- `wt/p08-knowledge`
- `wt/b01-employee-admin`
- `wt/b05-connectors`
- `wt/b04-b09-billing`

每个 worktree 内可以承载：
- 若干开发卡
- 对应 reviewer 修正
- 对应 QA 修正

但要求：
- 一个 worktree 只服务一个主功能域
- 不把多个无关功能域混在同一 worktree
- 达到阶段验收后及时收口/合并/清理

### 9.5 任务卡必须声明 worktree_scope

每张执行卡必须显式写：

- `worktree_scope: dedicated | shared-lane | master-forbidden`
- `worktree_name`: 如 `wt/p05-private-chat`

默认规则：
- `master-forbidden`：开发、review、QA 卡都不允许直接在主工作区执行
- `dedicated`：高风险/高冲突卡单独工作区
- `shared-lane`：同一功能域 lane 内的小卡共用工作区

### 9.6 进入实施阶段的最终建议

因此，这一步的推荐设计是：

1. **先补功能域验收规格包**，把“功能点级完成标准”定稿
2. **按功能域建立第一批 lane worktree**
3. **基于 `矩阵 + 规格包 + worktree_scope` 拆第一波任务卡**
4. **开发 / review / QA / PM 全都按同一规格条目闭环**

这比“每个功能点单独一篇文档”更可维护，也比“只靠功能域粗标准直接开工”更稳。

## 10. 这份计划的颗粒度说明

这份文档故意不写成“每一张 Kanban 卡的逐行代码级说明”，原因是：

1. 那样成本过高，等于在纸面重新实现一遍系统
2. 会让 orchestrator 和计划本身变成瓶颈
3. 不利于多 agent 并行

但它也没有停留在“很粗的大方向”层面，而是提供了：

- 单一真相层（PRD 行矩阵）
- 功能点级验收规格层（L0.5）
- 可选的能力包增强层（L1）
- 执行卡规则（L2）
- 粒度标准
- 并行轨道
- 门禁与验收机制

这正是为了避免再次出现：

- plan 太粗导致 Coding Agent 跑偏
- plan 太细导致写计划的人变成实现瓶颈

- **若拆卡过程中出现共享能力反复漂移、跨页面重复实现、跨团队协议不一致，再补能力包层。**

## 12. 最终建议

推荐设计：

> **用“PRD 行矩阵”做唯一验收真相，用本实施计划约束拆卡方式，并以“2~8 小时执行卡”驱动多 Coding Agent 并行开发。能力包层是可选增强，不是强制前置。**

边界：

- 不再允许 closeout 文档替代产品验收
- 不再允许 shell / stub / aggregate-only 被误报为 complete
- 不再允许单张卡同时混合后端主能力、前端产品表达、QA 收口三类工作

风险：

- 如果不先建立 PRD 行矩阵，新的并行开发仍会再次漂移
- 如果不把 reviewer/QA/PM 门禁强制绑定到 PRD 行，closeout 叙事仍会压过真实产品完成度

验证方式：

- 每个 PRD 行必须有 `accepted_verified` 证据
- 每个能力包必须有后端、前端、场景三类证据
- 最终交付以需求文档中的产品功能路径为准，而不是以代码文件数、路由数或 closeout 文档措辞为准
