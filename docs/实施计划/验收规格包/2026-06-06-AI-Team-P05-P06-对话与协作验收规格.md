## 3. 条目详细定义

### P05-F01 — 单聊页面结构与历史加载

- `covers_prd`: `P05`
- `feature_point`: 单聊页面结构与历史加载
- `user_visible_expectation`: 用户进入单聊页后，能看到历史消息、输入区、员工摘要、引用上下文。
- `dev_done_when`: 历史分页/滚动加载、空会话、异常重试入口完整。
- `review_checkpoints`: 需核对实现是否同时符合 PRD 原型、接口契约、状态/权限/异常路径。
- `qa_checks`: 需至少验证主路径、空态/错误态、权限态、回归影响。
- `pm_acceptance`: 需按用户视角确认该功能点已达到产品可讲解、可演示、可验证的完整度。
- `not_done_if`: 只能发新消息、无历史/空态时不得标 done。
- `evidence_required`: 页面录屏；历史加载测试；接口样例。
- `parallel_notes`: 建议 dedicated/shared-lane: wt/p05-private-chat

### P05-F02 — 发起私聊 run 与流式过程可见

- `covers_prd`: `P05`
- `feature_point`: 发起私聊 run 与流式过程可见
- `user_visible_expectation`: 用户发送消息后，可看到 timeline/streaming/busy/resolved 等过程态。
- `dev_done_when`: run 创建、事件消费、停止条件、断流恢复符合共享口径。
- `review_checkpoints`: 需核对实现是否同时符合 PRD 原型、接口契约、状态/权限/异常路径。
- `qa_checks`: 需至少验证主路径、空态/错误态、权限态、回归影响。
- `pm_acceptance`: 需按用户视角确认该功能点已达到产品可讲解、可演示、可验证的完整度。
- `not_done_if`: 只显示最终回复、过程不可见时不得标 done。
- `evidence_required`: 事件流日志；SSE 录屏；状态机说明。
- `parallel_notes`: 会改流式消费与 UI 状态，冲突高。

- `dev_done_when`: 上传状态、失败降级、附件展示、tool_call 渲染完整。
- `review_checkpoints`: 需核对实现是否同时符合 PRD 原型、接口契约、状态/权限/异常路径。
- `qa_checks`: 需至少验证主路径、空态/错误态、权限态、回归影响。
- `pm_acceptance`: 需按用户视角确认该功能点已达到产品可讲解、可演示、可验证的完整度。
- `not_done_if`: upload stub、tool_call 不可解释时不得标 done。

### P05-F04 — 重试 / 中止 / 错误恢复

- `pm_acceptance`: 需按用户视角确认该功能点已达到产品可讲解、可演示、可验证的完整度。
- `not_done_if`: 没有真实 abort/cancel 或错误恢复时不得标 done。

## 4. 任务拆分建议

- 一个开发卡建议覆盖 1 个 `spec_id`，或同一 `spec_id` 下的后端/前端子范围。
- 一个 review 卡可以覆盖 1~3 个同域 `spec_id`，但必须逐条给 verdict。
- 一个 QA 卡可以覆盖一个完整小场景，但必须明确映射到本文件的 `spec_id`。
- 一个 PM 验收卡只允许在相关 `spec_id` 的开发、review、QA 证据都齐全后创建。

## 5. 关闭条件

当且仅当本文件中所有 relevant `spec_id` 均已有开发 / review / QA / PM 证据，且对应矩阵行可以升级时，才允许视为该功能域开发-ready 和验收-ready。
