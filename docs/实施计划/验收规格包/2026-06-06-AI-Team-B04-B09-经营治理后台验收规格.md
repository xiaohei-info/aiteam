### 1.3 执行规则

1. 前端表现必须以 PRD HTML 原型为准。
2. 任务完成判定必须同时满足：实现存在、交互成立、异常闭环、证据齐全。
3. 任何 `spec_id` 未通过前，不允许用 closeout 口径替代验收。
4. `B09` 的 mock 边界只针对支付 provider，不针对产品路径；因此套餐、自定义金额、支付方式展示、成功失败反馈、余额刷新、预警阈值都必须单独验收。

- `user_visible_expectation`: 管理员发起充值后，即使 provider 为 mock，也能走完整下单、成功/失败反馈、余额刷新、充值记录落账链路。
- `dev_done_when`: 创建订单、拉起 mock 支付、成功/失败结果页或弹层、余额刷新、记录更新完整。
- `review_checkpoints`: 需核对 mock 替代的是 provider 调用，不是结果页、失败页、刷新逻辑、记录落账逻辑。
- `qa_checks`: 覆盖成功支付、失败支付、取消支付、重复回调或等价异常场景。
- `pm_acceptance`: 从产品视角确认充值在 mock 环境下仍是一条完整可演示的业务路径。
- `not_done_if`: 只返回假成功、无失败路径、余额不刷新时不得标 done。
- `evidence_required`: 充值录屏；订单/记录接口证据；成功失败截图。
- `parallel_notes`: 核心产品路径。

### B09-F05 — 预警阈值与余额不足提示

- `covers_prd`: `B09`
- `feature_point`: 预警阈值与余额不足提示
- `user_visible_expectation`: 管理员能配置或看到余额预警阈值，并在余额不足或临界时收到明确提示。
- `dev_done_when`: 预警阈值展示/配置、触发反馈、与消耗入口衔接成立。
- `review_checkpoints`: 需核对预警不是单纯字段存在，而是会在余额页/消耗入口产生产品级提醒。
- `qa_checks`: 覆盖阈值未设置、设置成功、余额达到阈值、低于阈值后的提示场景。
- `pm_acceptance`: 从产品视角确认 B09 不只是充值页，还具备基础经营预警能力。
- `not_done_if`: 没有预警语义或只能查数据库时不得标 done。
- `evidence_required`: 提示截图；配置证据；场景说明。
- `parallel_notes`: 与 B04/B08 通知联动。

## 4. 任务拆分建议

- 一个开发卡建议覆盖 1 个 `spec_id`，或同一 `spec_id` 下的后端/前端子范围。
- 一个 review 卡可以覆盖 1~3 个同域 `spec_id`，但必须逐条给 verdict。
- 一个 QA 卡可以覆盖一个完整小场景，但必须明确映射到本文件的 `spec_id`。
- 一个 PM 验收卡只允许在相关 `spec_id` 的开发、review、QA 证据都齐全后创建。

## 5. 关闭条件

当且仅当本文件中所有 relevant `spec_id` 均已有开发 / review / QA / PM 证据，且对应矩阵行可以升级时，才允许视为该功能域开发-ready 和验收-ready。


## 6. 开发留档（执行记录）

本节用于记录该功能域后续真正进入执行阶段后的信息，不再依赖独立的“第一批/第二批任务清单”文档。

推荐记录字段：
- `task_id`：内部任务标识（可选）

记录规则：
