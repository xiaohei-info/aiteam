1. 登录页结构、入口选择、企业入户流程必须以 PRD 原型为准。

4. PM 验收时必须显式核对以下原型级可见行为：默认扫码态、二维码生命周期、验证码冷却、登录成功后的进入反馈/引导反馈。

## 3. 条目详细定义

### P01-F01 — 登录页结构、默认入口与引导呈现
- `covers_prd`: `P01`
- `feature_point`: 登录页结构、默认入口与引导呈现
- `user_visible_expectation`: 用户打开登录页后，默认看到符合 PRD 的扫码登录主视图，同时能识别手机号入口、企业入口、引导文案与主 CTA。
- `dev_done_when`: 页面结构、主 CTA、入口切换、默认扫码态、引导文案、空态/错误态符合原型；不是占位页。
- `review_checkpoints`: 核对原型结构、主按钮位置、入口层级、默认态是否为扫码主态、是否与企业入户链路衔接。
- `qa_checks`: 验证桌面端主视图、默认扫码态、入口切换、浏览器刷新后回显行为。
- `pm_acceptance`: 从产品视角确认这是正式登录页，而非调试页/占位页。
- `not_done_if`: 只有单一按钮页、默认不是扫码主态、与 PRD 原型不符时不得标 done。
- `evidence_required`: 页面截图；PRD 对照说明；交互录屏。
- `parallel_notes`: lane: wt/p01-auth-onboarding

### P01-F02 — 微信扫码 / 手机验证码双入口与生命周期反馈
- `covers_prd`: `P01`
- `feature_point`: 微信扫码 / 手机验证码双入口与生命周期反馈
- `user_visible_expectation`: 用户可以选择扫码或验证码方式进入登录流程，并在二维码生成/过期/刷新、验证码发送/冷却/重发等阶段得到明确反馈。
- `dev_done_when`: provider-neutral/mock 方案下，扫码与验证码都能走完整交互链路；二维码生命周期、验证码倒计时、错误提示完整。
- `review_checkpoints`: 核对扫码状态、二维码过期与刷新、验证码倒计时、冷却重发、异常提示、mock 边界是否写清。
- `qa_checks`: 覆盖扫码成功/超时/失效后刷新、验证码正确/错误/冷却/重发场景。
- `pm_acceptance`: 用户从产品角度可理解并能完成两种进入方式，不会困惑于二维码过期或验证码冷却。
- `not_done_if`: 只保留一个入口，或只有表单无真实流程反馈时不得标 done。
- `evidence_required`: 交互录屏；接口证据；失败路径截图。
- `parallel_notes`: 与 auth provider 适配联动，建议 shared-lane

### P01-F03 — token / refresh / device / rate-limit 会话安全
- `covers_prd`: `P01`
- `feature_point`: token / refresh / device / rate-limit 会话安全
- `user_visible_expectation`: 用户登录后获得稳定会话；异常时能看到会话失效、刷新、限流等反馈。
- `dev_done_when`: token/refresh/device/rate-limit 路径成立；会话状态与错误反馈可验证。
- `review_checkpoints`: 核对会话模型、refresh 逻辑、device 绑定、rate-limit 反馈是否符合设计。
- `qa_checks`: 覆盖登录成功后刷新、过期、重复尝试限流等场景。
- `pm_acceptance`: 用户不会因隐藏的会话问题导致路径断裂。
- `not_done_if`: 只有 happy path，无刷新/失效/限流语义时不得标 done。
- `evidence_required`: 接口调用证据；状态录屏；测试记录。
- `parallel_notes`: 后端与前端状态提示需联调

### P01-F04 — 首次企业创建 / 加入企业路径与登录成功进入反馈
- `covers_prd`: `P01`
- `feature_point`: 首次企业创建 / 加入企业路径与登录成功进入反馈
- `user_visible_expectation`: 用户首次登录后，能够创建企业或加入已有企业，并在成功后收到明确的进入企业空间/继续引导反馈。
- `dev_done_when`: enterprise onboarding 路径完整；创建/加入/邀请码或等价入户路径成立；登录成功后有进入企业空间或新手引导反馈。
- `review_checkpoints`: 核对 enterprise session、入户状态、邀请路径与 B08 邀请设计是否一致。
- `qa_checks`: 覆盖首次创建、已有企业加入、无效邀请码、成功进入企业空间回跳场景。
- `pm_acceptance`: 登录成功后用户能真正进入企业空间开始使用产品，而不是停在无解释的中间态。
- `not_done_if`: 登录成功但无法进入企业空间，或成功后无明确进入/引导反馈时不得标 done。
- `evidence_required`: 录屏；接口证据；场景说明。
- `parallel_notes`: 与 settings/invite/enterprise session 设计耦合

### P01-F05 — 登录失败、超时与异常恢复
- `covers_prd`: `P01`
- `feature_point`: 登录失败、超时与异常恢复
- `user_visible_expectation`: 用户在验证码错误、扫码超时、二维码失效、邀请码无效等情况下得到明确恢复路径。
- `dev_done_when`: 常见失败场景均有错误提示、重试路径、返回路径；不会卡死在失效二维码或冷却状态。
- `review_checkpoints`: 核对异常分支是否完整，是否存在静默失败，是否提供从失效态回到可操作态的入口。
- `qa_checks`: 覆盖验证码错误、扫码超时、二维码失效、邀请码无效、网络失败等场景。
- `pm_acceptance`: 失败路径仍然符合可演示、可讲解标准。
- `not_done_if`: 失败后卡死、静默失败或只能刷新页面时不得标 done。
- `evidence_required`: 失败场景截图；录屏；QA checklist。
- `parallel_notes`: QA 必须覆盖失败矩阵

## 4. 任务拆分建议

- 一个开发卡建议覆盖 1 个 `spec_id`，或同一 `spec_id` 下的后端/前端子范围。
- 登录 provider/mock 适配、企业入户、会话安全、失败恢复建议拆成独立子卡。
- PM 验收卡必须在 `P01-F01` 至 `P01-F05` 证据齐全后再创建。

## 5. 关闭条件

当 `P01-F01` 至 `P01-F05` 均通过开发 / review / QA / PM 证据校验后，P01 才允许从 `integration_partial` 升级到 `accepted_verified`。
