## 3. 使用顺序

### 3.1 Architect / Orchestrator
1. 先读总实施计划
2. 再读 PRD 行级验收矩阵
3. 再读对应功能域验收规格包
4. 最后按任务卡模板拆卡并指定 worktree_scope

### 3.2 开发 / Review / QA / PM
1. 先读被分配卡片中的 `covers_prd`
2. 再读卡片中的 `spec_refs`
3. 再回看对应技术设计章节
4. 全流程不得绕开规格包直接凭感觉执行

## 4. 开发-ready 标志

当且仅当以下条件满足时，才允许开始大规模正式开发：

1. 总实施计划已定稿
2. PRD 行级验收矩阵已定稿
3. 对应功能域的验收规格包已经齐备，且覆盖 `P01-P10 / B01-B09 / S01-S04` 全部非豁免项
4. 任务卡模板、review/QA/PM 执行规则已固定
5. worktree 并行隔离规则已固定
6. 对于 `in_design_conflict` / `contract_locked_but_not_complete` 的项，已建立前置裁决门禁，且门禁写到可执行的“允许开什么卡 / 禁止开什么卡”粒度
7. 若 `B03` / `P10` 仍处于门禁态，必须同步引用 `2026-06-09-AI-Team-B03-P10-前置补充验收规格.md`

### 4.1 当前 PM blocker 与收口要求

- `P01`：虽允许 mock provider，但必须按原型验收默认扫码、二维码生命周期、验证码冷却、登录成功后的进入/引导反馈。
- `P09`：必须按办公室原型验收，不接受 seat/feed 聚合 dashboard 替代；至少覆盖画布与工位布局、状态动效/任务气泡、点击工位详情、全屏/平移/缩放、底部任务队列/统计/实时日志、实时更新机制。
- `P10`：当前不是可直接开发口径；仅允许先做 PM 裁决卡/补设计卡，禁止直接创建“P10 开发完成”类卡片。
- `B03`：当前不是可直接关闭口径；必须先明确“恢复独立后台”还是“合并后等价覆盖”的 PM 裁决，并写清映射证据要求。
- `B09`：mock 的是支付 provider，不是产品路径；套餐、自定义金额、支付方式展示、成功失败反馈、余额刷新、预警阈值仍属于硬验收范围。
- `B03/P10`：除主规格包外，还必须同时满足 `2026-06-09-AI-Team-B03-P10-前置补充验收规格.md` 中的前置条目，才允许创建真实实现卡。

## 5. 非目标

本轮文档体系的目标不是替代设计文档，也不是替代具体开发卡，而是：

- 把 PRD 功能点落成任务级验收口径
- 保证多角色、多 agent 并行开发时仍然能按同一标准交付
- 让后续拆卡与实施不再依赖临场口头解释
