# AI Team 技术设计文档体系

## 1. 文档定位

本文是 AI Team 技术设计目录的导航页，用于说明当前已经定稿的正式设计产物、各文档职责边界，以及开发团队进入实现阶段时应按什么顺序阅读。

当前文档体系已经形成：

- 业务解决方案
- 技术概要设计
- 详细设计主稿
- 1 份共享运行口径定稿文档
- 1 份 Phase 2 范围冻结文档
- 1 份 Auth 认证与会话企业入户契约收口文档
- 1 份 S01 系统账号管理契约收口文档
- 5 份模块级详细设计子文档

这套文档的目标不是展示思路，而是作为**开发前正式输入**，让前端、后端、运行时适配、测试与运维都能找到各自的实现依据。

本轮已根据参考文档《`参考文档/Agent Service项目结构与模块功能参考.md`》完成一次实施口径校准：

- 真正落地项目为 `Agent Service`，当前仓库中的代码目录路径：`/home/ubuntu/code/aiteam/app`
- `Agent Service` 是基于 Hermes Web UI 二次开发的服务端基座项目
- `Team Panel` 与 `Agent Gateway` 是其**内部两个核心模块层**，不是要求 V1 额外拆成两个独立部署服务
- 实施默认采用 **对扩展开放、对修改封闭** 的原则：新业务接口/页面/适配器优先落新模块，`server.py` / `api/routes.py` / `api/streaming.py` 只做挂接性或轻量抽取性修改

为避免开发阶段把“设计概念名”和“当前仓库物理目录名”混淆，阅读本目录下所有设计文档时，统一按以下映射理解：

- **Agent Service / `agent-service`** → 当前仓库中的 **`app/`**
- **Agent Runtime / Hermes Runtime** → 当前仓库中的 **`./.hermes/hermes-agent/`**

其中：
- `Agent Service` / `Agent Runtime` 是设计分层中的概念名
- `app/` / `./.hermes/hermes-agent/` 是当前 aiteam 仓库中的实际代码目录名

因此，文档中若出现 `Agent Service`、`Agent Runtime` 等表述，应优先按模块职责理解；若出现具体代码落点、目录结构、实施路径，则以当前仓库实际目录 `app/` 与 `./.hermes/hermes-agent/` 为准。

## 4.2 当前文档体系的完成度判断

本轮开发前 review 的阻塞点已经明确收敛为 5 类共享契约缺口：

- 时间线事件协议
- 事件游标协议
- 控制面主状态机
- 北向 API 最小固定契约
- 权限角色模型

因此，当前体系的正式判断不再是“原 4 份子文档各自收口即可开工”，而是：

- **共享口径** 必须先由 `2026-05-28-AI Team-共享运行口径定稿版.md` 统一裁决
- 5 份子文档再分别承接数据、内部实现、运行、流程、接口的模块内细节

现在的完成度判断应表述为：

- **共享口径已定稿**：跨模块契约有了唯一权威口径
- **数据口径已承接**：控制面对象、表结构、状态语义与角色模型有明确收口目标
- **运行口径已承接**：Gateway 事件转换、游标、补拉与对账有明确收口目标
- **流程口径已承接**：私聊、群聊、编排、Loop 的主链与异常闭环有明确收口目标
- **接口口径已承接**：页面、北向 API、SSE、权限矩阵有明确收口目标

因此，这套文档体系已经达到：

**共享口径与 5 份模块级子文档已完成收口，当前状态：可进入开发实施阶段；建议按推荐顺序推进，并在开发首周优先打通私聊闭环与群聊入口联调。**

## 5. 推荐开发顺序

建议研发团队按下面顺序推进：

1. **后端先行落实控制面数据层**
   - 以 `Team Panel领域模型与数据架构详细设计` 为准落表与仓储层
2. **同步实现 Gateway 适配骨架**
   - 以 `Agent Gateway运行时适配与事件流详细设计` 为准落运行接入层
3. **优先打通 Phase 1 私聊闭环**
   - 招募员工 -> Profile 供应 -> 单聊 -> Event 回流 -> usage/audit
4. **再上群聊与编排流程**
   - 按 `会话群聊编排Loop核心流程详细设计` 做状态与异常闭环
5. **前后端并行收口**
   - 以 `前端页面与接口契约详细设计` 为准对齐页面与接口
6. **最后补完 Loop、连接器、后台治理与审计经营能力**

## 6. 开发启动前检查清单

在正式开工前，建议项目负责人确认：

- [ ] 研发、测试、产品使用同一套文档版本
- [ ] 数据库命名与表设计以 Team Panel 数据架构子文档为准
- [ ] Gateway 不被误做成第二套业务后台
- [ ] 前端不直接绑定 Hermes Runtime 内部对象
- [ ] 群聊与编排不再沿用单聊 SSE 语义硬拼
- [ ] 所有 run/task/job 都保留 runtime_handle
- [ ] 连接器与密钥权威口径保留在 Team Panel / Secret Store 边界
- [ ] Phase 1 验收口径已被团队接受

建议按以下顺序进入实施：
